Diferencias con React y bugs corregidos

Un clon didáctico vale tanto por lo que enseña como por lo que reconoce no ser. Este capítulo enumera con franqueza dónde este motor se aparta de React real y documenta los cuatro defectos concretos que el análisis del código sacó a la luz, con su antes y su después.

Actualización · Motor v2

Varias de las diferencias que este capítulo describía como limitaciones ya fueron implementadas en la reescritura del motor: reconciliación por key (cap. 23), reciclado de fibers con doble buffer (cap. 24), bailout de subárboles sin cambios (cap. 25) y batching (cap. 26). Este capítulo se conserva como registro histórico del punto de partida.

Dónde se aparta de React

Las siguientes simplificaciones son deliberadas: mantienen el motor legible a costa de rendimiento o de funcionalidad. La mayoría están señaladas en el propio Readme.md del proyecto.

AspectoEste motorReact real
Render phase Recorre el árbol completo en cada render. Usa heurísticas para saltar subárboles donde nada cambió.
Commit phase commitWork recorre todo el árbol de fibers. Mantiene una lista enlazada de fibers con efectos y solo visita esos.
Reutilización de fibers Recrea los fibers desde cero en cada pasada. Recicla los fibers del árbol anterior.
Manejo de updates Reinicia el trabajo desde cero cuando llega una actualización. Prioriza con marcas de expiración (expiration / lanes).
Reconciliación por key No existe: compara solo por tipo y posición. Usa key para detectar reordenamientos de hijos.
Cola de efectos wipEffects es una variable global compartida. Cada componente mantiene su propia lista de efectos.
React.Fragment Referenciado en la config de tipos, pero no implementado. Permite agrupar hijos sin nodo DOM contenedor.
Memoización Sin useMemo, useCallback ni memo. Cachea valores y funciones para evitar recálculos y re-renders.
Nota

Que el motor recorra todo el árbol tanto en render como en commit no lo hace incorrecto, solo menos eficiente. Con árboles pequeños el resultado visible es idéntico al de React; la diferencia importa cuando el número de nodos crece. El roadmap propone cerrar varias de estas brechas.

Bugs corregidos

Durante el estudio se detectaron y corrigieron cuatro defectos reales en src/React.ts (y su compilado public/React.js). Cada uno se presenta con el código antes y después.

Bug 1 — Hijos falsy convertidos en texto

Bug corregido

createElement mapeaba todo hijo no-objeto a createTextElement, incluidos null, undefined y booleanos. Esto rompía el patrón idiomático {condicion && <Componente/>} (renderizaba el texto "false") y, peor, un hijo nulo intermedio podía dejar un "hueco" en el array que hacía a reconcileChildren acceder a prevSibling.sibling con prevSibling === null, lanzando una excepción.

src/React.ts — antes
children: flatten(children)
  .map((child) =>
    typeof child === 'object' ? child : createTextElement(child)
  ),
src/React.ts — después
children: flatten(children)
  // React ignora null, undefined y booleanos como hijos (permite {cond && <X/>}).
  .filter((child) => child != null && typeof child !== 'boolean')
  .map((child) =>
    typeof child === 'object' ? child : createTextElement(child)
  ),

Bug 2 — Limpieza de estilo string

Bug corregido

En updateDom, al remover props viejas, la rama de style hacía Object.keys(prevProps.style) asumiendo un objeto. Con style="background: salmon" (un string), Object.keys devuelve índices de caracteres y escribía dom.style["0"] = "", etc., corrompiendo la limpieza. La corrección restringe esa rama a estilos que sean objetos; el string cae al else, que limpia la propiedad completa.

src/React.ts — antes
if (name === "style") {
  Object.keys(prevProps.style || {}).forEach(styleName => {
    dom.style[styleName] = "";
  });
} else {
  dom[name] = "";
}
src/React.ts — después
if (name === "style" && typeof prevProps.style === "object") {
  Object.keys(prevProps.style || {}).forEach(styleName => {
    dom.style[styleName] = "";
  });
} else {
  dom[name] = "";
}

Bug 3 — useEffect sin deps previas

Bug corregido

El cálculo de hasChanged leía oldHook.deps[i] aunque el hook anterior no tuviera deps (por ejemplo, al pasar de useEffect(fn) a useEffect(fn, [x])), lo que lanzaba al indexar sobre undefined. La corrección añade la guarda (oldHook && oldHook.deps): solo se comparan dependencias si el render previo realmente registró un array; en otro caso el efecto se ejecuta.

src/React.ts — antes
const hasChanged = oldHook
  ? (!deps || deps.some((dep, i) => !Object.is(dep, oldHook.deps[i])))
  : true;
src/React.ts — después
const hasChanged = (oldHook && oldHook.deps)
  ? (!deps || deps.some((dep, i) => !Object.is(dep, oldHook.deps[i])))
  : true;

Bug 4 — Mayúsculas en index.html

Bug corregido

public/index.html cargaba ./app.js en minúscula, mientras el build de Babel genera App.js. En sistemas de archivos sensibles a mayúsculas (Linux/CI, o al servir con serve) el script no cargaba y la app quedaba en blanco. La corrección alinea el nombre: ./App.js.

public/index.html — antes
<script src="./app.js"></script>
public/index.html — después
<script src="./App.js"></script>
Cuidado

El Bug 4 es especialmente insidioso porque no falla en Windows ni en macOS con sus sistemas de archivos por defecto (insensibles a mayúsculas): allí app.js y App.js apuntan al mismo archivo. Solo aparece al desplegar en Linux o al servir con herramientas estrictas. Es un recordatorio de por qué conviene probar en el mismo entorno donde se despliega.
