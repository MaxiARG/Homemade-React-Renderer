Recorrido del árbol de fibers

Procesar un fiber implica una pregunta constante: ¿cuál es la siguiente unidad de trabajo? La respuesta se resuelve siguiendo tres enlaces —child, sibling y parent— en un orden preciso que recorre todo el árbol en profundidad sin recursión.

El orden de recorrido

Al final de performUnitOfWork, después de procesar un fiber, el motor busca el siguiente aplicando una regla de tres pasos, siempre en el mismo orden de prioridad:

  1. Primero el hijo (child): si el fiber tiene hijos, el primero es la próxima unidad de trabajo.
  2. Luego el hermano (sibling): si no hay hijo, se pasa al siguiente hermano.
  3. Luego el «tío»: si tampoco hay hermano, se sube al parent y se prueba con su hermano; se sigue subiendo por los padres hasta encontrar un hermano o llegar a la raíz.

El código lo expresa de forma directa. Primero intenta con el hijo; si no existe, entra en un bucle while que sube por los padres buscando un hermano:

src/React.ts — performUnitOfWork
// Return  next unit of work
// We first try with the child, then with the sibling, then with the uncle, and so on.
if (fiber.child) {
  return fiber.child
}
let nextFiber = fiber
while (nextFiber) {
  if (nextFiber.sibling) {
    return nextFiber.sibling
  }
  nextFiber = nextFiber.parent
}

Ese while es la clave del «tío». Cuando un fiber no tiene ni hijo ni hermano, nextFiber asciende al padre; si el padre tampoco tiene hermano, sube al abuelo, y así sucesivamente. Si el bucle llega hasta un fiber sin padre (la raíz), nextFiber pasa a ser null, la función termina sin devolver nada y el motor entiende que ya no queda trabajo: el render está completo.

Nota

Este mismo patrón de recorrido —child primero, luego sibling— se repite en commitWork durante la fase de commit, allí sí de forma recursiva porque el commit no necesita ser interrumpible.

Cómo se construyen los enlaces: reconcileChildren

El recorrido solo funciona si los enlaces child y sibling están correctamente tejidos. Quien los teje es reconcileChildren, recorriendo los elementos hijos y armando una lista enlazada: el primer hijo (index === 0) se cuelga del padre como child, y cada hijo posterior se engancha al anterior mediante sibling.

src/React.ts — reconcileChildren
if (index === 0) {
  wipFiber.child = newFiber
} else if (element) {
  prevSibling.sibling = newFiber
}
// Think of this as a double linked list!
prevSibling = newFiber
index++

La variable prevSibling recuerda el último fiber creado para poder enlazarle el siguiente. Cada newFiber también recibe parent: wipFiber, cerrando el enlace hacia arriba que el bucle del «tío» necesita para ascender. El resultado es una estructura donde, desde cualquier nodo, se puede alcanzar a todo el árbol siguiendo punteros explícitos.

Bug corregido

Si un hijo intermedio quedaba null (por ejemplo con {condicion && <Componente/>}), se abría un «hueco» en el array y prevSibling.sibling se ejecutaba con prevSibling === null, lanzando una excepción. Se corrigió filtrando los hijos falsy en createElement con .filter((child) => child != null && typeof child !== 'boolean').

Orden de visita en un árbol concreto

Consideremos el árbol div > h1 > (p, a) más un h2 hermano de h1. El recorrido en profundidad numera las visitas así:

┌───────────┐ │ 1. div │ └─────┬─────┘ child │ ▼ ┌───────────┐ sibling ┌───────────┐ │ 2. h1 │ ──────────▶ │ 5. h2 │ └─────┬─────┘ └───────────┘ child │ ▼ ┌───────────┐ sibling ┌───────────┐ │ 3. p │ ──────────▶ │ 4. a │ └───────────┘ └───────────┘ 1 div → tiene child → va a h1 2 h1 → tiene child → va a p 3 p → sin child, tiene sibling → va a a 4 a → sin child ni sibling → sube a h1, h1 tiene sibling → va a h2 5 h2 → sin child ni sibling → sube a div, div sin sibling → sube a raíz → FIN

El paso interesante es el de a (nodo 4): no tiene hijo ni hermano, así que el bucle sube a su padre h1, que sí tiene un hermano (h2). Ese es el «tío» de a. Luego h2 tampoco tiene hijo ni hermano; el bucle sube a div, que no tiene hermano, y sigue subiendo hasta agotar los padres: el render termina.

Por qué esta estructura permite pausar y retomar

Un recorrido recursivo tradicional guarda su posición en la pila de llamadas del navegador: cada nivel de anidamiento es un marco en la pila que no se puede «congelar» ni serializar. Una vez iniciada la recursión, no se puede interrumpir sin perder el punto exacto en el que se estaba.

Aquí, en cambio, la posición del recorrido es un simple dato: el valor de nextUnitOfWork. Como cada fiber lleva enlaces explícitos a su child, sibling y parent, el motor puede detenerse tras procesar cualquier fiber, guardar en esa variable cuál sigue, ceder el hilo al navegador, y más tarde retomar exactamente donde quedó. No hay estado oculto en la pila: todo el «dónde estoy» vive en enlaces de objetos que persisten entre frames.

Idea clave

Convertir la recursión en una lista enlazada explícita transforma «el navegador recuerda dónde voy» en «yo recuerdo dónde voy con una variable». Eso es precisamente lo que habilita el scheduler cooperativo del siguiente capítulo.
