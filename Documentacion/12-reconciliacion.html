El diffing

Reconciliación (diffing)

Reconciliar es comparar el árbol de elementos que queremos renderizar ahora contra el que ya commiteamos la vez anterior, para calcular el conjunto mínimo de cambios. Toda esa lógica vive en reconcileChildren.

Iterar dos estructuras a la vez

La función recorre simultáneamente dos cosas: el array elements[] (lo que queremos renderizar) y la lista enlazada de fibers viejos, a la que se llega vía wipFiber.alternate.child y de ahí siguiendo oldFiber.sibling. La condición del bucle (index < elements.length || oldFiber != null) garantiza que el recorrido no termine hasta agotar ambas: así se detectan tanto hijos nuevos que sobran en el array como hijos viejos que ya no aparecen.

src/React.ts
function reconcileChildren(wipFiber, elements) {
  let index = 0
  let prevSibling = null
  let oldFiber = wipFiber.alternate && wipFiber.alternate.child

  while (index < elements.length ||
    oldFiber != null
  ) {
    const element = elements[index]
    let newFiber = null
    const sameType =
      oldFiber &&
      element &&
      element.type == oldFiber.type

    if (sameType) {
      newFiber = {
        type: oldFiber.type,
        props: element.props,
        dom: oldFiber.dom,
        parent: wipFiber,
        alternate: oldFiber,
        effectTag: "UPDATE",
      }
    }
    if (element && !sameType) {
      newFiber = {
        type: element.type,
        props: element.props,
        dom: null,
        parent: wipFiber,
        alternate: null,
        effectTag: "PLACEMENT",
      }
    }
    if (oldFiber && !sameType) {
      oldFiber.effectTag = "DELETION"
      deletions.push(oldFiber)
    }

    if (oldFiber) {
      oldFiber = oldFiber.sibling
    }

    if (index === 0) {
      wipFiber.child = newFiber
    } else if (element) {
      prevSibling.sibling = newFiber
    }
    prevSibling = newFiber
    index++
  }
}

La clave del diff: sameType

La comparación se reduce a una variable: sameType, que es cierta cuando existen ambos, viejo y nuevo, y element.type == oldFiber.type. El tipo es lo que decide si un nodo se puede reutilizar. A partir de sameType surgen exactamente tres casos.

  1. sameType → UPDATE. Mismo tipo en la misma posición: podemos reutilizar el nodo del DOM viejo. El newFiber conserva dom: oldFiber.dom pero toma props: element.props (las props nuevas), enlaza alternate: oldFiber y se marca effectTag: "UPDATE". En el commit solo se aplicarán las props que cambiaron, no se recrea el nodo.
  2. element && !sameType → PLACEMENT. Hay un elemento nuevo cuyo tipo no coincide (o no había viejo): necesita un nodo del DOM nuevo. Se crea un newFiber con dom: null (se creará en la render phase), alternate: null y effectTag: "PLACEMENT".
  3. oldFiber && !sameType → DELETION. Había un fiber viejo que ya no tiene equivalente: sobra. Como no hay newFiber que representarlo, se marca directamente el viejo con oldFiber.effectTag = "DELETION" y se empuja al array global: deletions.push(oldFiber).
Nota

Los tres if no son excluyentes por casualidad: cuando el tipo cambia, se disparan a la vez el PLACEMENT del nuevo y el DELETION del viejo. Es un reemplazo, no una mutación. Los tres valores de effectTag se detallan en Effect tags.

Construcción del árbol nuevo

A la vez que se calcula el diff, se va tejiendo el árbol de fibers nuevo como una lista doblemente enlazada. El primer hijo (index === 0) se engancha al padre con wipFiber.child = newFiber; los demás se encadenan como hermanos con prevSibling.sibling = newFiber. Al final de cada vuelta, prevSibling = newFiber avanza el puntero. Sobre la lista vieja se avanza en paralelo con oldFiber = oldFiber.sibling. Nótese que cada newFiber guarda su alternate (el fiber viejo equivalente), el enlace que permitirá el diff en el próximo ciclo de render.

árbol viejo (alternate) elementos nuevos oldFiber.child ─┐ elements[0] ▼ │ [div] ── sibling ── [p] compara type == type │ │ ▼ ▼ ¿mismo type? → UPDATE (reusa dom, props nuevas) ¿solo nuevo? → PLACEMENT (dom: null) ¿solo viejo? → DELETION (deletions.push)
Cuidado

Esta reconciliación no usa key. Compara por posición y por tipo únicamente. Si reordenas una lista de hijos del mismo tipo, el motor no detecta que solo cambiaron de lugar: los ve como UPDATE de props posición por posición, lo que puede recrear o reescribir nodos innecesariamente. React real usa key para emparejar hijos entre renders y mover nodos en vez de reconstruirlos. Esta limitación se aborda en Diferencias con React y bugs y en el Roadmap y ejercicios.

En síntesis: reconcileChildren camina viejo y nuevo en paralelo, clasifica cada posición en UPDATE, PLACEMENT o DELETION, y arma el nuevo árbol de fibers con sus effectTag listos para que el commit los ejecute.
