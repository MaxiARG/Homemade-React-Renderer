Datos sin prop-drilling

Context API

El contexto permite que un valor viaje desde un Provider hasta cualquier descendiente sin pasarlo prop por prop. El motor lo implementa con una pila de contextos que se empuja y se saca durante el recorrido del árbol, y un truco para forzar el re-render de los consumidores.

Crear el contexto: createContext

createContext devuelve un objeto con el valor actual, el valor por defecto, una marca _isContextProvider y un componente Provider. La relación entre el Provider y su contexto es circular: el Provider lleva una referencia _context de vuelta al objeto, y es esa marca la que el reconciliador detecta.

src/react/context.ts
export function createContext<T>(defaultValue: T): ReactContext<T> {
  const context: ReactContext<T> = {
    _currentValue: defaultValue,
    _defaultValue: defaultValue,
    _isContextProvider: true,
    Provider: undefined as unknown as FunctionComponent<...>,
  };

  const Provider: ProviderComponent<T> = (props) => {
    // El reconciliador maneja push/pop del valor y reconcilia estos hijos
    // directamente; este cuerpo es un fallback si alguna vez se invocara.
    return { type: Fragment, key: null, props: { children: props.children ?? [] } };
  };
  Provider._context = context;

  context.Provider = Provider;
  return context;
}

Nótese que el cuerpo del Provider casi nunca se ejecuta: no es un componente función corriente. El reconciliador lo intercepta antes de llamarlo, hace el trabajo de contexto y reconcilia sus hijos directamente. El cuerpo que devuelve un Fragment es solo un fallback defensivo.

El Provider en el reconciliador

En beginWork, antes de cualquier otra clasificación, se comprueba isProvider(type). Si el tipo del fiber tiene una propiedad _context, el trabajo se delega a updateContextProvider. Este es el único caso que nunca hace bailout: un Provider debe empujar su valor a la pila en cada render, pase lo que pase.

src/react/reconciler.ts
function updateContextProvider(fiber: Fiber): Fiber | null {
  const context = (fiber.type as { _context: ReactContext<unknown> })._context;
  const value = (fiber.props as { value?: unknown }).value;
  const prevValue = fiber.alternate
    ? (fiber.alternate.props as { value?: unknown }).value
    : context._defaultValue;

  // push del valor, guardando el anterior para restaurarlo en completeWork.
  contextStack.push({ context, value: context._currentValue });
  context._currentValue = value;
  fiber.providerContext = context;

  if (fiber.alternate === null || !Object.is(prevValue, value)) {
    R.forceRenderDepth++;
    fiber.providerForced = true;
  }

  fiber.hasPendingUpdate = false;
  reconcileChildren(fiber, fiber.props.children ?? []);
  return fiber.child;
}

La secuencia es precisa. Primero se apila el valor anterior (context._currentValue, lo que había antes de entrar a este Provider) junto con la referencia al contexto. Luego se sobrescribe context._currentValue con el nuevo value de las props. Se marca fiber.providerContext para que completeWork sepa que este fiber tiene que hacer pop al terminar.

Forzar el re-render de los consumidores

El punto delicado: si el value cambió respecto al render anterior (!Object.is(prevValue, value)), o si es el primer render (fiber.alternate === null), se incrementa R.forceRenderDepth y se marca fiber.providerForced = true. Mientras forceRenderDepth > 0, el bailout queda deshabilitado para todo el subárbol, como se ve en canBailout:

src/react/reconciler.ts
function canBailout(fiber: Fiber): boolean {
  const current = fiber.alternate;
  if (current === null) return false;             // primer render, hay que montar
  if (fiber.hasPendingUpdate) return false;       // tiene setState pendiente
  if (R.forceRenderDepth > 0) return false;       // dentro de un Provider cuyo value cambió
  return fiber.props === current.props;           // props iguales ⇒ misma salida
}

Sin esta línea, un componente intermedio entre el Provider y el consumidor podría hacer bailout (por tener las mismas props referenciales) y cortar el recorrido antes de llegar al consumidor, que se quedaría con el valor viejo. Al mantener forceRenderDepth > 0 mientras se recorre el subárbol del Provider, se garantiza que todos los descendientes se vuelven a ejecutar y, por tanto, que cada useContext lee el valor nuevo.

Restaurar la pila: completeWork

Cuando el recorrido termina de procesar el subárbol de un Provider, completeWork deshace lo que hizo updateContextProvider: saca la entrada de la pila, restaura el _currentValue anterior y, si este Provider había forzado el render, decrementa forceRenderDepth.

src/react/reconciler.ts
function completeWork(fiber: Fiber): void {
  if (fiber.providerContext) {
    const entry = contextStack.pop();
    if (entry) entry.context._currentValue = entry.value; // restaura el anterior
    if (fiber.providerForced) R.forceRenderDepth--;
  }
}

Este emparejamiento push/pop es lo que permite Providers anidados del mismo contexto: cada consumidor ve el valor del Provider más cercano hacia arriba, porque al salir de un Provider se restaura exactamente el valor que regía justo por fuera de él.

Leer el valor: useContext

Con toda la maquinaria de la pila, el hook es trivial: lee el valor vigente en el momento en que corre el componente. Como useContext se ejecuta durante beginWork del consumidor, y el Provider ancestro ya empujó su valor a _currentValue (y aún no lo ha restaurado, porque el completeWork del Provider ocurre después de todo su subárbol), el valor es el correcto.

src/react/hooks.ts
export function useContext<T>(context: ReactContext<T>): T {
  const { fiber } = slot();
  const value = context._currentValue;
  const hook: ContextHook<T> = { tag: 'context', context, value };
  commitHook(fiber, hook as unknown as Hook);
  return value;
}

El hook consume su slot() y hace commitHook como cualquier otro, para no romper el orden de los índices (véase las reglas de los hooks), pero su lógica se reduce a devolver context._currentValue.

La pila de contextos durante el recorrido

Árbol: contextStack (Theme = 'light' por defecto) <App> ───────────────────────────────────────────── <Theme.Provider value="dark"> beginWork Provider: <Layout> push { Theme, 'light' } _currentValue = 'dark' <Consumer> forceRenderDepth: 0 → 1 ───────────────────────────────────────────── recorrido hacia abajo ▼ Consumer: useContext(Theme) ⇒ lee 'dark' ───────────────────────────────────────────── completeWork del Provider ▲ pop → _currentValue vuelve a 'light' forceRenderDepth: 1 → 0

El test de useContext

src/react/__tests__/engine.test.ts
const Theme = createContext('light');
function Consumer() {
  const theme = useContext(Theme);
  return h('span', null, theme);
}
function App() {
  const [theme, s] = useState('light');
  setTheme = s;
  return h(Theme.Provider, { value: theme }, h(Consumer));
}
const { container } = mount(h(App));
expect(container.textContent).toBe('light');
flushSync(() => setTheme('dark'));
expect(container.textContent).toBe('dark');

Al montar, el Consumer lee 'light'. Cuando setTheme('dark') re-renderiza App, el Provider recibe value="dark"; como difiere del anterior, forceRenderDepth sube a 1 y el Consumer no puede hacer bailout: se re-ejecuta y su useContext devuelve 'dark'. El texto del DOM pasa a 'dark'. El flushSync hace que ese cambio sea observable de inmediato (véase Batching y flushSync).

Simplificación

Este contexto es más grueso que el de React. forceRenderDepth desactiva el bailout de todo el subárbol del Provider cuando el valor cambia: se re-renderiza todo lo que cuelga del Provider, consuma o no el contexto. React, en cambio, mantiene una lista de suscriptores por contexto y solo re-renderiza los componentes que realmente llaman a useContext, propagando el cambio de forma fina aunque haya memo intermedios. Aquí no hay suscripción: hay «fuerza bruta acotada al subárbol», más simple de implementar y suficiente para el objetivo didáctico.
