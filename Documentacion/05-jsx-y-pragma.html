El azúcar sintáctico

JSX y el pragma createElement

JSX no es HTML dentro de JavaScript: es una notación que el compilador traduce a llamadas a funciones ordinarias. Entender esa traducción es el primer paso para entender cómo este motor construye su árbol de elementos.

De etiquetas a llamadas de función

El navegador no sabe interpretar JSX. Antes de ejecutarse, cada etiqueta anidada se transforma en una invocación a una función que devuelve un objeto JavaScript plano. En este proyecto esa función es React.createElement. Consideremos un fragmento real del punto de entrada src/App.tsx:

src/App.tsx — antes (JSX)
<div style="background: salmon">
  <h1>Construyendo React de manera casera</h1>
  <h2 style="text-align:right">Caso de estudio</h2>
</div>

El compilador (Babel, configurado con el pragma que veremos abajo) reescribe ese fragmento como una expresión anidada de llamadas. Cada elemento hijo se convierte en un argumento posicional del padre:

equivalente — después (JS)
React.createElement(
  "div",
  { style: "background: salmon" },
  React.createElement("h1", null, "Construyendo React de manera casera"),
  React.createElement("h2", { style: "text-align:right" }, "Caso de estudio")
);

Observa la correspondencia exacta: el nombre de la etiqueta se vuelve el primer argumento (type), los atributos se agrupan en un objeto de props (segundo argumento), y todo lo que estaba anidado entre las etiquetas de apertura y cierre pasa como una lista de argumentos adicionales. El resultado es un árbol de objetos que describe la interfaz sin tocar todavía el DOM: una representación declarativa que el motor consumirá más tarde.

La firma del pragma

El "pragma" es simplemente el nombre de la función a la que el compilador redirige cada etiqueta JSX. Aquí es React.createElement, cuya firma en src/React.ts declara los tres roles que acabamos de ver:

src/React.ts
export function createElement(
  type: string,
  props: Props,
  ...children: any[]
): Element {
  return {
    type,
    props: {
      ...props,
      children: flatten(children)
        // ...
    },
  };
}

Los tres parámetros reflejan la anatomía de una etiqueta: type identifica qué se renderiza, props contiene sus atributos, y ...children —un parámetro rest— recoge la cantidad variable de hijos. El objeto devuelto tiene la forma del tipo Element definido en src/types/index.ts: { type, props }, donde props.children anida recursivamente más elementos. En el próximo capítulo, createElement y flattening, analizaremos qué le ocurre a esos hijos.

Componentes función frente a etiquetas host

El primer argumento, type, admite dos naturalezas distintas y esa diferencia gobierna todo lo que sucede después. Cuando escribes una etiqueta en minúscula como <div> o <h1>, el compilador pasa un string literal ("div", "h1"). Son las llamadas host components: se corresponden con nodos reales del DOM. Cuando escribes una etiqueta capitalizada como <Counter /> o <Label />, el compilador pasa la referencia a la función misma, no un string.

En src/App.tsx conviven ambos casos: <h1> y <div> son etiquetas host, mientras que <Label theLabel={'Un label pasado por prop'}/>, <Box /> y <Counter /> son componentes función. El motor distingue ambos más adelante con una sola comprobación en performUnitOfWork:

src/React.ts
const isFunctionComponent = fiber.type instanceof Function

Si type es una función, el motor la ejecuta para obtener sus hijos (fiber.type(fiber.props)); si es un string, crea un nodo del DOM con document.createElement(fiber.type). Esta bifurcación es la razón por la que la convención de mayúsculas no es cosmética.

Cuidado

Mayúscula inicial = componente; minúscula = etiqueta DOM. Si escribieras <counter /> en minúscula, el compilador lo trataría como el string "counter" e intentaría crear un elemento DOM inexistente con ese nombre, en lugar de ejecutar tu función. La capitalización es la única señal que el transpilador usa para decidir entre pasar un string o pasar la referencia a la función.

¿Por qué window.React global?

El código transpilado emite llamadas literales a React.createElement(...). Para que ese identificador React exista en tiempo de ejecución dentro de cada módulo de ejemplo, el proyecto lo expone en el objeto global del navegador. La última línea de src/React.ts lo hace explícito:

src/React.ts
export const React = {
  createElement,
  render,
  useState,
  useEffect
};

// ...
window.React = React;

Al asignar el objeto a window.React, cualquier componente puede llamar a window.React.useState(...) o window.React.useEffect(...) sin importar el módulo explícitamente —como hace Counter—, y las llamadas a React.createElement que produce el compilador encuentran su destino. Es una simplificación didáctica: React real no depende de un global de este tipo, sino de importaciones y de un runtime de JSX moderno. Aquí, exponerlo globalmente evita fricción de bundling y mantiene el foco en el motor.

Nota

El bloque declare global { interface Window { React: … } } en src/React.ts no genera código: solo informa a TypeScript de que window.React existe y con qué forma, para que el compilador no proteste al usarlo desde los ejemplos.

Con la traducción de JSX comprendida, el siguiente paso es abrir la caja de createElement y ver cómo procesa la lista variádica de hijos: normaliza, aplana arrays anidados y descarta valores vacíos.
