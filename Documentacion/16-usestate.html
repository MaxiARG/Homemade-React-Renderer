Estado que sobrevive al re-render

Hook useState

useState es solo una función, pero consigue «recordar» su valor entre renders apoyándose en el fiber anterior (alternate) y en un índice de hook estable.

El problema que resuelve

Un componente de función se vuelve a ejecutar entero en cada render: sus variables locales nacen y mueren con cada llamada. ¿Cómo puede entonces useState(2) devolver 3 tras un click, si el 2 inicial se vuelve a evaluar cada vez? La respuesta es que el estado no vive en el componente, sino en el fiber, y useState lo rescata del fiber del render anterior a través de la propiedad alternate.

La función completa

src/React.ts
function useState(initial) {
  const oldHook =
    wipFiber.alternate &&
    wipFiber.alternate.hooks &&
    wipFiber.alternate.hooks[hookIndex]
  const hook = {
    state: oldHook ? oldHook.state : initial,
    queue: [],
  }

  const actions = oldHook ? oldHook.queue : []
  actions.forEach(action => {
    hook.state = action(hook.state)
  })

  const setState = action => {
    hook.queue.push(action)
    wipRoot = {
      dom: currentRoot.dom,
      props: currentRoot.props,
      alternate: currentRoot,
    }
    nextUnitOfWork = wipRoot //Esto hace que se ejecute un nuevo render
    deletions = []
  }

  wipFiber.hooks.push(hook)
  hookIndex++
  return [hook.state, setState]
}

Recuperar el hook anterior

Lo primero es buscar el hook viejo en el mismo fiber del render anterior. Se accede a wipFiber.alternate.hooks[hookIndex]: wipFiber es el fiber que se está procesando (fijado en updateFunctionComponent), alternate apunta al fiber comprometido en el commit previo, y hookIndex identifica cuál de los hooks del componente es. Las comprobaciones encadenadas con && protegen el primer render, cuando aún no hay alternate ni hooks.

Reconstruir el estado con la cola de acciones

En vez de guardar un número mutable, cada hook lleva una queue de acciones pendientes. El estado se inicializa desde el hook viejo (oldHook ? oldHook.state : initial) y luego se aplican una a una las acciones que quedaron encoladas desde el render anterior:

src/React.ts
const actions = oldHook ? oldHook.queue : []
actions.forEach(action => {
  hook.state = action(hook.state)
})

Cada action es una función estado → nuevoEstado. Al aplicarlas en secuencia sobre hook.state, el estado que finalmente se devuelve ya está actualizado. Este diseño diferido — encolar ahora, aplicar en el próximo render — es lo que permite que setState no tenga que recalcular nada de inmediato.

setState: encolar y programar el re-render

setState hace dos cosas. Primero, empuja la acción a la cola del hook con hook.queue.push(action). Segundo, programa un nuevo render recreando el wipRoot a partir de currentRoot (el último árbol comprometido) y apuntando nextUnitOfWork a esa nueva raíz:

src/React.ts
const setState = action => {
  hook.queue.push(action)
  wipRoot = {
    dom: currentRoot.dom,
    props: currentRoot.props,
    alternate: currentRoot,
  }
  nextUnitOfWork = wipRoot //Esto hace que se ejecute un nuevo render
  deletions = []
}

Es exactamente el mismo mecanismo que usa la función render para arrancar el primer dibujado: se establece un wipRoot con alternate: currentRoot y se coloca como nextUnitOfWork. La próxima vez que el workLoop encuentre trabajo (véase Scheduler y workLoop), arrancará una nueva render phase y, al terminar, un nuevo commit. También se reinicia deletions = [] para acumular limpiamente las eliminaciones del nuevo diffing.

Guardar el hook y devolver el par

Al final, el hook se añade al fiber actual con wipFiber.hooks.push(hook), se incrementa hookIndex++ para que la próxima llamada a un hook use su propio índice, y se devuelve el par [hook.state, setState], la firma familiar de const [x, setX] = useState(...).

Cuidado

Reglas de los hooks. Todo este mecanismo depende de que hookIndex sea estable: el hook nº0 de este render debe corresponder al hook nº0 del render anterior. Por eso los hooks no se pueden llamar dentro de condicionales ni bucles: si en un render se saltara un useState, los índices se desalinearían y cada hook recuperaría el estado equivocado de alternate.hooks[hookIndex]. El orden de las llamadas es la identidad de cada hook.

Ejemplo: el contador

El componente Counter del proyecto ejercita todo lo anterior. Declara su estado con useState(2) y lo actualiza pasando una función actualizadora a setCounter:

src/Examples/components/Counter.tsx
const Counter = () => {
  const [counter, setCounter] = window.React.useState(2)

  return (
    <div>
      <button onClick={() => {
        console.log('onClick');
        setCounter(c => c + 1);
      }}>
        Click me
      </button>
      Counter: {counter}
    </div>
  );
};

Seguir un click paso a paso ilumina toda la maquinaria. El usuario pulsa el botón; setCounter(c => c + 1) encola la acción c => c + 1 en hook.queue y programa el re-render. En el siguiente render, useState encuentra el oldHook con state: 2 y su cola con esa acción; la aplica (2 → 3) y devuelve 3. El componente renderiza Counter: 3, la reconciliación etiqueta el nodo de texto como UPDATE y el commit lo escribe en pantalla.

Idea clave

Pasar una función (c => c + 1) en lugar de un valor (counter + 1) es lo correcto aquí porque el estado se calcula aplicando la cola sobre el estado base. Si se encolaran varias acciones antes de renderizar, cada una recibiría el resultado de la anterior, sin depender de una variable counter ya obsoleta.

El estado ya sobrevive a los renders; falta ver cómo se ejecuta lógica después de pintar. Eso se aborda al hablar del hook useEffect y cleanup, y el recorrido completo, desde el click hasta el DOM, se recompone en el ciclo de vida completo.
