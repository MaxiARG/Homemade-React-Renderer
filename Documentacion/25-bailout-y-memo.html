No re-renderizar de más

Bailout y memoización

Un setState profundo no debería obligar a re-ejecutar toda la aplicación. El motor se salta (bailout) el trabajo de cualquier fiber cuyas props no cambiaron y no tenga actualizaciones pendientes, reusando su subárbol vía cloneChildFibers.

La decisión: canBailout

Antes de ejecutar un componente, beginWork consulta canBailout. Son tres condiciones que deben cumplirse todas para poder saltar el trabajo:

src/react/reconciler.ts
function canBailout(fiber: Fiber): boolean {
  const current = fiber.alternate;
  if (current === null) return false; // primer render, hay que montar
  if (fiber.hasPendingUpdate) return false; // tiene setState pendiente
  if (R.forceRenderDepth > 0) return false; // dentro de un Provider cuyo value cambió
  // props referencialmente iguales => misma salida garantizada
  return fiber.props === current.props;
}
  1. Debe existir un alternate. Si es el primer render, no hay nada que reusar: hay que montar.
  2. No debe tener hasPendingUpdate. Un dispatch de useState marca este flag (véase el manejo del doble buffer en 24); si está encendido, el componente debe re-ejecutarse para procesar su cola.
  3. R.forceRenderDepth debe ser 0. Estamos dentro de un Provider cuyo value cambió: los consumidores podrían depender de él, así que se fuerza el render (véase 27 · Context API).

La condición final es la más importante: fiber.props === current.props. Se compara por referencia, no por contenido. Si el padre no re-renderizó, el mismo objeto de props se reutiliza y la igualdad referencial se cumple gratis, garantizando que la salida del componente será idéntica.

Las ramas de bailout en beginWork

beginWork aplica canBailout tanto a los fibers estructurales (ROOT, Fragment, host) como a los componentes función. Cuando el bailout procede, en lugar de reconciliar se clonan los hijos y se devuelve el primero:

src/react/reconciler.ts
export function beginWork(fiber: Fiber): Fiber | null {
  const type = fiber.type;

  // Los providers nunca hacen bailout (deben empujar su valor siempre).
  if (isProvider(type)) return updateContextProvider(fiber);

  if (type === 'ROOT' || type === Fragment || isHostType(fiber)) {
    if (canBailout(fiber)) {
      cloneChildFibers(fiber.alternate as Fiber, fiber);
      return fiber.child;
    }
    return type === 'ROOT' || type === Fragment
      ? updateFragmentLike(fiber)
      : updateHostComponent(fiber);
  }

  if (isMemo(type)) return updateMemoComponent(fiber, type);
  if (isClassComponent(type)) return updateClassComponent(fiber);

  // Componente función.
  if (canBailout(fiber)) {
    cloneChildFibers(fiber.alternate as Fiber, fiber);
    return fiber.child;
  }
  return updateFunctionComponent(fiber);
}

cloneChildFibers (detallado en 24) reconstruye los hijos wip a partir del árbol current con las mismas props, sin ejecutar ningún componente. Como esos clones también tendrán props referencialmente iguales a las suyas viejas, el bailout se propaga hacia abajo: subárboles enteros se saltan en cascada.

Cómo un setState re-ejecuta SOLO su componente

Este es el pago del mecanismo. Cuando llamás a setState en un componente hondo, el render arranca desde la raíz, pero cada ancestro cuyas props no cambiaron hace bailout y clona su subárbol. El único fiber con hasPendingUpdate === true es el que disparó el dispatch: solo él falla canBailout y se re-ejecuta. Sus hermanos, con props idénticas, se clonan intactos y conservan su estado.

El test "un setState no re-renderiza al hermano sin cambios" lo verifica de forma directa: un componente Stateful y un Sibling viven bajo el mismo div. Tras un setN(5), solo Stateful vuelve a ejecutarse.

src/react/__tests__/engine.test.ts
test('un setState no re-renderiza al hermano sin cambios', () => {
  let siblingRenders = 0;
  let statefulRenders = 0;
  let setN!: Dispatch<SetStateAction<number>>;
  function Sibling() {
    siblingRenders++;
    return h('span', null, 'sib');
  }
  function Stateful() {
    statefulRenders++;
    const [n, s] = useState(0);
    setN = s;
    return h('span', null, String(n));
  }
  function App() {
    return h('div', null, h(Stateful), h(Sibling));
  }
  mount(h(App));
  expect(siblingRenders).toBe(1);
  expect(statefulRenders).toBe(1);
  flushSync(() => setN(5));
  expect(statefulRenders).toBe(2);
  expect(siblingRenders).toBe(1); // bailout: no volvió a renderizar
});
flushSync(() => setN(5)) App props === -> canBailout -> clona hijos ├─ Stateful hasPendingUpdate=true -> RE-EJECUTA (statefulRenders: 1 -> 2) └─ Sibling props === -> canBailout -> CLONA (siblingRenders queda en 1)

Memoización explícita: updateMemoComponent y memo()

El bailout automático depende de la igualdad referencial de props. Pero si un padre se re-renderiza, crea props nuevas (aunque su contenido sea el mismo), y fiber.props === current.props falla. Ahí entra memo: envuelve un componente para compararlo por contenido superficial en lugar de por referencia.

src/react/index.ts
function shallowEqual(a: Props, b: Props): boolean {
  if (a === b) return true;
  const ak = Object.keys(a);
  const bk = Object.keys(b);
  if (ak.length !== bk.length) return false;
  return ak.every((k) => Object.is(a[k], b[k]));
}

export function memo<P extends object = Props>(
  type: FunctionComponent<P>,
  compare: (prev: P, next: P) => boolean = shallowEqual as (p: P, n: P) => boolean,
): MemoComponent<P> {
  return { $$typeof: 'react.memo', type, compare };
}

Un MemoComponent es un objeto marcado con $$typeof: 'react.memo' que guarda el componente original y una función compare. En beginWork, el chequeo isMemo(type) lo desvía a updateMemoComponent:

src/react/reconciler.ts
function updateMemoComponent(fiber: Fiber, memo: MemoComponent): Fiber | null {
  const current = fiber.alternate;
  if (
    current !== null &&
    !fiber.hasPendingUpdate &&
    R.forceRenderDepth === 0 &&
    memo.compare(current.props as Props, fiber.props)
  ) {
    cloneChildFibers(current, fiber);
    return fiber.child;
  }
  R.wipFiber = fiber;
  R.hookIndex = 0;
  fiber.hooks = [];
  fiber.hasPendingUpdate = false;
  const child = memo.type(fiber.props);
  collectPassiveEffects(fiber);
  reconcileChildren(fiber, child ? [child] : []);
  return fiber.child;
}

Las tres primeras condiciones son las mismas de canBailout (hay alternate, no hay update pendiente, no hay force). La cuarta reemplaza la comparación referencial por memo.compare(...): la comparación superficial. Si todas se cumplen, se clona el subárbol y se salta la ejecución igual que en un bailout normal.

Bailout automático vs. memo

Bailout automáticomemo()
ComparaciónReferencial (props === props)Superficial (Object.is clave por clave)
Cuándo aplicaCuando el padre no re-renderizó y reusa el mismo objeto de propsCuando el padre sí re-renderizó pero produjo props con el mismo contenido
CostoCero (una comparación de punteros)Recorre las claves de props una vez

La complementariedad

Los dos mecanismos cubren casos distintos. El bailout automático hace que un setState hondo no toque los hermanos cuyo padre común no cambió las props. memo extiende ese ahorro a los casos donde el padre sí se re-renderizó pero el hijo recibe datos equivalentes. Combinados con useMemo y useCallback (para mantener referencias estables en props de objeto o función), permiten podar re-renders con precisión quirúrgica.
