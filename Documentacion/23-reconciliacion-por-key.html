Identidad estable

Reconciliación por key

El diffing por posición del capítulo 12 era barato pero frágil ante reordenamientos. El motor refactorizado empareja los hijos por key, detecta cuáles se movieron y reusa sus fibers, preservando estado y nodos del DOM.

El algoritmo en reconcileChildren

Toda la lógica vive en reconcileChildren dentro de src/react/reconciler.ts. El punto de partida es construir un Map de los fibers viejos (los hijos del alternate del padre) indexados por su key o, cuando no la tienen, por su índice de posición.

src/react/reconciler.ts
export function reconcileChildren(returnFiber: Fiber, elements: ReactElement[]): void {
  const root = returnFiber.root as RootStateContainer;
  const deletions = root.deletions;

  // Mapa de fibers viejos por key (o índice si no hay key).
  const existing = new Map<Key | number, Fiber>();
  let old = returnFiber.alternate ? returnFiber.alternate.child : null;
  let oldIdx = 0;
  while (old) {
    const k = old.key != null ? old.key : oldIdx;
    existing.set(k, old);
    old.index = oldIdx;
    old = old.sibling;
    oldIdx++;
  }

  let prevSibling: Fiber | null = null;
  let lastPlacedIndex = 0;

  for (let i = 0; i < elements.length; i++) {
    const element = elements[i];
    const k: Key | number = element.key != null ? element.key : i;
    const matched = existing.get(k);
    let fiber: Fiber;

    if (matched && sameType(matched, element)) {
      // Reusar (posiblemente moviendo).
      fiber = createWorkInProgress(matched, element.props);
      fiber.effectTag = 'UPDATE';
      existing.delete(k);
      if (matched.index < lastPlacedIndex) {
        fiber.moved = true; // se movió a la derecha: reinsertar en el DOM
      } else {
        lastPlacedIndex = matched.index;
      }
    } else {
      if (matched) {
        // misma key, distinto tipo => borrar el viejo
        matched.effectTag = 'DELETION';
        deletions.push(matched);
        existing.delete(k);
      }
      fiber = createFiberFromElement(element);
      fiber.effectTag = 'PLACEMENT';
    }

    fiber.parent = returnFiber;
    fiber.root = root;
    fiber.index = i;

    if (i === 0) returnFiber.child = fiber;
    else if (prevSibling) prevSibling.sibling = fiber;
    prevSibling = fiber;
  }

  // Los que sobraron en el mapa se eliminan.
  for (const fiber of existing.values()) {
    fiber.effectTag = 'DELETION';
    deletions.push(fiber);
  }
}

Emparejar por key, no por posición

Para cada elemento nuevo se calcula su clave k (la key del elemento o su índice) y se busca en el Map existing. Si aparece un fiber viejo con esa misma clave y además sameType confirma que el type coincide, el fiber viejo se reusa. La reutilización pasa por createWorkInProgress(matched, element.props) (véase 24 · Doble buffer), que hereda el dom, el stateNode y los hooks del fiber previo. Esa herencia es exactamente lo que preserva el estado y el nodo real cuando la lista se reordena.

src/react/reconciler.ts
function sameType(a: Fiber, b: ReactElement): boolean {
  return a.type === b.type;
}

Si la clave existe pero el tipo cambió (misma key, distinto componente), el fiber viejo se marca DELETION y se crea uno nuevo con PLACEMENT. Si la clave no existía, es un elemento nuevo: createFiberFromElement + PLACEMENT. Al final, cualquier fiber que quedó en existing (nadie lo reclamó) se elimina.

Detectar movimientos con lastPlacedIndex

Reusar un fiber no significa que su posición en el DOM siga siendo correcta. El motor usa la misma heurística que React: recorre los elementos nuevos de izquierda a derecha y lleva un lastPlacedIndex, que es el mayor índice antiguo ya "colocado". Para cada fiber reusado compara matched.index (su posición en el árbol viejo) contra ese umbral:

  • Si matched.index < lastPlacedIndex: el elemento estaba antes de algo que ahora quedó a su izquierda, o sea que en el orden nuevo se corrió a la derecha. Se marca fiber.moved = true para reinsertarlo en el commit.
  • Si no, este elemento ya estaba en orden creciente: se avanza lastPlacedIndex = matched.index y no se mueve.

La idea es minimizar operaciones sobre el DOM: solo se reubican los elementos estrictamente necesarios para reconstruir el orden, dejando quietos a los que ya forman una subsecuencia creciente.

Ejemplo: [a, b, c] → [c, a, b]

Supongamos tres hijos con keys a, b, c en índices viejos 0, 1 y 2. El nuevo orden pedido es c, a, b. El recorrido decide así:

viejo: [ a(0) b(1) c(2) ] nuevo: [ c a b ] paso i=0 key=c matched.index=2 2 < lastPlacedIndex(0)? no -> lastPlacedIndex=2 paso i=1 key=a matched.index=0 0 < lastPlacedIndex(2)? SI -> a.moved = true paso i=2 key=b matched.index=1 1 < lastPlacedIndex(2)? SI -> b.moved = true resultado: c queda fijo (ancla); a y b se reinsertan con insertBefore

Observá que se marcan como movidos a y b, no c. Es contraintuitivo (uno "siente" que fue c el que se movió al frente), pero el algoritmo elige el mínimo de reinserciones tomando a c como ancla y arrastrando el resto detrás. Ninguno de los tres fibers se recrea: los tres conservan su DOM y su estado.

Cómo el commit ejecuta el reorden

La fase de render solo marca banderas (moved, PLACEMENT). Es el commit el que toca el DOM. En commitWork, un fiber con UPDATE que además tiene moved dispara commitPlacement:

src/react/reconciler.ts
function commitWork(fiber: Fiber | null): void {
  if (!fiber) return;
  if (fiber.effectTag === 'PLACEMENT') {
    commitPlacement(fiber);
  } else if (fiber.effectTag === 'UPDATE') {
    if (fiber.dom && fiber.alternate) {
      updateDom(fiber.dom, fiber.alternate.props, fiber.props);
    }
    if (fiber.moved) commitPlacement(fiber);
  }
  commitWork(fiber.child);
  commitWork(fiber.sibling);
}

commitPlacement resuelve el DOM padre y luego llama a getHostSibling para encontrar el nodo ante el cual insertar, respetando el orden final. Ese "sibling" salta fibers que a su vez están en PLACEMENT o moved (todavía no están en su lugar) hasta hallar un nodo host estable que sirva de referencia. Con esa referencia, placeNode usa insertBefore (o appendChild si no hay referencia):

src/react/reconciler.ts
function commitPlacement(fiber: Fiber): void {
  const parentDom = getHostParentDom(fiber);
  const before = getHostSibling(fiber);
  placeNode(fiber, parentDom, before);
}

function placeNode(fiber: Fiber, parentDom: DomNode, before: DomNode | null): void {
  if (isHostType(fiber)) {
    if (fiber.dom) {
      if (before) parentDom.insertBefore(fiber.dom, before);
      else parentDom.appendChild(fiber.dom);
    }
  } else {
    // Fragment / función / clase: colocar sus hijos host.
    let child = fiber.child;
    while (child) {
      placeNode(child, parentDom, before);
      child = child.sibling;
    }
  }
}

Como el mismo nodo del DOM se reinserta (no se recrea), un <input> con foco o un video reproduciéndose sobrevive al reorden. Esa es la diferencia práctica frente al matching por posición del capítulo 12, donde reordenar mutaba props sobre nodos ajenos y perdía estado.

Por qué las keys preservan estado

EstrategiaAl pasar de [a,b,c] a [c,a,b]Consecuencia
Matching por posición (doc 12) El fiber del slot 0 (antes a) se reusa para c: se sobrescriben sus props. El estado interno y el DOM quedan asociados a la posición, no al elemento lógico. Se corrompen los hooks.
Matching por key (doc actual) El fiber de a se empareja con el elemento a por su key, sin importar el slot. El estado y el DOM viajan con la identidad lógica del elemento. Solo se reinserta el nodo si hace falta.
Limitación conocida

La detección de movimientos opera a nivel de elementos host. Reordenar un subárbol envuelto en un componente función o clase se resuelve correctamente en cuanto a los nodos del DOM finales — getHostSibling y placeNode descienden hasta los hijos host y los reinsertan en orden. Pero el motor no tiene un tratamiento especial para "mover un componente entero" como una unidad indivisible: la corrección emerge de reubicar sus nodos host uno a uno. En árboles muy anidados con muchos wrappers, esto puede producir más operaciones insertBefore de las mínimas teóricas.

Regla práctica

Poné key estables y únicos (nunca el índice del array si la lista se reordena o filtra). La key es el contrato de identidad entre renders: sin ella, el motor cae al emparejamiento por posición y pierde las garantías descritas acá.
