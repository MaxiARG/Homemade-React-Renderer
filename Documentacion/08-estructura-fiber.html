La unidad de trabajo

La estructura Fiber

Un fiber es el nodo que el motor procesa de a uno por vez: contiene la descripción del elemento, su nodo del DOM y los enlaces que permiten recorrer, pausar y reconciliar el árbol. Es el corazón del renderizado interrumpible.

¿Qué es un fiber?

Cada elemento que queremos mostrar en pantalla se convierte, durante la fase de render, en un fiber: un objeto JavaScript que representa una unidad de trabajo. Mientras un Element es solo una descripción estática de lo que quieres renderizar, el fiber es la estructura viva que el motor recorre para hacer ese trabajo. El objetivo de esta estructura de datos es, en palabras del propio proyecto, «make it easy to find the next unit of work»: por eso cada fiber tiene un enlace a su primer hijo, a su siguiente hermano y a su padre.

En este clon los fibers no se declaran con una clase ni con un constructor: se crean como objetos literales dentro de reconcileChildren y de render. Las propiedades que realmente se usan en src/React.ts son: type, props, dom, parent, child, sibling, alternate, effectTag y —para componentes de función— hooks.

PropiedadRol dentro del motor
typeLa etiqueta ("div", "TEXT_ONLY") o la función del componente. Determina si es host o function component.
propsLos atributos y children del elemento asociado.
domEl nodo real del DOM (HTMLElement o Text). Es null en componentes de función.
parentEnlace hacia arriba, al fiber contenedor.
childEnlace al primer hijo.
siblingEnlace al siguiente hermano.
alternateEnlace al fiber viejo (el del árbol previo ya committeado).
effectTagMarca "PLACEMENT", "UPDATE" o "DELETION" para la fase de commit.
hooksArray de hooks del componente de función (estado de useState/useEffect).

El tipo Fiber en el código

El proyecto declara un tipo Fiber en src/types/index.ts. Vale la pena notar que está anotado como «aún no implementada»: sirve como documentación del contrato, pero el motor construye los fibers como objetos literales sin referirse a este tipo. Aun así, describe fielmente los enlaces que sí se usan.

src/types/index.ts
// aun no implementada
export type Fiber = {
    type: string | FunctionComponent;
    props: Props;
    dom: HTMLElement | Text | null;
    parent?: Fiber;
    child?: Fiber;
    sibling?: Fiber;
    alternate?: Fiber;
    effectTag?: "PLACEMENT" | "UPDATE" | "DELETION";
};
Nota

El tipo no incluye hooks, pero el motor sí lo agrega en tiempo de ejecución dentro de updateFunctionComponent con wipFiber.hooks = []. Es un ejemplo de cómo el objeto real supera al tipo declarado.

Element vs Fiber

La distinción es fundamental. Un Element es una descripción inmutable: el objeto plano { type, props } que devuelve createElement. No sabe nada del DOM, no tiene enlaces a hermanos ni a padres, y no guarda estado. Es solo «lo que queremos ver».

Un Fiber, en cambio, es la unidad de trabajo que envuelve a ese element. Añade lo que el element no tiene: un enlace a su nodo del DOM (dom), enlaces estructurales (parent, child, sibling) para recorrer el árbol sin recursión, un enlace al pasado (alternate) para comparar, una etiqueta de efecto (effectTag) para el commit, y —si es un componente— su estado en hooks. En reconcileChildren se ve cómo un element se transforma en fiber copiando element.props pero agregando todos esos enlaces:

src/React.ts — reconcileChildren
newFiber = {
  type: oldFiber.type,
  props: element.props,
  dom: oldFiber.dom,
  parent: wipFiber,
  alternate: oldFiber,
  effectTag: "UPDATE",
}

alternate, wipRoot y currentRoot

La propiedad alternate es el enlace de cada fiber a «su versión anterior»: el fiber que se committeó al DOM en el render previo. Gracias a ella el motor puede comparar el árbol nuevo con el viejo y decidir qué cambió, en lugar de recrear todo a ciegas. Toda la reconciliación de diffing depende de recorrer wipFiber.alternate.child en paralelo con los nuevos elements.

A nivel de árbol completo, el motor mantiene dos raíces globales:

  1. wipRoot (work in progress root): la raíz del árbol que se está construyendo ahora, todavía no visible. Se crea en render y en el setState de useState.
  2. currentRoot: la raíz del último árbol ya committeado al DOM, es decir, lo que el usuario ve en este instante.

Cuando render arranca, enlaza ambos mundos con alternate: currentRoot, de modo que el árbol nuevo «recuerda» al viejo:

src/React.ts — render
wipRoot = {
  dom: container,
  props: {
    children: [element],
  },
  alternate: currentRoot
}

Al terminar el commit, commitRoot hace currentRoot = wipRoot y luego wipRoot = null: el árbol en progreso «se gradúa» y pasa a ser el árbol actual, listo para servir de alternate en el próximo render.

Diagrama de un nodo fiber

Cada fiber apunta a su primer hijo (child), a su siguiente hermano (sibling), hacia arriba a su padre (parent) y hacia el pasado a su alternate:

currentRoot (árbol viejo) wipRoot (árbol nuevo) │ │ ▼ ▼ ┌──────────┐ alternate ┌──────────┐ │ <div> │ ◀───────────────────│ <div> │ │ fiber │ │ fiber │ └──────────┘ └────┬─────┘ parent ▲ │ child │ ▼ ┌──────────┐ sibling ┌──────────┐ │ <h1> │ ──────────▶ │ <h2> │ │ fiber │ │ fiber │ └────┬─────┘ └──────────┘ child │ ▼ ┌──────────┐ │ <p> │ │ fiber │ └──────────┘ child → primer hijo sibling → siguiente hermano parent → contenedor alternate → mismo nodo en el árbol previo
Idea clave

Un fiber es un element «con memoria y con cables»: recuerda su pasado mediante alternate y se conecta a sus vecinos mediante child, sibling y parent. Esos cables son lo que permite recorrer el árbol sin usar la pila de llamadas del navegador, tema del próximo capítulo.
