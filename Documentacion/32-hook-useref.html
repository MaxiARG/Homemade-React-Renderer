Caja mutable

useRef — ciclo de vida completo

useRef es el hook más simple del motor: entrega una caja mutable { current } cuya identidad no cambia entre renders. Mutar .current no dispara ningún re-render, lo que lo vuelve ideal para guardar valores que deben persistir sin afectar la pintura.

Firma y para qué sirve

La firma es useRef<T>(initial: T): { current: T }. Devuelve un objeto con una única propiedad current, inicializada la primera vez con el valor recibido. Sirve para referencias a nodos del DOM, temporizadores, valores previos o cualquier dato mutable que no deba provocar un render al cambiar.

src/react/hooks.ts
export function useRef<T>(initial: T): { current: T } {
  const { fiber, old } = slot();
  const hook =
    (old as RefHook<T> | undefined) ??
    ({ tag: 'ref', ref: { current: initial } } as RefHook<T>);
  commitHook(fiber, hook as unknown as Hook);
  return hook.ref;
}

Un uso mínimo: const ref = useRef(0); luego ref.current++ muta la caja sin re-renderizar.

Dónde vive el estado

La caja vive en un RefHook, la rama más escueta de la unión Hook. Solo tiene su etiqueta tag: 'ref' y el objeto ref: { current: T }. No hay cola ni dispatch: nada que agende trabajo.

src/react/types.ts
export interface RefHook<T = unknown> {
  tag: 'ref';
  ref: { current: T };
}

Como los demás hooks, el RefHook persiste reutilizándose desde fiber.alternate.hooks[hookIndex] a través de slot(). La clave es que se reutiliza el objeto ref entero, no solo su valor: por eso la identidad de la caja es estable render tras render.

Primer render (mount)

  1. updateFunctionComponent prepara R.wipFiber, R.hookIndex = 0 y fiber.hooks = [].
  2. slot() devuelve old = undefined, así que se crea el RefHook con la caja ref: { current: initial }.
  3. El valor inicial se guarda una única vez: en renders futuros nunca se vuelve a leer initial.
  4. commitHook(fiber, hook) empuja el hook e incrementa R.hookIndex++.
  5. Se devuelve hook.ref — la caja que el componente usará.

Renders siguientes (update)

En cada re-render, slot() encuentra el hook viejo y el operador ?? lo reutiliza: no se crea una caja nueva y el initial pasado se ignora por completo. Se devuelve el mismo objeto ref que en el mount, con el valor que tuviera en .current (incluidas las mutaciones que el componente haya hecho entre renders).

Identidad estable

Como el RefHook se saca de alternate.hooks[hookIndex] y se devuelve hook.ref sin reconstruirlo, dos renders consecutivos ven exactamente el mismo objeto. Esa es la propiedad que hace útil a useRef: un contenedor persistente que no se recrea.

El test useRef mantiene identidad entre renders lo verifica: acumula la ref devuelta en cada render y comprueba que la referencia es la misma tras un setN(1).

src/react/__tests__/engine.test.ts
test('useRef mantiene identidad entre renders', () => {
  const refs: Array<{ current: number }> = [];
  let setN!: Dispatch<SetStateAction<number>>;
  function App() {
    const [n, s] = useState(0);
    setN = s;
    const ref = useRef(0);
    refs.push(ref);
    return h('span', null, String(n));
  }
  mount(h(App));
  flushSync(() => setN(1));
  expect(refs[0]).toBe(refs[1]); // misma referencia
});

La actualización / disparador

useRef no tiene disparador. A diferencia de useState, su hook no guarda queue ni dispatch, y nunca llama a scheduleUpdate ni marca hasPendingUpdate. Mutar ref.current = X cambia el contenido de la caja de forma inmediata y silenciosa: el DOM no se actualiza y no se agenda ningún trabajo en el scheduler.

La diferencia con useState

useState encola la acción y llama a scheduleUpdate(target), lo que agenda un microtask y termina en un nuevo render. useRef no hace nada de eso: escribís .current y listo. Si necesitás que la UI reaccione a un cambio, ese cambio debe pasar por estado, no por una ref.

El valor mutado sí queda visible en el próximo render que ocurra por otra causa (por ejemplo un setState vecino), porque la caja es la misma. Pero la ref por sí sola nunca es la causa de ese render.

Orden de ejecución

useRef corre durante la render phase, en el cuerpo del componente dentro de updateFunctionComponent, en el punto exacto en que se lo invoca. Es síncrono y sin efectos secundarios sobre el scheduler: su único trabajo es reservar/recuperar el slot y devolver la caja. Las mutaciones a .current las hace el componente cuando quiera (durante el render, en un handler o en un efecto).

Reglas de los hooks

Aunque no tenga estado reactivo, useRef ocupa un slot y avanza R.hookIndex en commitHook igual que cualquier otro hook. Por eso su posición en la lista de llamadas importa: slot() recupera su caja por alternate.hooks[hookIndex].

Cuidado

No llames useRef condicionalmente. Si un render lo omite, los índices de todos los hooks siguientes se desalinean y cada uno recupera el slot equivocado — incluida la ref, que podría devolver la caja de otro hook.

Diagrama de secuencia

MOUNT updateFunctionComponent: R.wipFiber = fiber, R.hookIndex = 0, fiber.hooks = [] useRef(0) slot() -> old = undefined crea RefHook { tag:'ref', ref: { current: 0 } } commitHook -> hooks.push, hookIndex++ return hook.ref (caja A) | v MUTACION ref.current = 42 escribe en la caja A directamente NO hay queue, NO hay scheduleUpdate, NO hasPendingUpdate el DOM no cambia, no se agenda render | v UPDATE (render disparado por OTRA causa, p.ej. un setState vecino) slot() -> old = alternate.hooks[hookIndex] (MISMO RefHook) ?? reutiliza el hook: initial se IGNORA commitHook -> hookIndex++ return hook.ref (MISMA caja A, current = 42) -> refs[0] === refs[1]
