Arquitectura general

Antes de bajar al detalle de cada función, conviene ver el mapa completo: qué módulos existen, cómo fluyen los datos desde el JSX hasta el DOM, y qué estado global sostiene todo el proceso.

Módulos del proyecto

La arquitectura es deliberadamente plana. Cinco piezas concentran toda la responsabilidad:

MóduloRol
src/React.ts El motor completo: createElement, scheduler, reconciliación, commit y hooks. Es la fuente de verdad.
src/types/index.ts Tipos Element, Props, FunctionComponent y Fiber que documentan la forma de los datos.
src/App.tsx Punto de entrada: construye el árbol raíz en JSX y llama a React.render.
src/Examples/ Componentes de demostración (Counter, Box, Label, StyledComponent, etc.).
public/ Salida compilada por Babel que el navegador ejecuta directamente.

Flujo de datos

Escribir JSX en src/App.tsx desencadena una tubería que termina en nodos reales del DOM. El pragma de Babel reescribe cada etiqueta como una llamada a React.createElement (véase 04 · Tooling), y a partir de ahí el motor toma el control:

JSX (src/App.tsx) │ Babel + pragma "React.createElement" ▼ React.createElement(type, props, ...children) │ flatten + filter + createTextElement ▼ Árbol de Elements { type, props: { children } } │ React.render(element, container) ▼ wipRoot ──► nextUnitOfWork = wipRoot │ ▼ workLoop(deadline) ◄── requestIdleCallback │ while (nextUnitOfWork && !shouldYield) ▼ performUnitOfWork(fiber) │ updateFunctionComponent / updateHostComponent ▼ reconcileChildren(wipFiber, elements) │ asigna effectTag: PLACEMENT / UPDATE / DELETION ▼ (sin más unidades) ──► commitRoot() │ commitWork(wipRoot.child) + deletions ▼ DOM real ──► currentRoot = wipRoot CICLO DE UPDATE (setState): setState(action) ──► wipRoot = { ...currentRoot, alternate: currentRoot } ──► nextUnitOfWork = wipRoot ──► (vuelve al workLoop)

El detalle clave del ciclo de actualización es que setState no muta el DOM directamente: encola la acción y reprograma un render partiendo de currentRoot como alternate, de modo que la reconciliación pueda comparar el árbol nuevo contra el ya comprometido.

src/React.ts — el disparador del render
export function render(element, container) {
  wipRoot = {
    dom: container,
    props: { children: [element] },
    alternate: currentRoot
  }
  deletions = [];
  nextUnitOfWork = wipRoot // ejecuta nuevo render
}

Las dos fases

Como en React real, el trabajo se divide en dos fases con propiedades opuestas. Esta separación es la razón de ser de toda la arquitectura Fiber.

  1. Fase de render (pausable, no muta el DOM). Ocurre dentro de workLoop y performUnitOfWork. Construye el árbol de fibers y calcula qué cambios hacen falta, pero no toca el DOM. Puede interrumpirse entre unidades de trabajo cuando deadline.timeRemaining() < 1, cediendo el control al navegador. Si se interrumpiera a mitad, no habría UI inconsistente porque nada se ha aplicado todavía.
  2. Fase de commit (no pausable, muta el DOM). Ocurre en commitRoot. Una vez que nextUnitOfWork es nulo y existe un wipRoot, se aplican todas las mutaciones de golpe, sin interrupción, para garantizar que el usuario nunca vea un árbol a medio construir.
Idea clave

La transición entre fases está en una sola condición del bucle: if (!nextUnitOfWork && wipRoot) { commitRoot(); }. Mientras haya unidades de trabajo, seguimos en render; cuando se agotan, se hace un único commit atómico. Detalle profundo en 10 · Scheduler y workLoop y 14 · Commit phase.

Estado global mutable

El motor no encapsula su estado en una clase: usa variables de módulo mutables, declaradas al principio de src/React.ts. Entender qué guarda cada una es entender la máquina completa.

VariableQué guarda
wipFiber El fiber de componente que se está procesando ahora mismo; da contexto a useState/useEffect para saber a quién pertenecen los hooks.
hookIndex Índice del hook actual dentro del componente; se incrementa en cada llamada para mantener la correspondencia entre hooks por posición.
nextUnitOfWork El próximo fiber que workLoop debe procesar; ser nulo señala el fin de la fase de render.
wipRoot La raíz del árbol «en progreso» (work in progress) que se está construyendo pero aún no se ha comprometido al DOM.
currentRoot La raíz del último árbol ya comprometido; sirve de alternate para comparar en la próxima reconciliación.
deletions Array de fibers viejos marcados para eliminación; se recorre en commitRoot porque el nuevo árbol no los contiene.
wipEffects Lista global de efectos de useEffect pendientes; se vacía y ejecuta al final de commitRoot, después de mutar el DOM.
Cuidado

wipEffects es global y compartido entre todos los componentes montados, no está asociado a cada fiber. Es una simplificación conocida respecto a React real, donde cada componente mantiene su propia lista de efectos. Se detalla en 17 · Hook useEffect y 19 · Diferencias con React y bugs.
