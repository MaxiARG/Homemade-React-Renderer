Dos árboles, dos buffers

Doble buffer (alternate)

Cada render construye un árbol nuevo, pero no crea objetos fiber desde cero: reusa el árbol del render anterior alternando entre dos buffers enlazados por el puntero alternate. Es la misma técnica que React llama double buffering.

El puntero alternate

Existen en todo momento (como máximo) dos versiones de cada fiber: la que está montada en el DOM (current) y la que se está calculando (work-in-progress, o wip). Ambas se apuntan mutuamente mediante alternate. Al terminar el commit, los roles se invierten: el wip pasa a ser el current, y el viejo current queda libre para servir de wip en el siguiente render. Nunca se acumulan más de dos generaciones, así que la presión sobre el recolector de basura es mínima.

createWorkInProgress en detalle

La función que materializa el patrón vive en src/react/reconciler.ts. Recibe el fiber current y las props nuevas, y devuelve un wip listo para procesar:

src/react/reconciler.ts
export function createWorkInProgress(current: Fiber, pendingProps: Props): Fiber {
  let wip = current.alternate;
  if (wip === null) {
    wip = newFiber(current.type, pendingProps, current.key);
    wip.alternate = current;
    current.alternate = wip;
  } else {
    wip.props = pendingProps;
    wip.type = current.type;
    wip.key = current.key;
  }
  // Se hereda lo que persiste entre renders.
  wip.dom = current.dom;
  wip.stateNode = current.stateNode;
  wip.hooks = current.hooks; // el bailout preserva estado; el re-render los resetea.
  wip.root = current.root;
  // Combinamos la marca de "sucio" de ambos buffers: un dispatch pudo marcar el
  // fiber que ahora es el `alternate` (wip) mientras el `current` quedó limpio.
  wip.hasPendingUpdate = current.hasPendingUpdate || wip.hasPendingUpdate;
  // Se resetea lo específico de este render.
  wip.child = null;
  wip.sibling = null;
  wip.parent = null;
  wip.effectTag = 'NONE';
  wip.moved = false;
  wip.index = 0;
  wip.providerContext = null;
  wip.providerForced = false;
  return wip;
}

Dos caminos: crear o reciclar

  1. Primer render de ese fiber (current.alternate === null). No hay buffer que reciclar, así que se crea uno con newFiber y se enlazan mutuamente: wip.alternate = current y current.alternate = wip. A partir de acá el par queda establecido para siempre.
  2. Renders siguientes. El alternate ya existe: es el árbol de hace dos generaciones, que quedó ocioso tras el último commit. Se lo recicla escribiéndole las props, el type y la key nuevos, sin asignar un objeto nuevo.

Qué se hereda y qué se resetea

La distinción es la clave del patrón. Algunos campos describen identidad persistente (el nodo del DOM, la instancia de clase, los hooks) y deben viajar entre renders; otros describen la estructura de este render en particular (hijos, hermanos, banderas) y deben partir limpios para no arrastrar basura del render anterior.

CampoTratoPor qué
domHeredaEl nodo real del DOM ya existe; reusarlo es lo que evita recrear elementos.
stateNodeHeredaLa instancia de un componente de clase debe sobrevivir entre renders.
hooksHeredaEl estado de useState/useRef/etc. vive acá; se comparte con el current y se resetea recién al re-ejecutar el componente.
rootHeredaEl contenedor de estado del root es el mismo durante toda la vida del árbol.
child, sibling, parentReseteaLa forma del árbol se recalcula en reconcileChildren de este render.
effectTagResetea a NONELa bandera de mutación pertenece a un solo commit.
moved, indexReseteaLa posición y el movimiento los redecide la reconciliación por key.
providerContext, providerForcedReseteaEl manejo de la pila de contextos es propio de cada pasada.

La línea sutil: hasPendingUpdate

De todos los campos, el más delicado es hasPendingUpdate. No se hereda a secas ni se resetea: se combina con un OR lógico.

src/react/reconciler.ts
wip.hasPendingUpdate = current.hasPendingUpdate || wip.hasPendingUpdate;

La razón es el desfase entre buffers. Cuando un dispatch de useState marca un fiber como "sucio" (con actualización pendiente), marca la versión que estaba montada en ese instante. Pero para el próximo render, esa versión puede ser justamente la que ahora recibimos como alternate (el wip), no el current. Si simplemente reseteáramos wip.hasPendingUpdate a false, perderíamos la señal de que ese componente tiene un setState pendiente y canBailout lo saltaría por error. Al combinar ambas marcas con OR, la actualización pendiente sobrevive sin importar en qué buffer se registró. Este flag es el que 25 · Bailout y memoización consulta para decidir si un fiber puede saltarse.

Por qué importa

Sin ese OR, un patrón como "click rápido, dos setState en renders consecutivos" podría intercalar buffers de modo que la segunda actualización se pierda silenciosamente. El OR garantiza que el estado "sucio" sea una propiedad del componente lógico, no del objeto fiber físico que casualmente lo represente en esta pasada.

Clonar sin re-ejecutar: cloneChildFibers

El doble buffer también habilita el bailout. Cuando un fiber no necesita re-ejecutarse, sus hijos igual deben existir en el árbol wip para que el recorrido continúe. En vez de reconciliar, se clonan a partir del árbol current llamando a createWorkInProgress con las mismas props (por eso se preservan tal cual):

src/react/reconciler.ts
function cloneChildFibers(current: Fiber, wip: Fiber): void {
  let currentChild = current.child;
  let prev: Fiber | null = null;
  while (currentChild) {
    const clone = createWorkInProgress(currentChild, currentChild.props);
    clone.parent = wip;
    clone.index = currentChild.index;
    if (prev === null) wip.child = clone;
    else prev.sibling = clone;
    prev = clone;
    currentChild = currentChild.sibling;
  }
}

Cada clon reusa el alternate de su hijo current, hereda su DOM, hooks y estado, y se reengancha bajo el nuevo wip. Como las props son idénticas por referencia, el clon a su vez podrá volver a hacer bailout hacia abajo, propagando el ahorro por todo el subárbol sin ejecutar ningún componente.

El ciclo current ⇄ alternate

render N commit N current ───alternate──▶ wip wip pasa a ser current (montado) (en cálculo) (el viejo current queda libre) A ◀───────────────▶ A' A' ◀───────────────▶ A current wip current (alternate ocioso) render N+1: se recicla el alternate ocioso como nuevo wip A ───alternate──▶ A' (mismos dos objetos, roles invertidos otra vez) Nunca se asignan objetos fiber nuevos salvo en el PRIMER render de cada nodo.
Comparación con React

React implementa exactamente esta técnica: dos árboles (current y workInProgress) enlazados por alternate, y un intercambio de raíz al final del commit (root.current = finishedWork). Nuestro motor hace lo propio en commitRoot con root.currentRoot = root.wipRoot. La motivación es idéntica: renders repetidos no deben inundar el GC con objetos efímeros.
