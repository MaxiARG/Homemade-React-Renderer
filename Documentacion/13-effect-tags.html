Qué hacer con cada nodo

Effect tags

Un effectTag es la etiqueta que la reconciliación deja pegada a cada fiber para decirle al commit qué operación aplicar sobre el DOM. Solo existen tres valores: PLACEMENT, UPDATE y DELETION.

Un puente entre fases

La render phase no toca el DOM: solo calcula. Ese cálculo necesita registrarse en algún sitio para que la commit phase sepa qué hacer sin volver a razonar el diff. El effectTag es exactamente ese registro: una instrucción de una sola palabra grabada en el fiber durante la reconciliación y leída, más tarde, durante el commit.

effectTag Cuándo se asigna (reconcileChildren) Qué hace el commit (commitWork)
PLACEMENT element && !sameType: hay elemento nuevo sin viejo del mismo tipo. domParent.appendChild(fiber.dom): inserta el nodo nuevo en su padre.
UPDATE sameType: mismo tipo en la misma posición; se reutiliza el dom viejo. updateDom(fiber.dom, fiber.alternate.props, fiber.props): aplica solo las props cambiadas.
DELETION oldFiber && !sameType: sobra un fiber viejo; se marca y se empuja a deletions. commitDeletion(fiber, domParent): corre cleanups y hace removeChild.

Dónde se asignan los tags

Los tres tags nacen dentro de reconcileChildren, cada uno en su rama. Para UPDATE y PLACEMENT el tag se escribe en el newFiber recién creado; para DELETION, como no existe fiber nuevo que represente al nodo borrado, el tag se pega directamente sobre el oldFiber.

src/React.ts — reconcileChildren
if (sameType) {
  newFiber = {
    type: oldFiber.type,
    props: element.props,
    dom: oldFiber.dom,
    parent: wipFiber,
    alternate: oldFiber,
    effectTag: "UPDATE",
  }
}
if (element && !sameType) {
  newFiber = {
    type: element.type,
    props: element.props,
    dom: null,
    parent: wipFiber,
    alternate: null,
    effectTag: "PLACEMENT",
  }
}
if (oldFiber && !sameType) {
  oldFiber.effectTag = "DELETION"
  deletions.push(oldFiber)
}

El array global deletions

Los PLACEMENT y UPDATE viven en el árbol nuevo, el wipRoot, así que el commit los encuentra recorriéndolo. Pero un nodo borrado no forma parte del árbol nuevo: precisamente por eso desaparece. Si dependiéramos solo del recorrido del wipRoot nunca visitaríamos los nodos a eliminar y quedarían colgados en el DOM. La solución es un array global, deletions, donde se acumulan los fibers marcados con DELETION para que el commit pueda procesarlos aparte.

Idea clave

El comentario del propio código lo resume: “when we commit the fiber tree to the DOM we do it from the work in progress root, which doesn't have the old fibers”. Por eso hace falta la lista aparte.

deletions se reinicia a [] en cada nuevo render, tanto en render como dentro del setState de useState, de modo que cada ciclo empieza con su propia lista limpia de eliminaciones. El commit la vacía antes que nada:

src/React.ts — commitRoot
function commitRoot() {
  deletions.forEach(commitWork)
  commitWork(wipRoot.child)
  currentRoot = wipRoot
  wipRoot = null
  // ...ejecutar useEffects...
}

Nótese el orden: primero deletions.forEach(commitWork) procesa las bajas y solo después commitWork(wipRoot.child) recorre el árbol nuevo aplicando PLACEMENT y UPDATE. En commitWork, el effectTag se lee con una cadena de if/else if que despacha cada caso a su operación de DOM.

Nota

Un fiber sin effectTag (por ejemplo la raíz, o un componente de función que no tiene dom) simplemente no cae en ninguna rama de commitWork: no se aplica ninguna mutación sobre él, aunque su recorrido continúe hacia child y sibling. El detalle completo del recorrido está en la Commit phase.
