Rendering interrumpible

Scheduler y workLoop

El workLoop es el bucle que va procesando fibers de a poco, cediendo el hilo al navegador cuando se acaba el tiempo libre de cada frame. Es lo que convierte el render en una tarea cooperativa en vez de un bloqueo monolítico.

El bucle real

El scheduler de este clon cabe en una función. Se apoya en requestIdleCallback del navegador y procesa unidades de trabajo mientras haya trabajo pendiente y quede tiempo en el frame:

src/React.ts — workLoop
function workLoop(deadline) {
  let shouldYield = false
  while (nextUnitOfWork && !shouldYield) {
    nextUnitOfWork = performUnitOfWork(
      nextUnitOfWork
    )
    shouldYield = deadline.timeRemaining() < 1
  }

  if (!nextUnitOfWork && wipRoot) {
    commitRoot();
  }

  requestIdleCallback(workLoop)
}

//aca es donde se ejecuta el workloop cuando el navegador esta idle
// this function is one with the lowest priority to execute.
requestIdleCallback(workLoop)

requestIdleCallback: trabajar en los huecos

requestIdleCallback es una API del navegador que ejecuta una función cuando el hilo principal está ocioso, es decir, cuando ha terminado sus tareas prioritarias (procesar eventos, calcular layout, pintar) y le sobra tiempo antes del siguiente frame. Como dice el comentario del código, es «la función con la prioridad más baja para ejecutarse»: no compite con el trabajo urgente del navegador, sino que aprovecha sus ratos libres.

Al invocar el callback, el navegador le pasa un objeto deadline que responde deadline.timeRemaining(): los milisegundos que quedan de tiempo ocioso en este frame. Esa es la brújula que el bucle usa para saber cuándo detenerse. La última línea, requestIdleCallback(workLoop), arranca todo el mecanismo la primera vez que se carga el módulo.

El bucle while y shouldYield

El corazón es while (nextUnitOfWork && !shouldYield). Se procesa un fiber por iteración: performUnitOfWork hace el trabajo de ese fiber y devuelve el siguiente (siguiendo child → sibling → tío, como vimos en el recorrido del árbol). El bucle continúa mientras se cumplan dos condiciones:

  1. Que haya trabajo: nextUnitOfWork no es null.
  2. Que no toque ceder: shouldYield sigue en false.

Después de cada unidad se recalcula shouldYield = deadline.timeRemaining() < 1. Cuando queda menos de un milisegundo de tiempo ocioso, shouldYield pasa a true, el while se corta y la función devuelve el control al navegador. El estado del recorrido no se pierde: vive en la variable global nextUnitOfWork. En el siguiente rato ocioso, requestIdleCallback vuelve a llamar a workLoop y este retoma exactamente donde quedó.

Nota

Ceder el hilo (yield) significa devolvérselo al navegador para que pueda responder a scroll, clics o animaciones sin congelarse. El render «espera su turno» en lugar de acaparar el CPU.

Cuando el trabajo termina: commitRoot

Tras el while, el bucle evalúa if (!nextUnitOfWork && wipRoot). La combinación es significativa: no queda ninguna unidad de trabajo pendiente (nextUnitOfWork es null) pero todavía existe un árbol en progreso sin publicar (wipRoot no es null). Eso significa que se terminó de construir todo el árbol nuevo y llegó el momento de volcarlo al DOM de una sola vez: se llama a commitRoot(). Todo el trabajo previo solo calculó qué cambiar; el commit es el único punto donde se toca el DOM real.

Scheduling cooperativo: dividir el render en chunks

El efecto neto de este diseño es que un render grande se reparte en chunks a lo largo de varios frames. En vez de procesar los cientos de fibers de un tirón —lo que congelaría la interfaz—, el motor hace un puñado de fibers, cede, deja que el navegador respire y pinte, y continúa en el siguiente hueco ocioso. A esto se lo llama scheduling cooperativo: el render y el navegador se turnan el hilo principal de mutuo acuerdo.

Contraste con el Stack Reconciler

Antes de Fiber, React (versión 15 y anteriores) usaba el Stack Reconciler. Era un recorrido recursivo y síncrono: React descendía por el árbol de componentes usando la pila de llamadas del navegador, sin pausas. Una vez iniciado, no podía suspenderse. Las consecuencias eran directas:

Stack Reconciler (React ≤ 15)workLoop de este clon (Fiber)
Recursivo, apoyado en el call stackIterativo, apoyado en enlaces de fibers
Síncrono y bloqueante: procesa «de golpe»Interrumpible: cede con shouldYield
No se puede pausar ni retomarSe retoma vía nextUnitOfWork
Bloquea el hilo si el árbol es grandeSe reparte en chunks entre frames

Como resume la documentación de conceptos del proyecto, con el Stack Reconciler «no podías pausar el trabajo para dejar que el navegador respondiera a eventos como scroll, animaciones o entrada del usuario», y «si un render grande estaba en progreso, el navegador podía congelarse». Fiber reemplazó ese modelo precisamente para hacer el renderizado interrumpible mediante cooperative scheduling, y para separar el render del commit: calcular qué cambiar por un lado, aplicar los cambios al DOM por otro.

Cuidado

Durante la fase de render el motor no muta el DOM. Cada performUnitOfWork solo crea fibers en memoria y calcula effectTag. Si el DOM se tocara aquí, ceder el hilo a mitad de camino dejaría al usuario viendo una UI incompleta. Por eso toda mutación se difiere al commit phase, que se ejecuta de un solo golpe en commitRoot() cuando ya no queda trabajo pendiente.
