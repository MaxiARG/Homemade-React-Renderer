Leer contexto

useContext — ciclo de vida completo

useContext lee el valor que el Provider más cercano dejó activo durante el recorrido del árbol. Es el hook más simple del motor: no guarda estado propio, no se suscribe a nada; lee context._currentValue y confía en que el reconciliador fuerce el re-render cuando el valor cambia.

Firma y para qué sirve

La firma real en src/react/hooks.ts es useContext<T>(context: ReactContext<T>): T. Recibe el objeto de contexto creado con createContext y devuelve su valor actual. Sirve para consumir datos "ambientales" (tema, usuario, idioma) sin pasarlos manualmente por props a través de cada nivel del árbol.

uso mínimo
const Theme = createContext('light');

function Consumer() {
  const theme = useContext(Theme); // lee el value del Provider más cercano
  return h('span', null, theme);
}

function App() {
  const [theme, setTheme] = useState('light');
  return h(Theme.Provider, { value: theme }, h(Consumer));
}

Dónde vive el estado

El estado del hook es un ContextHook. Nótese que no almacena estado mutable ni una queue: es apenas un registro del contexto leído y del valor obtenido en este render.

src/react/types.ts
export interface ContextHook<T = unknown> {
  tag: 'context';
  context: ReactContext<T>;
  value: T;
}

El valor "vivo" no vive en el hook sino en el propio objeto de contexto, en el campo _currentValue:

src/react/types.ts
export interface ReactContext<T> {
  _currentValue: T;   // el valor activo AHORA (lo setea el Provider durante el recorrido)
  _defaultValue: T;   // el valor por defecto de createContext
  Provider: FunctionComponent<{ value: T; children?: ReactElement[] }>;
  _isContextProvider?: boolean;
}

Aunque el ContextHook se reconstruye en cada render y se recupera por posición desde wipFiber.alternate.hooks[hookIndex] como cualquier otro hook, su value no se "recuerda": siempre refleja lo que _currentValue tenga en el instante de la lectura.

Primer render (mount) y siguientes: una sola lectura

El cuerpo entero del hook es una lectura directa; no hay rama de mount vs. update porque no depende del hook anterior:

src/react/hooks.ts
export function useContext<T>(context: ReactContext<T>): T {
  const { fiber } = slot();
  const value = context._currentValue;
  const hook: ContextHook<T> = { tag: 'context', context, value };
  commitHook(fiber, hook as unknown as Hook);
  return value;
}
  1. slot() obtiene R.wipFiber (no necesita el hook viejo).
  2. Se lee context._currentValue: el valor que el Provider dejó activo.
  3. Se crea el ContextHook como registro y commitHook avanza R.hookIndex.
  4. Se devuelve value.

Quién setea _currentValue: el Provider durante el recorrido

La clave está en updateContextProvider (ver Context API). Al entrar a un Provider, el reconciliador empuja el valor previo a contextStack y escribe el nuevo en _currentValue; al salir, completeWork lo restaura. Como el recorrido es en profundidad, cualquier useContext que se ejecute dentro del subárbol del Provider lee el valor correcto:

src/react/reconciler.ts (updateContextProvider)
contextStack.push({ context, value: context._currentValue }); // guarda el anterior
context._currentValue = value;                                 // activa el nuevo
fiber.providerContext = context;

if (fiber.alternate === null || !Object.is(prevValue, value)) {
  R.forceRenderDepth++;         // el value cambió: deshabilita el bailout del subárbol
  fiber.providerForced = true;
}

La actualización / disparador: no hay suscripción explícita

useContext no se suscribe al contexto ni registra un listener. Entonces, ¿cómo se entera un consumidor de que el value cambió? El disparador es el setState del componente que renderiza el Provider con un value distinto. Ese re-render vuelve a recorrer el subárbol, y cuando pasa por el Provider detecta !Object.is(prevValue, value) e incrementa R.forceRenderDepth. Mientras esa profundidad es > 0, el bailout queda deshabilitado:

src/react/reconciler.ts (canBailout)
function canBailout(fiber: Fiber): boolean {
  const current = fiber.alternate;
  if (current === null) return false;
  if (fiber.hasPendingUpdate) return false;
  if (R.forceRenderDepth > 0) return false; // dentro de un Provider cuyo value cambió
  return fiber.props === current.props;
}
Por qué no hace falta suscripción

En vez de mantener una lista de consumidores y notificarlos (lo que hace React real de forma optimizada), este motor usa una estrategia más simple: cuando el value cambia, forceRenderDepth obliga a re-renderizar todo el subárbol del Provider. Como cada consumidor vuelve a ejecutarse, su useContext vuelve a leer _currentValue y obtiene el valor nuevo. Es correcto, aunque menos granular: incluso componentes que no consumen el contexto se re-renderizan.

Diferencia con React real

React real propaga los cambios de contexto solo a los suscriptores mediante un mecanismo de "lanes" y listas de dependencias, permitiendo que ramas sin consumidores hagan bailout. Acá el precio de la simplicidad es re-renderizar el subárbol completo del Provider que cambió.

Orden de ejecución y reglas de los hooks

useContext corre en línea durante el render del consumidor, en el momento de la llamada. Para que la lectura sea correcta, el Provider ya tuvo que haber seteado _currentValue antes (garantizado por el orden del recorrido: el Provider es ancestro del consumidor). Como consume un slot vía commitHook y avanza R.hookIndex, debe llamarse incondicionalmente: llamarlo dentro de un if correría el índice y rompería la correspondencia posicional con el resto de los hooks del componente.

Test que lo fija

El test "lee el valor del provider y re-renderiza al cambiar" monta App con Theme.Provider value={theme} y un Consumer que hace useContext(Theme). Tras el mount container.textContent es 'light'; un setTheme('dark') (setState en App, que cambia el value del Provider) fuerza el re-render del subárbol y el Consumer vuelve a leer _currentValue, dando 'dark'.

Diagrama de secuencia

MOUNT (theme = 'light') App render → Theme.Provider value='light' updateContextProvider: push viejo; _currentValue = 'light' Consumer render → useContext(Theme) value = _currentValue = 'light' ──► <span>light</span> completeWork(Provider): pop → restaura _currentValue DISPARADOR: setTheme('dark') (setState en App) scheduleUpdate(App) → re-render del subárbol de App App render → Theme.Provider value='dark' prevValue='light', value='dark' → !Object.is → forceRenderDepth++ (canBailout = false mientras forceRenderDepth > 0) Consumer render (NO puede hacer bailout) → useContext(Theme) value = _currentValue = 'dark' ──► <span>dark</span> completeWork(Provider): forceRenderDepth-- ; pop restaura _currentValue (useContext nunca se suscribe: solo re-lee _currentValue en cada render forzado.)
