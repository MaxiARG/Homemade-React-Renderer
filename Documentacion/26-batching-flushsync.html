Agrupar actualizaciones

Batching y flushSync

Varios setState disparados en el mismo tick no producen varios renders: se acumulan en un conjunto de roots «sucios» y se drenan de una vez en un queueMicrotask. flushSync es la puerta de escape que fuerza el trabajo a correr sincrónicamente.

El problema que resuelve el batching

Si cada setState arrancara su propia render phase, un manejador de eventos que actualiza tres estados provocaría tres renders y tres commits consecutivos, dos de ellos desechados de inmediato. El batching agrupa todas las actualizaciones que ocurren antes de que el motor tenga la oportunidad de correr, y las colapsa en un único render. El scheduler del motor implementa esto con tres piezas de estado a nivel de módulo: un conjunto de roots pendientes, una bandera de «ya hay un flush agendado» y un contador de profundidad de flushSync.

src/react/scheduler.ts
const pendingRoots = new Set<RootStateContainer>();
let flushScheduled = false;
let batchDepth = 0;

/** Llamado por los hooks/clases cuando hay una actualización de estado. */
export function scheduleUpdate(fiber: Fiber): void {
  const root = fiber.root;
  if (!root) return;
  root.dirtyFibers.add(fiber);
  pendingRoots.add(root);
  ensureFlush();
}

Cuando un dispatch de useState marca su fiber como hasPendingUpdate y llama a scheduleUpdate, no se renderiza nada en el acto. El fiber se añade a root.dirtyFibers, el root al conjunto pendingRoots, y todo el trabajo real queda a cargo de ensureFlush. Un mismo root añadido dos veces al Set sigue siendo una sola entrada: ahí ya empieza a manifestarse el agrupamiento.

ensureFlush: un solo microtask por lote

src/react/scheduler.ts
function ensureFlush(): void {
  if (batchDepth > 0) return; // dentro de flushSync: se drena al salir
  if (flushScheduled) return;
  flushScheduled = true;
  queueMicrotask(() => {
    flushScheduled = false;
    flushAll();
  });
}

Aquí está el corazón del batching. La bandera flushScheduled garantiza que, aunque se llame a scheduleUpdate diez veces seguidas, solo se agenda un queueMicrotask. Como un microtask no corre hasta que la pila de llamadas sincrónica se vacía, todos los setState de un mismo tick (por ejemplo, los de un mismo manejador de eventos) terminan de ejecutarse antes de que el callback del microtask empiece. Para cuando flushAll corre, pendingRoots ya contiene todos los roots afectados y cada uno se procesa una única vez.

La primera guarda, if (batchDepth > 0) return, es la que conecta el batching con flushSync: si ya estamos dentro de un flushSync, no tiene sentido agendar un microtask, porque el trabajo se va a drenar sincrónicamente al salir. Esa es la razón de ser del contador batchDepth.

flushSync: drenar todo, ya

src/react/scheduler.ts
/** Ejecuta un callback y drena TODO el trabajo pendiente de forma síncrona. */
export function flushSync<T>(fn?: () => T): T | undefined {
  batchDepth++;
  let result: T | undefined;
  try {
    if (fn) result = fn();
  } finally {
    batchDepth--;
    if (batchDepth === 0) flushAll();
  }
  return result;
}

flushSync incrementa batchDepth, ejecuta el callback (durante el cual cualquier setState solo encola trabajo, sin agendar microtasks gracias a la guarda de ensureFlush), y en el finally decrementa el contador. Cuando batchDepth vuelve a cero — es decir, al salir del flushSync más externo — llama a flushAll para procesar sincrónicamente todo lo acumulado. El contador, en vez de un simple booleano, permite anidar llamadas a flushSync: solo el más externo dispara el drenado.

Este es el mecanismo que usa el render inicial (para que el primer árbol aparezca de inmediato, sin esperar a un microtask) y, sobre todo, los tests: envolver una acción en flushSync hace que sus efectos sobre el DOM sean observables en la línea siguiente, sin necesidad de await ni de temporizadores.

flushAll y performRootWork

src/react/scheduler.ts
function flushAll(): void {
  // Copiamos porque performRootWork puede re-agendar (efectos con setState).
  const roots = Array.from(pendingRoots);
  pendingRoots.clear();
  for (const root of roots) performRootWork(root);
}

function performRootWork(root: RootStateContainer): void {
  if (!root.wipRoot) {
    if (!root.currentRoot) return;
    prepareFreshWip(root);
  }
  R.activeRoot = root;
  workLoopSync(root);
  commitRoot(root);
  root.dirtyFibers.clear();
  R.activeRoot = null;
}

flushAll copia pendingRoots a un array y lo limpia antes de iterar. El comentario explica el porqué: durante performRootWork puede correr un useEffect que a su vez llame a setState, lo que volvería a poblar pendingRoots; copiar primero evita procesar ese nuevo trabajo dentro del mismo bucle de forma inconsistente. Para cada root, performRootWork prepara un work-in-progress fresco si hace falta (prepareFreshWip), corre el workLoopSync completo, comitea y limpia dirtyFibers.

src/react/scheduler.ts
function workLoopSync(root: RootStateContainer): void {
  while (root.nextUnitOfWork) {
    try {
      root.nextUnitOfWork = performUnitOfWork(root.nextUnitOfWork);
    } catch (error) {
      handleRenderError(root, error);
    }
  }
}

A diferencia del workLoop original, que troceaba el trabajo según el tiempo disponible del frame, workLoopSync es un bucle sin interrupciones: recorre todos los performUnitOfWork del árbol de un tirón. El nombre lo dice: es render síncrono. El try/catch canaliza cualquier excepción hacia el Error Boundary más cercano.

El test que lo demuestra

El test «batching: dos setState en un tick = un solo re-render» aísla exactamente este comportamiento contando cuántas veces se ejecuta el cuerpo del componente:

src/react/__tests__/engine.test.ts
const { container } = mount(h(App));
expect(renders).toBe(1);
flushSync(() => {
  setA(1);
  setB(2);
});
expect(container.textContent).toBe('1-2');
expect(renders).toBe(2); // no 3

Dentro del flushSync, setA(1) y setB(2) encolan sus acciones y añaden el mismo root a pendingRoots. Como batchDepth > 0, ninguno agenda un microtask. Al cerrarse el flushSync, un único flushAll procesa el root una sola vez: el contador renders pasa de 1 a 2, no a 3. Dos actualizaciones, un render.

Línea de tiempo

tick sincrónico │ cola de microtasks ─────────────────────────────────────────┼─────────────────────── setState(A) │ └─ scheduleUpdate → pendingRoots={r} │ └─ ensureFlush → flushScheduled=true ─┼─▶ [ flushAll ] setState(B) │ └─ scheduleUpdate → pendingRoots={r} │ (mismo Set, misma entrada) └─ ensureFlush → flushScheduled ya │ es true ⇒ NO agenda otro microtask │ │ (la pila se vacía) ─────────────────────┼─▶ corre el microtask: │ flushAll() │ └─ performRootWork(r) │ ├─ workLoopSync (1 render) │ └─ commitRoot (1 commit)
Comparación con React

React 18 introdujo el batching automático: antes solo se agrupaban las actualizaciones dentro de manejadores de eventos de React, y las que ocurrían en promesas, setTimeout o callbacks nativos escapaban del lote. Desde React 18, con createRoot, cualquier grupo de setState se agrupa sin importar el origen. Este motor logra un efecto parecido apoyándose en queueMicrotask en vez del scheduler de prioridades de React, y expone flushSync con la misma intención que ReactDOM.flushSync: salir del lote y aplicar los cambios de inmediato.

Diferencia

El batching aquí es «todo o nada» por microtask, no interrumpible ni por prioridades: no hay lanes, ni transiciones (useTransition), ni render concurrente. workLoopSync corre hasta terminar el árbol. Es suficiente para colapsar múltiples setState en un render, pero no ofrece las garantías de responsividad del scheduler real de React.
