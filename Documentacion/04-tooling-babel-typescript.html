De TSX a JavaScript

Tooling: Babel + TypeScript

El navegador no entiende JSX ni TypeScript. Dos herramientas se encargan de traducir el código fuente a JavaScript ejecutable: Babel realiza la transpilación y TypeScript aporta el sistema de tipos. Ambas comparten una decisión central: el pragma React.createElement.

La configuración de Babel

El archivo .babelrc declara tres presets, y cada uno resuelve un problema distinto de la traducción:

.babelrc
{
  "presets": [
    ["@babel/preset-env", { "modules": false }],
    ["@babel/preset-react", { "pragma": "React.createElement" }],
    "@babel/preset-typescript"
  ]
}
  • @babel/preset-env con { "modules": false }: transpila la sintaxis moderna de JavaScript a una versión ampliamente compatible, pero deja intactos los import/export ES nativos en lugar de convertirlos a CommonJS.
  • @babel/preset-react con { "pragma": "React.createElement" }: transforma cada etiqueta JSX en una llamada a la función indicada por el pragma. Esta es la pieza que conecta la sintaxis declarativa con el motor del proyecto.
  • @babel/preset-typescript: elimina las anotaciones de tipo de los archivos .ts/.tsx. Nótese que no comprueba tipos: solo los borra. La verificación es responsabilidad de TypeScript.

La configuración de TypeScript

El tsconfig.json mantiene la coherencia con Babel. Lo crítico es que su jsxFactory apunta exactamente a la misma función que el pragma de Babel:

tsconfig.json
{
  "compilerOptions": {
    "jsx": "react",
    "jsxFactory": "React.createElement",
    "jsxFragmentFactory": "React.Fragment",
    "target": "es2020",
    "lib": ["es2020", "dom"]
  }
}
  • "jsx": "react": indica a TypeScript que interprete la sintaxis JSX según el modelo clásico de React, emitiendo llamadas a una fábrica.
  • "jsxFactory": "React.createElement": nombra esa fábrica. Debe coincidir con el pragma de Babel para que el tipado y la transpilación describan la misma transformación.
  • "target": "es2020" y "lib": ["es2020", "dom"]: fijan el nivel del lenguaje y exponen las APIs del DOM que el motor usa (document.createElement, requestIdleCallback, etc.).
Cuidado

jsxFragmentFactory apunta a React.Fragment, pero ese símbolo no está implementado en src/React.ts: aparece únicamente referenciado en la declaración global de Window como opcional. Usar la sintaxis de fragmento <>...</> fallaría en tiempo de ejecución. Es una de las limitaciones conocidas que se recogen en 19 · Diferencias con React y bugs.

Los scripts del proyecto

El package.json define cómo se compila y sirve el proyecto. Son dos pasos encadenados sobre el CLI de Babel:

package.json
"scripts": {
  "build": "babel src --out-dir public --extensions \".js,.jsx,.ts,.tsx\"",
  "start": "npm run build && npx serve public",
  "format": "prettier --write ."
}
  • build: invoca Babel sobre todo el directorio src y vuelca el resultado en public, procesando las cuatro extensiones .js,.jsx,.ts,.tsx. Cada src/App.tsx se convierte en public/App.js, src/React.ts en public/React.js, y así sucesivamente.
  • start: ejecuta primero build y luego sirve el directorio public con npx serve, de modo que el navegador cargue el JavaScript ya transpilado.

Del TSX a las llamadas createElement

El corazón del tooling es la transformación que hace el pragma. Cuando escribimos en src/App.tsx algo como:

src/App.tsx
<div style="background: salmon">
  <h1>Construyendo React de manera casera</h1>
  <Counter />
</div>

Babel lo reescribe, apoyándose en el pragma, en llamadas anidadas equivalentes a: 

React.createElement(
  "div",
  { style: "background: salmon" },
  React.createElement("h1", null, "Construyendo React de manera casera"),
  React.createElement(Counter, null)
)

Cada etiqueta se convierte en una invocación de React.createElement con tres partes: el type (una cadena para elementos del host como "div", o la referencia a la función para componentes como Counter), las props, y los children como argumentos restantes. Es exactamente la firma que espera la función real del motor:

src/React.ts
export function createElement(
  type: string,
  props: Props,
  ...children: any[]
): Element { /* ... */ }
Idea clave

El pragma es el puente entre dos mundos: la sintaxis declarativa de JSX y la API imperativa del motor. Sin él, Babel y TypeScript no sabrían a qué función traducir las etiquetas. Por eso el mismo nombre —React.createElement— aparece idénticamente en el .babelrc y en el tsconfig.json: cualquier discrepancia rompería la traducción. El mecanismo se estudia en profundidad en 05 · JSX y el pragma createElement.

Nota

¿Por qué { "modules": false }? Porque el proyecto usa módulos ES nativos: src/App.tsx hace import { React } from './React.js' y el navegador los carga directamente vía <script type="module">. Si Babel convirtiera los import/export a CommonJS (require/module.exports), el resultado no funcionaría en el navegador sin un empaquetador. Dejar modules: false preserva la semántica ES y permite servir el código sin bundler alguno.
