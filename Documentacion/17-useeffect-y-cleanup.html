Efectos secundarios

Hook useEffect y cleanup

Un componente puro solo describe interfaz. Pero el mundo real exige suscripciones, temporizadores y registros: efectos que ocurren fuera del cálculo del árbol. useEffect es el mecanismo que este motor ofrece para ejecutar ese trabajo después de mutar el DOM, y para deshacerlo limpiamente con una función de cleanup.

La firma real del hook

A diferencia de useState, que sí produce y devuelve estado, useEffect no retorna nada: su labor consiste en registrar un hook en la lista de hooks del fiber en curso, para que la fase de commit decida más tarde si debe ejecutarlo. Este es el código completo tal como aparece en src/React.ts:

src/React.ts
export function useEffect(effect, deps) {
  const oldHook =
    wipFiber.alternate &&
    wipFiber.alternate.hooks &&
    wipFiber.alternate.hooks[hookIndex];

  // Solo comparamos contra oldHook.deps si el hook anterior REALMENTE tenía deps.
  const hasChanged = (oldHook && oldHook.deps)
    ? (!deps || deps.some((dep, i) => !Object.is(dep, oldHook.deps[i])))
    : true;

  const hook = {
    effect,
    deps,
    hasChanged,
    cleanup: oldHook?.cleanup, // Copia el cleanup anterior si existe
  };

  wipFiber.hooks.push(hook);
  hookIndex++;
}

Igual que useState, lo primero es recuperar el oldHook: el hook que este mismo useEffect registró en el render anterior. Se localiza por posición, usando wipFiber.alternate.hooks[hookIndex]. Esa dependencia del orden es exactamente la razón por la que los hooks no pueden llamarse dentro de condicionales: el hookIndex debe coincidir entre renders.

El cálculo de hasChanged

La decisión de re-ejecutar el efecto se reduce a comparar el array de dependencias actual (deps) contra el del render previo (oldHook.deps), elemento por elemento, con Object.is. Si alguna dependencia difiere, deps.some(...) devuelve true y el efecto debe volver a correr. Si el hook no tenía versión anterior (primer montaje), hasChanged es true directamente: todo efecto se ejecuta al menos una vez.

Object.is se elige, en lugar de ===, por la misma razón que React real: distingue correctamente casos límite como NaN (donde NaN === NaN es false pero Object.is(NaN, NaN) es true) y los dos ceros con signo. Es una comparación superficial (shallow): compara referencias, no contenido profundo de objetos.

Bug corregido

useEffect sin deps previas. La versión original calculaba hasChanged leyendo oldHook.deps[i] sin verificar que el hook anterior tuviera deps. Al pasar de useEffect(fn) (sin deps) a useEffect(fn, [x]), el código indexaba undefined[i] y lanzaba una excepción. La corrección introduce la guarda (oldHook && oldHook.deps) ? … : true: solo se comparan dependencias cuando el render previo realmente registró un array de deps; en cualquier otro caso el efecto se considera cambiado y se ejecuta.

Recolección de efectos en updateFunctionComponent

useEffect solo registra; no ejecuta. Tras reconciliar los hijos del componente función, el motor recorre sus hooks y encola en el array global wipEffects aquellos que sean efectos y hayan cambiado:

src/React.ts — updateFunctionComponent
wipFiber.hooks.forEach(hook => {
  if (hook.effect && hook.hasChanged) {
    wipEffects.push(() => {
      // Ejecutar cleanup anterior si existe
      if (typeof hook.cleanup === 'function') {
        hook.cleanup();
      }
      // Ejecutar el efecto y guardar el nuevo cleanup si devuelve uno
      const cleanup = hook.effect();
      if (typeof cleanup === 'function') {
        hook.cleanup = cleanup;
      } else {
        hook.cleanup = undefined;
      }
    });
  }
});

Observa el orden encapsulado dentro de la función que se encola: primero se ejecuta el cleanup anterior (si lo había), y solo después se ejecuta el efecto nuevo. Si el efecto devuelve una función, se guarda como hook.cleanup para la próxima ronda; si no devuelve nada, el cleanup se descarta. Este es exactamente el contrato de React: la función de limpieza corre antes de cada re-ejecución del efecto.

Ejecución diferida en commitRoot

El array wipEffects no se vacía durante el render. Se drena al final de commitRoot, después de haber aplicado todas las mutaciones al DOM:

src/React.ts — commitRoot
function commitRoot() {
  deletions.forEach(commitWork)
  commitWork(wipRoot.child)
  currentRoot = wipRoot
  wipRoot = null

  // Ejecutar todos los useEffect. Siempre luego de haber ejecutado todo lo demas.
  wipEffects.forEach(e => e());
  wipEffects = [];
}

Ese diferimiento es intencional: garantiza que cuando el efecto corre, el DOM ya refleja el render actual, de modo que el efecto puede medir nodos, adjuntar listeners externos o registrar valores ya consolidados. En React real este es el rol de los passive effects, que se ejecutan tras la fase de commit.

Cleanup en el desmontaje

Hay un segundo momento en que el cleanup debe correr: cuando el componente se elimina del árbol y ya no volverá a renderizar. Esto ocurre en commitDeletion, que recorre los hooks del fiber a borrar y ejecuta sus cleanups antes de quitar el nodo del DOM:

src/React.ts — commitDeletion
function commitDeletion(fiber, domParent) {
  // Ejecutar cleanups de hooks antes de eliminar el nodo
  if (fiber.hooks) {
    fiber.hooks.forEach(hook => {
      if (typeof hook.cleanup === 'function') {
        hook.cleanup();
      }
    });
  }
  if (fiber.dom) {
    domParent.removeChild(fiber.dom);
  } else if (fiber.child) {
    commitDeletion(fiber.child, domParent);
  }
}

Así el cleanup se ejecuta en las dos situaciones que promete React: antes de re-ejecutar el efecto, y una última vez al desmontar el componente.

Semántica del array de dependencias

El segundo argumento de useEffect gobierna con qué frecuencia se ejecuta el efecto. Los tres casos surgen naturalmente del cálculo de hasChanged:

depsCuándo se ejecuta
ausente (undefined)En cada render. Con !deps, hasChanged es siempre true.
[] (vacío)Una sola vez, al montar. Sin elementos, some(...) nunca halla diferencias en renders posteriores.
[x]Al montar y cada vez que x cambie según Object.is.

El efecto del Counter en acción

El componente Counter usa el tercer caso: depende de [counter] y devuelve una función de limpieza.

src/Examples/components/Counter.tsx
window.React.useEffect(() => {
  console.log('Counter useEffect ejecutado exitosamente')

  return (() => {
     console.log('xx Counter useEffect cleanup ejecutado exitosamente xx')
  })
}, [counter])

En el primer montaje verás en consola Counter useEffect ejecutado exitosamente. Al pulsar el botón, setCounter incrementa el estado; como counter cambia, hasChanged vale true, y la función encolada primero imprime xx Counter useEffect cleanup ejecutado exitosamente xx (limpiando el render anterior) y luego Counter useEffect ejecutado exitosamente otra vez. Esa secuencia —cleanup, después efecto— es la evidencia observable del contrato que acabamos de recorrer.

Nota

wipEffects es una variable global, no una propiedad del fiber. Con un único componente con estado esto funciona, pero es una limitación conocida: con varios componentes montados los efectos se mezclan en la misma cola. Lo retomamos en Diferencias con React y bugs.
