Roadmap y ejercicios

Este motor es un punto de partida, no un destino. Cerramos la serie trazando las siguientes capacidades a implementar, explicando cómo ejecutar el proyecto, y proponiendo ejercicios que convierten la lectura pasiva en comprensión activa.

Buena parte del roadmap ya está hecha

La reescritura v2 implementó reconciliación por key, doble buffer, bailout, memo, batching, Context, Fragment, Error Boundaries y los hooks useReducer, useRef, useMemo, useCallback y useContext. Todo se documenta en los capítulos 21–36. Lo que sigue son extensiones pendientes.

Roadmap

Las siguientes mejoras cierran las brechas señaladas en Diferencias con React y bugs. Están ordenadas de mayor a menor impacto sobre la fidelidad del motor.

  1. Reconciliación por key. Como indica el Readme.md: construir un mapa de old fibers indexados por su key, reconciliar primero por clave y solo caer a la comparación por tipo y posición cuando no haya key. Esto permite detectar reordenamientos de listas sin recrear nodos, y es la carencia más visible del reconcileChildren actual.
  2. Hooks de memoización: useMemo, useCallback y memo. useCallback cachea la definición de una función entre renders: en el render inicial devuelve la función pasada, y en los siguientes devuelve la misma referencia mientras sus dependencias no cambien (comparadas con Object.is). Es útil sobre todo al pasar esa función como prop a un componente envuelto en memo, para que memo pueda saltarse el re-render al ver la misma referencia. useMemo haría lo análogo con valores calculados. La infraestructura ya existe: el patrón de oldHook + deps + Object.is de useEffect se reutiliza casi tal cual.
  3. React.Fragment. Hoy aparece referenciado en la declaración de tipos (Window.React.Fragment?) pero no está implementado. Permitiría agrupar varios hijos sin introducir un nodo DOM contenedor.
  4. Batching de updates. Actualmente cada setState reprograma un wipRoot completo. Agrupar varias actualizaciones disparadas en el mismo tick evitaría renders redundantes.
  5. Mover el estado global al fiber. Variables como wipEffects (y en parte hookIndex) son globales. Llevar los efectos a una lista por fiber —tal como los propios comentarios TODO de src/React.ts sugieren— eliminaría el riesgo de que efectos de distintos componentes se mezclen en una misma cola.

Cómo ejecutar el proyecto

Los scripts reales están en package.json. El flujo es transpilar con Babel y servir la carpeta de salida:

terminal
# Transpila src/ (JSX/TSX) hacia public/
npm run build

# Equivale a: npm run build && npx serve public
npm start

Internamente, build ejecuta babel src --out-dir public --extensions ".js,.jsx,.ts,.tsx", y start encadena ese build con npx serve public. Una vez servido, se abre index.html en el navegador; ese archivo carga ./App.js (recuerda el Bug 4: la mayúscula importa). Con las herramientas de desarrollador abiertas verás en consola la traza de Counter descrita en el Ciclo de vida completo.

Nota

No hay bundler ni servidor de desarrollo con recarga en caliente: cada cambio en src/ exige volver a ejecutar npm run build. Es una fricción intencional que mantiene el tooling mínimo y transparente.

Ejercicios de comprensión

Estas cuatro preguntas provienen del documento Docs/Optimization Problems/Optimizations.md. Respóndelas apoyándote en lo que has leído sobre hooks y reconciliación:

  1. ¿Qué ocurre cuando actualizas un estado dentro de useEffect? (Pista: piensa en cómo setState reprograma wipRoot y qué pasa con hasChanged en el siguiente render.)
  2. ¿Qué problemas evitas usando useMemo?
  3. ¿Qué problemas evitas usando useCallback?
  4. ¿Qué problemas evitas usando memo?

Y estos retos de implementación llevan el motor un paso más allá:

  1. Implementar useRef. Añade un hook que devuelva un objeto { current } estable entre renders, copiándolo desde el oldHook igual que hace useState con su estado.
  2. Reconciliación por key. Modifica reconcileChildren para indexar los old fibers por key y verifica, con una lista reordenada, que los nodos DOM se reutilizan en lugar de recrearse.
  3. Contador con reset. Extiende Counter con un segundo botón que llame a setCounter(() => 0). Observa en consola la secuencia cleanup → efecto al resetear.
  4. Efecto de montaje único. Agrega a un componente un useEffect(fn, []) y confirma, con un console.log, que su cuerpo solo corre una vez y su cleanup solo al desmontar el componente (fuérzalo con un {condicion && <Componente/>}, que ahora funciona gracias al Bug 1 corregido).
Idea clave

Si un ejercicio no funciona sin memoización, el objetivo no es añadir useCallback a ciegas: es entender el problema de fondo primero. La memoización es una optimización, no una corrección de lógica.

Lecturas recomendadas

Para profundizar, dos referencias son especialmente afines a este proyecto. La serie "Build your own React" (conocida como Didact, de Rodrigo Pombo) construye paso a paso un motor con la misma arquitectura de fibers, workLoop y reconciliación que aquí estudiamos; buena parte de la terminología del código proviene de ella. La documentación oficial de React es la referencia canónica para el comportamiento esperado de useState, useEffect, useMemo, useCallback y memo: contrastar este clon con esa especificación es la mejor forma de calibrar qué se ha simplificado.

Con esto cierra la serie. Has recorrido el motor desde el pragma de JSX hasta el ciclo de vida completo, sus diferencias con React y su hoja de ruta. El siguiente paso ya no es leer: es abrir src/React.ts y extenderlo.
