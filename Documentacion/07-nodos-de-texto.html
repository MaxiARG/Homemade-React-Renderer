Cuando un hijo es texto

Nodos de texto (TEXT_ONLY)

No todo hijo es una etiqueta. Cuando un hijo es una cadena o un número, el motor lo envuelve en un elemento sintético de tipo TEXT_ONLY para que el árbol conserve una sola forma: todo es un Element.

El envoltorio: createTextElement

Recordemos el .map de createElement visto en el capítulo anterior: si un hijo no es un objeto, se pasa por createTextElement. Esta es la función real:

src/React.ts
function createTextElement(text: string): Element {
  return {
    type: 'TEXT_ONLY',// Elements are of type Object or TEXT_ONLY
    props: {
      nodeValue: text,
      children: [], // Text elements don't have children
    },
  };
}

Tres decisiones definen este objeto:

  1. Un type centinela. En lugar de un nombre de etiqueta real, usa el string especial 'TEXT_ONLY'. Ese valor no existe en el DOM: es una marca interna que el motor reconocerá más tarde para tratar el nodo de manera distinta.
  2. El contenido en props.nodeValue. El texto recibido se guarda bajo la clave nodeValue, el mismo nombre que usa la API del DOM para el contenido de un Text. No es casualidad: facilita volcarlo directamente al nodo real.
  3. children vacío. Un nodo de texto es una hoja del árbol: nunca tiene hijos, por lo que children se inicializa como un array vacío []. Esto mantiene la forma { type, props: { … children } } idéntica a la de cualquier otro elemento.

Cómo lo trata createDom

La marca TEXT_ONLY cobra sentido cuando el motor materializa el elemento en un nodo real del DOM. Esto sucede en createDom:

src/React.ts
function createDom(fiber) {
  const dom =
    fiber.type == "TEXT_ONLY"
      ? document.createTextNode("")
      : document.createElement(fiber.type)

  updateDom(dom, {}, fiber.props)

  return dom
}

La bifurcación es directa: si el type del fiber es "TEXT_ONLY", se crea un document.createTextNode("") —un nodo de texto vacío—; en cualquier otro caso se crea un elemento host con document.createElement(fiber.type). Acto seguido, updateDom(dom, {}, fiber.props) aplica las props. Para un nodo de texto, eso significa copiar nodeValue desde las props al nodo, porque nodeValue pasa el filtro isProperty de updateDom (no empieza por "on" ni es children) y se asigna con dom["nodeValue"] = …. Así el texto aparece en pantalla.

¿Por qué envolver el texto?

La razón es la uniformidad del árbol. El motor recorre, reconcilia y confirma miles de nodos con un único conjunto de funciones —reconcileChildren, performUnitOfWork, commitWork—. Todas ellas asumen que cada nodo es un Element con type y props.children. Si el texto se representara como un string "pelado" dentro de la lista de hijos, cada una de esas funciones tendría que preguntar en cada paso "¿esto es texto o es un elemento?" y ramificar su lógica. Al envolverlo, el texto se convierte en un ciudadano de primera clase del árbol: tiene type, tiene props, tiene children (vacío) y participa en la reconciliación por posición como cualquier otro hijo. La única rama especial que sobrevive es la de createDom, que decide entre createTextNode y createElement.

Idea clave

Todo hijo del árbol es un Element. El texto no es una excepción: es un Element con type: 'TEXT_ONLY'. Esta homogeneidad es lo que permite que el reconciliador y el commit traten texto y etiquetas con el mismo código, salvo la única bifurcación de createDom.

Un ejemplo real: "Counter: {counter}"

El componente Counter ilustra la mezcla de texto estático y dinámico junto a elementos host. Su return es:

src/Examples/components/Counter.tsx
return (
  <div>
    <button onClick={() => {
      setCounter(c => c + 1);
    }}>
      Click me
    </button>
    Counter: {counter}
  </div>
);

Dentro del <div> conviven dos hijos: el <button> (un nodo host) y, como hermano suyo, el texto Counter: {counter}. Ese texto no es un único string: la interpolación {counter} lo parte, de modo que el compilador entrega varios hijos —el string literal "Counter: " y el valor numérico de counter—, cada uno de los cuales createElement convierte en su propio elemento TEXT_ONLY. El resultado es un árbol donde el <div> tiene como hijos: un nodo host button y uno o más nodos TEXT_ONLY hermanos.

div (host) ├─ button (host) <- document.createElement("button") │ └─ TEXT_ONLY "Click me" ├─ TEXT_ONLY "Counter: " <- document.createTextNode("") └─ TEXT_ONLY <valor de counter>
Nota

React real usa una idea equivalente: internamente representa los nodos de texto de forma uniforme y los materializa como Text del DOM. La diferencia principal es que React distingue además fragmentos, portales y otros tipos; este motor se limita al par "objeto o TEXT_ONLY", que es todo lo que necesita para renderizar los ejemplos.

Con el texto ya integrado al árbol como un elemento más, estamos listos para pasar de la descripción estática (elementos) a la unidad de trabajo del motor: el fiber.
