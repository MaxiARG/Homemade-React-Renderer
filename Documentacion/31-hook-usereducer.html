Estado con reducer

useReducer — ciclo de vida completo

useReducer es el hook de estado más general del motor: mantiene un valor y lo actualiza aplicando un reducer a las acciones encoladas. De hecho, useState es un caso particular construido sobre esta misma mecánica de cola persistente y dispatch estable.

Firma y para qué sirve

La firma es useReducer<S, A>(reducer, initialArg, init?): [S, Dispatch<A>]. Recibe una función pura reducer(state, action) => nuevoState, un argumento inicial y un init opcional para inicialización perezosa. Devuelve el par [estado, dispatch], donde dispatch(action) encola una acción que el reducer procesará en el próximo render.

src/react/hooks.ts
export function useReducer<S, A>(
  reducer: Reducer<S, A>,
  initialArg: S,
  init?: (arg: S) => S,
): [S, Dispatch<A>] {
  const { fiber, old } = slot();
  const hook =
    (old as ReducerHook<S, A> | undefined) ??
    ({
      tag: 'reducer',
      state: init ? init(initialArg) : initialArg,
      queue: [],
      reducer,
      dispatch: null as unknown as Dispatch<A>,
    } as ReducerHook<S, A>);
  // ...
}

Un uso mínimo: const [n, dispatch] = useReducer((s, a) => a === 'inc' ? s + 1 : s, 0), y luego dispatch('inc').

Dónde vive el estado

El estado vive en un ReducerHook, otra rama de la unión Hook. Su etiqueta es tag: 'reducer' y, además del state, la queue y el dispatch, guarda una referencia al reducer actual.

src/react/types.ts
export interface ReducerHook<S = unknown, A = unknown> {
  tag: 'reducer';
  state: S;
  queue: A[];
  reducer: Reducer<S, A>;
  dispatch: Dispatch<A>;
}

A diferencia del StateHook, cuya queue guarda funciones prev → next, aquí la queue es un array de acciones (A[]): el reducer es quien las interpreta. El objeto persiste entre renders reutilizándose desde fiber.alternate.hooks[hookIndex] vía slot(), exactamente igual que los demás hooks.

Primer render (mount)

  1. updateFunctionComponent fija R.wipFiber, pone R.hookIndex = 0 e inicializa fiber.hooks = [].
  2. slot() devuelve old = undefined, así que se crea un ReducerHook nuevo.
  3. El estado inicial usa init si se pasó: init ? init(initialArg) : initialArg. Ese init opcional evita recalcular un valor caro en cada render — solo corre en el mount.
  4. Se guarda el reducer recibido y la queue arranca vacía.
  5. commitHook(fiber, hook) empuja el hook e incrementa R.hookIndex++.

Renders siguientes (update)

En un re-render, slot() recupera el mismo ReducerHook. Dos cosas pasan antes de devolver el estado. Primero, se refresca el reducer (hook.reducer = reducer) para que use siempre la última versión de la función. Segundo, se aplica la cola de acciones: cada acción pasa por el reducer, que produce el nuevo estado, y luego la cola se vacía.

src/react/hooks.ts
hook.reducer = reducer;

// Aplicar las acciones encoladas desde el último render.
if (hook.queue.length > 0) {
  for (const action of hook.queue) {
    hook.state = reducer(hook.state, action);
  }
  hook.queue.length = 0;
}

Aplicar las acciones en orden sobre hook.state reproduce el batching: si se hicieron tres dispatch antes del render, el reducer las procesa una tras otra, cada una viendo el resultado de la anterior.

La actualización / disparador

El dispatch se crea una sola vez (if (!hook.dispatch)) para tener identidad estable. Empuja la acción cruda a la cola, apunta al fiber vigente vía fiberRef.current, marca hasPendingUpdate y llama a scheduleUpdate.

src/react/hooks.ts
if (!hook.dispatch) {
  const capturedHook = hook;
  hook.dispatch = (action: A) => {
    capturedHook.queue.push(action);
    // El fiber "vivo" es el último comprometido: lo guardamos en cada render.
    const target = fiberRef.current;
    if (target) {
      target.hasPendingUpdate = true;
      scheduleUpdate(target);
    }
  };
}
// fiberRef apunta siempre al fiber más reciente que montó este hook.
const fiberRef = (hook as unknown as { _fiberRef?: { current: Fiber } })._fiberRef ?? {
  current: fiber,
};
fiberRef.current = fiber;

La diferencia con el dispatch de useState es mínima: aquí la acción se encola tal cual (el reducer la interpretará), mientras que en useState primero se normaliza a una función actualizadora. El fiberRef mutable resuelve el mismo problema — un dispatch estable que debe apuntar al fiber más reciente del doble buffer, no al del mount.

src/react/scheduler.ts
function ensureFlush(): void {
  if (batchDepth > 0) return; // dentro de flushSync: se drena al salir
  if (flushScheduled) return;
  flushScheduled = true;
  queueMicrotask(() => {
    flushScheduled = false;
    flushAll();
  });
}

El batching por microtask lo aporta ensureFlush: dentro de flushSync (batchDepth > 0) el trabajo se pospone hasta salir, agrupando todos los dispatch en un solo render.

useState se construye sobre esto

useState es useReducer con un reducer implícito (state, action) => action(state): por eso su cola guarda funciones actualizadoras en vez de acciones. Comparten la misma maquinaria — cola persistente, dispatch estable con fiberRef y scheduleUpdate — solo cambia cómo se interpreta cada elemento de la cola. Conceptualmente, useState es azúcar sobre useReducer, tal como en React real.

Orden de ejecución

useReducer corre durante la render phase, en el cuerpo del componente dentro de updateFunctionComponent. La cola se procesa antes de devolver el estado, así que el render ya usa el valor reducido. El re-render que dispara dispatch es asíncrono (microtask) salvo bajo flushSync. Los efectos corren después del commit.

Reglas de los hooks

Como todo hook, useReducer depende de que R.hookIndex sea estable: slot() recupera su estado por posición (alternate.hooks[hookIndex]), y el índice avanza solo en commitHook, en el orden de invocación.

Cuidado

No llames useReducer condicionalmente. Si un render omite la llamada, los índices se desplazan y el hook recupera el estado de otro hook. El orden de las llamadas define la identidad de cada uno.

Diagrama de secuencia

MOUNT updateFunctionComponent: R.wipFiber = fiber, R.hookIndex = 0, fiber.hooks = [] useReducer(reducer, 0, init?) slot() -> old = undefined crea ReducerHook { tag:'reducer', state: init?init(0):0, queue:[], reducer, dispatch:null } crea dispatch estable + fiberRef.current = fiber commitHook -> hooks.push, hookIndex++ return [state, dispatch] | v DISPATCH dispatch('inc') queue.push('inc') (acción cruda, sin normalizar) target = fiberRef.current target.hasPendingUpdate = true scheduleUpdate(target) -> dirtyFibers.add, ensureFlush (microtask / flushSync) | v UPDATE (nuevo render del mismo fiber) slot() -> old = alternate.hooks[hookIndex] (MISMO ReducerHook) hook.reducer = reducer (refresca) aplica queue: state = reducer(state, 'inc') ; queue.length = 0 dispatch ya existe (identidad estable) ; fiberRef.current = fiber nuevo commitHook -> hookIndex++ return [nuevoState, dispatch] -> commit escribe el DOM
