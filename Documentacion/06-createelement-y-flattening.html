Construyendo el árbol

createElement y flattening

Cada llamada que emite el compilador desemboca en createElement. Aquí se normalizan los hijos: se aplanan listas anidadas, se descartan valores vacíos y todo se convierte en un objeto uniforme listo para el reconciliador.

La función completa

Este es el cuerpo real de createElement tal como vive en el motor:

src/React.ts
export function createElement(
  type: string,
  props: Props,
  ...children: any[]
): Element {
  return {
    type,
    props: {
      ...props,
      children: flatten(children)
        // React ignora null, undefined y booleanos como hijos (permite {cond && <X/>}).
        .filter((child) => child != null && typeof child !== 'boolean')
        .map((child) =>
          typeof child === 'object' ? child : createTextElement(child)
        ),
    },
  };
}

Aunque son pocas líneas, condensan cuatro operaciones encadenadas. Vamos una por una.

1. type y el spread de props

El objeto devuelto copia type tal cual y expande las props recibidas con ...props. Ese spread preserva todos los atributos que el compilador agrupó (por ejemplo style u onClick) y, acto seguido, sobrescribe la clave children con la versión ya procesada. El orden importa: primero se vuelcan las props originales, luego se fija children definitivo.

2. El parámetro variádico ...children

Como vimos en JSX y el pragma, cada hijo anidado llega como un argumento posicional independiente. El parámetro rest ...children los recoge en un único array. Pero ese array puede contener, a su vez, otros arrays: es lo que ocurre cuando un hijo es el resultado de un .map().

3. flatten(): aplanado recursivo

Antes de procesar los hijos, createElement los pasa por flatten, un reduce recursivo que colapsa cualquier nivel de anidamiento en un único array plano:

src/React.ts
function flatten(arr) {
  return arr.reduce((flat, item) => {
    return flat.concat(Array.isArray(item) ? flatten(item) : item);
  }, []);
}

Por cada elemento, si es un array vuelve a llamarse a sí misma; si no, lo concatena directamente. El resultado es una secuencia lineal de hijos, independientemente de cuántas listas anidadas hubiera. Un ejemplo de entrada y salida:

flatten([ [1, 2], [3] ]) reduce paso a paso: [] .concat( flatten([1,2]) ) -> [1, 2] [1, 2] .concat( flatten([3]) ) -> [1, 2, 3] salida: [1, 2, 3]

4. El .filter que descarta valores vacíos

Sobre el array ya aplanado se aplica un filtro que elimina null, undefined y booleanos antes de convertir nada a texto. Esta línea es una corrección deliberada:

Bug corregido

Hijos falsy convertidos en texto. La versión original mapeaba todo hijo no-objeto a createTextElement, incluidos null, undefined y booleanos. Eso rompía el patrón idiomático {condicion && <Componente/>}, que renderizaba el texto literal "false"; y, peor aún, un hijo intermedio nulo podía dejar un "hueco" en el array que hacía que reconcileChildren accediera a prevSibling.sibling siendo prevSibling === null, lanzando una excepción. La corrección añade .filter((child) => child != null && typeof child !== 'boolean') antes del .map(...). El capítulo 19 reúne todos los bugs.

5. El .map que uniformiza a Element

Por último, cada hijo superviviente pasa por un .map que decide su forma final: si ya es un objeto (un Element producido por otra llamada a createElement) se deja intacto; si es un valor primitivo (un número o un string) se envuelve con createTextElement. Así todo hijo, sin excepción, termina siendo un objeto Element. Ese detalle —envolver el texto— se estudia en Nodos de texto (TEXT_ONLY).

¿Por qué hace falta el flattening?

El motivo práctico es el patrón más común al renderizar colecciones: lista.map(x => <div>{x}</div>). Ese .map devuelve un array, no un elemento suelto. Como el array se coloca como un único hijo entre otros hijos, sin aplanar quedaría anidado un nivel de más. Este patrón aparece literalmente en src/App.tsx:

src/App.tsx
<Label theLabel={'Array de elementos'}/>
{
  [1,2,3].map((elem)=>(<div>{elem}</div>))
}
<HolaMundo />

Aquí createElement recibe, entre sus hijos, un elemento <Label>, luego un array de tres <div> (fruto del .map), y luego <HolaMundo>. Sin flatten, ese array intermedio rompería la linealidad de la lista de hijos y el reconciliador —que recorre los hijos como una lista enlazada de hermanos— no sabría tratarlo. Al aplanar, los tres <div> quedan como hermanos regulares junto a <Label> y <HolaMundo>, exactamente como si los hubieras escrito uno a uno.

Idea clave

El aplanado ocurre antes de enviar los nodos al reconciliador. De ese modo, cuando reconcileChildren recorre props.children, encuentra siempre una lista plana de objetos Element: sin arrays anidados, sin null, sin booleanos. Toda la complejidad de normalización queda encapsulada en createElement.

Nota

React real resuelve el mismo problema, pero además exige una prop key en las listas para reconciliar por identidad. Este motor no implementa key: aplana y reconcilia por posición. Es una de las limitaciones documentadas en el capítulo 12.
