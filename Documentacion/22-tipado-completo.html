Adiós al any

Tipado completo con TypeScript

El refactor no solo reorganizó el motor en módulos: también lo tipó de punta a punta. src/react/types.ts describe cada estructura del núcleo —elementos, fibers, hooks, contextos y el estado por-root— de modo que el compilador atrape errores que antes solo aparecían en tiempo de ejecución. Este capítulo recorre esos tipos y muestra cómo la unión discriminada de Hook permite razonar sobre cada hook con seguridad.

De dónde se parte: el any como deuda

Una implementación temprana de un motor así abusa del any: los props son any, el fiber es any, los hooks son un array heterogéneo sin forma común. Funciona, pero cada acceso a un campo es una apuesta. El objetivo del tipado es convertir esas apuestas en garantías: si un fiber puede no tener DOM, el tipo lo dice (DomNode | null); si un hook es de estado, sus campos (queue, dispatch) existen y tienen forma conocida.

El elemento y sus tipos base

Todo empieza en ReactElement, la descripción inmutable que produce createElement. Su type es una unión cerrada: ElementType.

src/react/types.ts
/** El `type` de un elemento: etiqueta host, componente función, clase o Fragment. */
export type ElementType =
  | string
  | FunctionComponent<any>
  | ComponentClass<any>
  | FragmentType;

/** Descripción inmutable de UI producida por `createElement`. */
export interface ReactElement {
  type: ElementType;
  props: Props;
  key: Key;
}

FunctionComponent<P> es una función que recibe props y devuelve ReactElement | null (nótese el null: habilita el early-return de un componente). ComponentClass describe el constructor —con su getDerivedStateFromError estático— y ComponentInstance la instancia viva, ambos usados por los Error Boundaries. Key es string | number | null, la clave de reconciliación que veremos en Reconciliación por key.

El Fiber: cada campo declarado

El Fiber es la estructura más grande y la que más se benefició del tipado. Antes, campos como alternate, hooks o stateNode eran any; ahora cada uno declara exactamente qué puede contener y cuándo puede ser null.

src/react/types.ts
export interface Fiber {
  type: ElementType | 'ROOT';
  props: Props;
  key: Key;

  dom: DomNode | null;              // solo fibers host y el root

  parent: Fiber | null;
  child: Fiber | null;
  sibling: Fiber | null;

  alternate: Fiber | null;          // el fiber del render anterior (doble buffer)
  effectTag: EffectTag;             // 'PLACEMENT' | 'UPDATE' | 'DELETION' | 'NONE'
  index: number;                    // posición entre hermanos (movimientos con keys)
  moved: boolean;                   // ¿reusado pero reubicado en el DOM?

  hooks: Hook[] | null;             // hooks del componente función, en orden
  stateNode: ComponentInstance | null;  // instancia de clase, si aplica
  hasPendingUpdate: boolean;        // ¿setState pendiente? (para el bailout)

  providerContext: ReactContext<unknown> | null;
  providerForced: boolean;
  ctxDepth: number;                 // pila de contextos al empezar (error boundaries)
  forceDepth: number;

  root: RootStateContainer | null;  // raíz a la que pertenece
}

Los tres punteros parent/child/sibling tipados como Fiber | null hacen que el recorrido del árbol (en performUnitOfWork) sea a prueba de accesos a null: el compilador exige el chequeo antes de bajar por node.child. Y root: RootStateContainer | null es el puntero que —como vimos en el capítulo anterior— conecta cada fiber con el estado de su root para agendar re-renders.

La unión discriminada de Hook

Aquí está la pieza más elegante del tipado. Los siete hooks del motor guardan datos muy distintos: useState necesita una cola de actualizaciones, useEffect un cleanup, useRef un objeto { current }. En vez de un tipo laxo con todo opcional, cada variante es una interfaz propia con un campo literal tag que la identifica. La unión de todas es Hook.

src/react/types.ts
export interface StateHook<S = unknown> {
  tag: 'state';
  state: S;
  queue: Array<(prev: S) => S>;
  dispatch: Dispatch<SetStateAction<S>>;
}

export interface EffectHook {
  tag: 'effect';
  effect: () => void | (() => void);
  deps: unknown[] | undefined;
  cleanup: (() => void) | undefined;
  hasChanged: boolean;
}

export interface RefHook<T = unknown> {
  tag: 'ref';
  ref: { current: T };
}

export type Hook =
  | StateHook
  | ReducerHook
  | EffectHook
  | RefHook
  | MemoHook
  | ContextHook;

Las variantes restantes siguen el mismo patrón: ReducerHook añade reducer y una queue de acciones; MemoHook guarda value y deps (lo usa tanto useMemo como useCallback); ContextHook guarda el context y su value leído.

Cómo el tag discrimina en runtime

El literal tag cumple doble función. En compilación, TypeScript estrecha el tipo: dentro de un if (hook.tag === 'effect') el compilador sabe que hook es un EffectHook y por tanto expone effect, cleanup y hasChanged. En ejecución, ese mismo chequeo filtra el hook correcto de un array heterogéneo. Dos lugares del reconciliador lo usan exactamente así.

src/react/reconciler.ts
/** Recolecta los efectos pasivos (useEffect) que cambiaron, para el commit. */
function collectPassiveEffects(fiber: Fiber): void {
  if (!fiber.hooks) return;
  const root = fiber.root;
  if (!root) return;
  for (const hook of fiber.hooks) {
    if (hook.tag === 'effect' && hook.hasChanged) {
      const eff = hook as EffectHook;
      root.pendingEffects.push(() => {
        if (typeof eff.cleanup === 'function') eff.cleanup();
        const cleanup = eff.effect();
        eff.cleanup = typeof cleanup === 'function' ? cleanup : undefined;
      });
    }
  }
}

Y en la fase de borrado, runCleanups recorre el mismo array de hooks mezclados y ejecuta solo los cleanups de los efectos, filtrando otra vez por el tag:

src/react/reconciler.ts
function runCleanups(fiber: Fiber): void {
  if (fiber.hooks) {
    for (const hook of fiber.hooks) {
      if (hook.tag === 'effect' && typeof hook.cleanup === 'function') {
        hook.cleanup();
      }
    }
  }
  // ... recorre recursivamente los hijos
}

Sin la unión discriminada, ese hook.cleanup obligaría a un any o a un cast inseguro. Con ella, el tag es la única fuente de verdad y el acceso queda validado por el compilador.

HooktagCampos característicos
useState'state'state, queue, dispatch
useReducer'reducer'state, queue, reducer, dispatch
useEffect'effect'effect, deps, cleanup, hasChanged
useRef'ref'ref: { current }
useMemo / useCallback'memo'value, deps
useContext'context'context, value

Context y el estado por-root, también tipados

ReactContext<T> parametriza el valor del contexto: su _currentValue, _defaultValue y el Provider comparten el mismo T, de modo que useContext(ctx) devuelve el tipo correcto sin casts. Y RootStateContainer —visto en el capítulo anterior— tipa cada campo del estado por-root: currentRoot/wipRoot como Fiber | null, pendingEffects como Array<() => void>, dirtyFibers como Set<Fiber>. También se tipa MemoComponent, el envoltorio de memo, con su marca $$typeof: 'react.memo' que el reconciliador usa para reconocerlo.

Dónde el any es inevitable

El propio encabezado de types.ts lo admite: "no se usa any salvo en fronteras inevitables". Esas fronteras son dos. Primera, las props arbitrarias del usuario: Props se declara como { [key: string]: unknown } —y ElementType acepta FunctionComponent<any>— porque el motor no puede conocer de antemano la forma de los props de un componente cualquiera. Segunda, el mapa de estilos: StyleObject = Record<string, string | number>, un diccionario abierto que refleja la naturaleza dinámica del objeto style del DOM. En ambos casos el any/unknown vive en el borde del sistema, no en su núcleo: apenas cruza hacia adentro, se estrecha con guardas de tipo.

Borde del sistema (any/unknown) Núcleo (totalmente tipado) ───────────────────────────── ────────────────────────────── props del usuario ─┐ Props[key]: unknown │ guardas de tipo Fiber, Hook (unión por tag), StyleObject ├───────────────────► ReactElement, ReactContext<T>, FunctionComponent │ (hook.tag ===...) RootStateContainer <any> ┘ → el compilador garantiza los accesos

El resultado es un motor donde el compilador es un colaborador: refactorizar el reconciliador o agregar un hook nuevo obliga a declarar su tag y su forma, y cualquier acceso incoherente se marca antes de ejecutar una sola línea. En el próximo capítulo aprovecharemos justamente el campo key de ReactElement y Fiber para reconciliar listas por clave.
