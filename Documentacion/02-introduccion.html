Punto de partida

Introducción y objetivos

Este proyecto es un estudio del funcionamiento interno de React: una reimplementación mínima y legible de su motor de renderizado, construida para entenderlo, no para reemplazarlo.

¿Qué es este proyecto?

Homemade React Renderer es un clon académico del núcleo de React escrito en TypeScript. Su propósito no es competir con la librería oficial, sino exponer, en unas pocas centenas de líneas, los mecanismos que React esconde tras su API pública. Como dice el propio Readme.md del repositorio, es «a study of how React works internally, inspired by its architecture and execution model». Todo el motor vive en un único archivo, src/React.ts, que sirve como fuente de verdad a lo largo de toda esta documentación.

El trabajo está inspirado en la serie «Build your own React» y su proyecto didáctico Didact, de Rodrigo Pombo. De ahí provienen la nomenclatura (wipRoot, nextUnitOfWork, reconcileChildren) y la estrategia general: reconstruir el modelo Fiber y el scheduler cooperativo desde cero, paso a paso, para que cada decisión de diseño quede a la vista.

Filosofía

La premisa es que la mejor forma de comprender una abstracción es reconstruirla. React oculta deliberadamente su maquinaria; aquí la exponemos. El código prioriza la claridad sobre el rendimiento: no hay optimizaciones que compliquen la lectura, y muchos comentarios del fuente explican el porqué de cada línea. El objetivo es que quien lea src/React.ts pueda seguir un render completo con el dedo, desde el JSX hasta la mutación final del DOM.

Nota

Cada capítulo enlaza el concepto abstracto de React real con una porción concreta y citada del código del proyecto. Cuando algo diverge del React oficial, se marca de forma explícita (véase 19 · Diferencias con React y bugs).

¿Qué implementa?

A pesar de su tamaño, el motor cubre el camino completo de un render y varias piezas del sistema de hooks:

  • createElement: convierte los nodos JSX en objetos Element planos, con type y props.
  • Árbol de fibers: cada elemento se transforma en una unidad de trabajo enlazada por child, sibling y parent.
  • Scheduler cooperativo: el workLoop troceado sobre requestIdleCallback que cede el control al navegador para no bloquear la interfaz.
  • Reconciliación (diffing): reconcileChildren compara el árbol nuevo con el alternate anterior y asigna effectTag (PLACEMENT, UPDATE, DELETION).
  • Commit al DOM: commitRoot y commitWork aplican de una sola vez los cambios calculados.
  • Hooks: useState (con cola de acciones y re-render) y useEffect (con dependencias y cleanup).
  • Estilos inline: soporte tanto para objetos de estilo como para cadenas, gestionado en updateDom.
  • Flattening de children: aplanado de arrays anidados para soportar patrones como {lista.map(...)}.

Todas estas piezas se apoyan en un pequeño conjunto de variables globales mutables que actúan como estado del renderer:

src/React.ts
let wipFiber = null
let hookIndex = null
let nextUnitOfWork = null
let wipRoot = null
let currentRoot = null
let deletions = null;
let wipEffects = []; // Global para recolectar efectos

Alcance y no-objetivos

Es fundamental fijar los límites. Este renderer no está pensado para producción. Reconstruye el árbol completo en cada actualización, no implementa reconciliación por key, mantiene los efectos en una lista global (wipEffects) compartida entre componentes, y referencia React.Fragment en la configuración sin llegar a implementarlo. No hay batching de actualizaciones, ni prioridades, ni portales, ni contexto, ni Suspense. Estas ausencias no son descuidos: son la frontera deliberada entre «entender React» y «reescribir React».

Cuidado

No utilices este motor en una aplicación real. Su valor es pedagógico: cada atajo que toma es una oportunidad para razonar sobre qué hace React de verdad y por qué. El capítulo 20 · Roadmap y ejercicios propone extensiones para quien quiera acercarlo más al original.

Público objetivo

Esta documentación se dirige a personas que ya usan React y quieren mirar debajo del capó: estudiantes de ingeniería de software, desarrolladores que preparan entrevistas técnicas sobre el modelo Fiber, y curiosos que se preguntan qué ocurre entre setState y el repintado de la pantalla. Se asume familiaridad con JavaScript, JSX y el concepto general de componentes, pero no con las interioridades de React.

¿Qué preguntas responde esta documentación?

A lo largo de los veinte capítulos encontrarás respuesta a interrogantes como: ¿cómo se convierte una etiqueta JSX en una llamada a función? ¿qué es exactamente un fiber y cómo se recorre? ¿cómo puede React interrumpir y reanudar un render sin dejar la interfaz a medias? ¿cómo «recuerda» useState su valor entre renders si es solo una función? ¿por qué los efectos se ejecutan después de pintar? y ¿en qué se diferencia todo esto del React real? Empezamos en el siguiente capítulo con la vista de pájaro de la arquitectura general.
