De monolito a módulos

Refactor: módulos y estado por-root

El motor pasó de un único archivo con variables globales de módulo a una carpeta src/react/ donde cada pieza tiene una responsabilidad clara y el estado volátil vive encapsulado por root. Este capítulo explica qué se movió, por qué, y cómo ese cambio habilita montar varios roots independientes en la misma página.

El punto de partida: un monolito con estado global

Las primeras versiones (documentadas hasta el capítulo 20) concentraban todo en src/React.ts. El ciclo de render se coordinaba con variables a nivel de módulo: wipRoot (árbol en construcción), currentRoot (último árbol comprometido), nextUnitOfWork (la próxima unidad de trabajo), deletions (fibers a borrar), wipEffects (efectos pendientes), más el estado efímero del render en curso: wipFiber y hookIndex.

Ese diseño funciona para un árbol. El problema aparece en cuanto querés montar dos aplicaciones en la misma página: como todas esas variables son únicas y compartidas por el módulo, un segundo render() pisaría el wipRoot del primero. El estado del render de una app se filtra al de la otra. La reconciliación deja de ser aislada.

El problema de fondo

Una variable global de módulo es, en la práctica, un singleton implícito. Sirve mientras exista exactamente una instancia. En el momento en que la abstracción necesita existir "N veces" (N roots), ese estado tiene que dejar de ser global y pasar a ser un campo de un objeto que se pueda instanciar tantas veces como haga falta.

La nueva estructura de módulos

El motor se dividió en once módulos dentro de src/react/. Cada uno importa de los demás por su nombre, sin dependencias circulares problemáticas, y index.ts reexpone todo como el objeto público React.

MóduloResponsabilidad
types.tsTodos los tipos del núcleo: ReactElement, Fiber, la unión Hook, ReactContext y el RootStateContainer. Sin lógica.

Dos clases de estado: por-root vs. render-scoped

El refactor separa deliberadamente dos naturalezas de estado que antes convivían como variables sueltas.

(a) Estado por-root: RootStateContainer

Todo lo que persiste entre renders y pertenece a un árbol concreto se agrupó en una interfaz en types.ts. Cada root tiene su propia instancia:

src/react/types.ts
export interface RootStateContainer {
  container: DomNode;
  /** Árbol comprometido en el último commit. */
  currentRoot: Fiber | null;
  /** Árbol en construcción. */
  wipRoot: Fiber | null;
  nextUnitOfWork: Fiber | null;
  deletions: Fiber[];
  /** Efectos pasivos recolectados en el render en curso, se ejecutan tras el commit. */
  pendingEffects: Array<() => void>;
  /** Fibers marcados como sucios por setState (para saber desde dónde re-renderizar). */
  dirtyFibers: Set<Fiber>;
}

Comparación directa con el pasado: wipRoot, currentRoot, nextUnitOfWork y deletions eran globales; ahora son campos de este contenedor. wipEffects se convirtió en pendingEffects (una lista de efectos de este root, resolviendo el riesgo que señalaba el roadmap de mezclar efectos entre componentes). Y dirtyFibers es nuevo: registra qué fibers ensuciaron un setState, base del bailout.

(b) Estado render-scoped: internals.ts y el objeto R

En cambio, hay estado que es intrínsecamente efímero: solo tiene sentido mientras se ejecuta el cuerpo de un componente. Ese estado sí puede ser un singleton, porque solo un fiber se renderiza a la vez. Vive en R:

src/react/internals.ts
export const R: RenderState = {
  wipFiber: null,       // fiber cuyo cuerpo se está ejecutando AHORA
  hookIndex: 0,         // próximo hook a leer/crear en wipFiber.hooks
  activeRoot: null,     // root cuyo árbol se está construyendo
  forceRenderDepth: 0,  // > 0 dentro de un Provider cuyo value cambió
};

/** Pila de valores de contexto (push al entrar a un Provider, pop al salir). */
export const contextStack: Array<{ context: ReactContext<unknown>; value: unknown }> = [];
Por qué esta división

R.wipFiber y R.hookIndex son el equivalente al currentlyRenderingFiber de React: apuntan al componente que se está ejecutando en este instante. No hace falta una copia por root porque el render es secuencial. En cambio wipRoot o currentRoot describen un árbol que sobrevive entre renders, así que sí necesitan una instancia por root. La regla: lo que sobrevive al render se encapsula por root; lo que solo existe durante el render puede ser singleton.

createRoot: un ciclo de vida independiente

createRoot(container) crea un RootStateContainer fresco y devuelve un objeto con render() y unmount(). Nada de esto toca variables globales: todo el estado del árbol vive en la clausura de ese root.

src/react/root.ts
export function createRoot(container: DomNode): ReactRoot {
  const state = createRootState(container);
  return {
    render(element: ReactElement): void {
      // El render inicial se drena de forma síncrona (resultado determinista).
      flushSync(() => {
        updateContainer(state, element);
        enqueueRoot(state);
      });
    },
    unmount(): void {
      if (state.container instanceof HTMLElement) state.container.innerHTML = '';
      state.currentRoot = null;
      state.wipRoot = null;
      state.nextUnitOfWork = null;
    },
    _internal: state,
  };
}

updateContainer es quien decide entre montaje inicial y re-render, y prepara el work-in-progress de este root sin tocar ningún otro:

src/react/root.ts
function updateContainer(root: RootStateContainer, element: ReactElement): void {
  if (root.currentRoot) {
    // Re-render del mismo root: doble buffer sobre el árbol comprometido.
    root.wipRoot = createWorkInProgress(root.currentRoot, { children: [element] });
  } else {
    // Primer montaje.
    root.wipRoot = createHostRootFiber(root.container, { children: [element] });
  }
  root.wipRoot.dom = root.container;
  root.wipRoot.root = root;
  root.deletions = [];
  root.pendingEffects = [];
  contextStack.length = 0;
  R.forceRenderDepth = 0;
  root.nextUnitOfWork = root.wipRoot;
}

Fijate en el detalle clave: root.wipRoot.root = root. Cada fiber guarda un puntero a su RootStateContainer (campo root en el Fiber). Así, cuando un setState dispara scheduleUpdate(fiber), el scheduler recupera fiber.root y agenda solo ese root, sin ambigüedad sobre a qué árbol pertenece la actualización.

El scheduler orquesta múltiples roots

El scheduler mantiene un Set<RootStateContainer> de roots pendientes. Un microtask drena todos los que tengan trabajo, cada uno con su propio workLoopSync y commitRoot. R.activeRoot apunta, durante el drenado, al root que se está procesando en ese momento.

React.createRoot(#app1) ──► RootStateContainer A ─┐ │ React.createRoot(#app2) ──► RootStateContainer B ─┤ ▼ pendingRoots: Set{ A, B } │ (microtask / flushSync) ▼ for root of roots: R.activeRoot = root workLoopSync(root) ── usa root.nextUnitOfWork commitRoot(root) ── usa root.wipRoot / deletions root.dirtyFibers.clear() A.currentRoot ─── árbol de #app1 B.currentRoot ─── árbol de #app2 (aislados: ningún campo se comparte entre A y B)

Montar varios roots sin interferencia

La consecuencia práctica del encapsulamiento es que este código funciona sin que una app pise a la otra: cada createRoot mantiene su propio currentRoot, su propia cola de deletions y sus propios pendingEffects.

ejemplo de uso
const app1 = React.createRoot(document.getElementById('app1'));
const app2 = React.createRoot(document.getElementById('app2'));

app1.render(React.createElement(Contador, null));
app2.render(React.createElement(Reloj, null));

// Un setState dentro de Contador agenda SOLO el root de app1;
// el árbol de app2 no se recalcula ni se re-commitea.
app1.unmount(); // limpia app1 sin tocar app2
Por qué encapsular habilita múltiples roots

Mientras el estado del árbol vivía en variables de módulo, "el motor" era un objeto único: solo podía existir un wipRoot a la vez, así que dos apps competían por la misma memoria. Al mover ese estado a RootStateContainer y darle a cada fiber un puntero fiber.root, cada árbol se vuelve autocontenido. El scheduler los recorre en un Set y el estado render-scoped (R) se reasigna al entrar a cada root. Así, N roots conviven, se actualizan de forma independiente y se desmontan sin efectos colaterales, exactamente como el createRoot de React 18.
