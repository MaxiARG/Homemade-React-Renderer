Función cacheada

useCallback — ciclo de vida completo

useCallback memoiza una función en vez de un valor de datos. En este motor no es un hook aparte: es literalmente useMemo(() => fn, deps). Su razón de existir es dar identidad estable a un callback entre renders.

Firma y para qué sirve

La firma real en src/react/hooks.ts es useCallback<T>(callback: T, deps: unknown[] | undefined): T, donde T es una función. Devuelve la misma función mientras las deps no cambien, en vez de la nueva instancia que el componente crea en cada render.

uso mínimo
function Padre({ items }) {
  const [n, setN] = useState(0);
  // `onSelect` conserva su identidad aunque `n` cambie.
  const onSelect = useCallback((id) => console.log('elegido', id), []);
  return h(ListaMemo, { items, onSelect });
}

Dónde vive el estado

No hay un CallbackHook en src/react/types.ts. Como useCallback se implementa sobre useMemo, el estado que persiste es un MemoHook corriente, cuyo value resulta ser la función:

src/react/types.ts
export interface MemoHook<T = unknown> {
  tag: 'memo';
  value: T;                       // aquí: la función memoizada
  deps: unknown[] | undefined;
}

Igual que cualquier hook, ese objeto se reutiliza entre renders leyéndolo desde wipFiber.alternate.hooks[hookIndex], de modo que la referencia guardada en value sobrevive de un render al siguiente.

Renders siguientes (update): es useMemo por dentro

La implementación completa cabe en una línea, y es la clave de todo el capítulo:

src/react/hooks.ts
export function useCallback<T extends (...args: never[]) => unknown>(
  callback: T,
  deps: unknown[] | undefined,
): T {
  return useMemo(() => callback, deps);
}

En vez de pasarle a useMemo una fábrica que calcula un valor, se le pasa una fábrica trivial () => callback que simplemente devuelve la función. Todo lo demás lo hereda de useMemo:

src/react/hooks.ts (useMemo, para recordar)
let hook: MemoHook<T>;
if (oldMemo && deps && depsEqual(deps, oldMemo.deps)) {
  hook = oldMemo; // deps iguales: reusar el valor cacheado (= la función anterior)
} else {
  hook = { tag: 'memo', value: factory(), deps }; // factory() devuelve el callback nuevo
}
  • Si depsEqual es true, hook = oldMemo: se devuelve la misma referencia de función del render anterior. El callback nuevo que recibió esta llamada se descarta.
  • Si alguna dep cambió, factory() devuelve el callback recién creado y esa pasa a ser la nueva identidad estable hasta el próximo cambio de deps.
Sutileza: la clausura sí es nueva

El componente siempre crea una función nueva en cada render (es una expresión literal). Lo que useCallback hace no es evitar esa creación, sino ignorarla y seguir devolviendo la vieja mientras las deps coincidan. Por eso, si el callback lee variables que quedaron fuera de deps, va a capturar valores viejos (stale closure): la identidad estable tiene ese costo.

La actualización / disparador

Como useMemo, useCallback no tiene disparador: no crea dispatch, no encola en ninguna queue ni llama a scheduleUpdate. Lo que lo hace "recalcular" (devolver la función nueva) es un re-render provocado por otro hook combinado con un cambio en Object.is de alguna dependencia.

Orden de ejecución y por qué sirve con memo

La evaluación ocurre en línea durante el render, en la posición donde se llama. Su utilidad real aparece al combinarlo con memo. Un componente envuelto en memo hace bailout comparando props con shallowEqualProps (Object.is por clave). Si el padre le pasa una función literal nueva en cada render, esa prop cambia de identidad y el bailout nunca ocurre. Estabilizando la función con useCallback, la prop conserva su referencia y el hijo memoizado puede efectivamente saltarse el re-render.

Reglas de los hooks

Al delegar en useMemo, useCallback consume exactamente un slot de hook y avanza R.hookIndex una vez (vía commitHook). Debe llamarse incondicionalmente y en orden fijo: si lo saltearas en algún render, la posición del MemoHook se correría y el motor recuperaría el hook equivocado desde fiber.alternate.hooks.

Test que lo fija

El test "useCallback devuelve la misma función si deps no cambian" monta un componente con const cb = useCallback(() => {}, []) y guarda cada cb en un arreglo. Tras un setB(1) que fuerza el re-render, cbs[0] y cbs[1] son toBe (la misma referencia): las deps [] nunca cambian, así que depsEqual siempre da true y se reusa la función original.

Diagrama de secuencia

useCallback(cb, deps) ≡ useMemo(() => cb, deps) MOUNT factory = () => cbA old = undefined value = factory() = cbA deps = [] ──► devuelve cbA UPDATE por setB(1) ── el componente crea cbB (función nueva, literal) useCallback(cbB, []) → useMemo(() => cbB, []) depsEqual([], []) = true hook = oldMemo ──► devuelve cbA (¡cbB se descarta!) cbs[0] === cbs[1] UPDATE con deps distintas ── depsEqual = false value = factory() = cbNueva ──► devuelve la función nueva; nueva identidad estable
