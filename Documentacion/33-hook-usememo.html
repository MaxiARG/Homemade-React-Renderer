Valor cacheado

useMemo — ciclo de vida completo

useMemo memoiza el resultado de un cálculo: guarda un valor y las dependencias con las que se produjo, y solo lo recalcula cuando esas dependencias cambian. No agenda re-renders; es puramente una optimización de trabajo durante el render.

Firma y para qué sirve

La firma real en src/react/hooks.ts es useMemo<T>(factory: () => T, deps: unknown[] | undefined): T. Recibe una función fábrica que produce el valor y un arreglo de dependencias. Devuelve el valor cacheado mientras las dependencias no cambien.

uso mínimo
function Lista({ items, filtro }) {
  // Solo re-filtra cuando `items` o `filtro` cambian de identidad.
  const visibles = useMemo(
    () => items.filter((it) => it.includes(filtro)),
    [items, filtro],
  );
  return h('ul', null, ...visibles.map((t) => h('li', null, t)));
}

Se usa para evitar recomputar resultados costosos en cada render, o para estabilizar la identidad de un objeto/arreglo que se pasa como prop a un hijo envuelto en memo (si el objeto se recrea en cada render, el hijo nunca puede hacer bailout).

Dónde vive el estado

El estado del hook es un objeto MemoHook, un miembro de la unión discriminada Hook en src/react/types.ts:

src/react/types.ts
export interface MemoHook<T = unknown> {
  tag: 'memo';
  value: T;                       // el valor cacheado
  deps: unknown[] | undefined;    // las deps con las que se produjo
}

Ese objeto persiste entre renders gracias al modelo de hooks persistentes: cada componente función guarda su arreglo hooks en el fiber, y en el próximo render el hook viejo se recupera desde wipFiber.alternate.hooks[hookIndex]. Así, el value y las deps del render anterior siguen disponibles para comparar.

Primer render (mount)

En el montaje no existe hook anterior, así que old es undefined y la fábrica se ejecuta sí o sí. Veamos el cuerpo completo:

src/react/hooks.ts
export function useMemo<T>(factory: () => T, deps: unknown[] | undefined): T {
  const { fiber, old } = slot();
  const oldMemo = old as MemoHook<T> | undefined;

  let hook: MemoHook<T>;
  if (oldMemo && deps && depsEqual(deps, oldMemo.deps)) {
    hook = oldMemo; // deps iguales: reusar el valor cacheado
  } else {
    hook = { tag: 'memo', value: factory(), deps };
  }
  commitHook(fiber, hook as unknown as Hook);
  return hook.value;
}
  1. slot() lee R.wipFiber y devuelve old = fiber.alternate?.hooks?.[R.hookIndex]; en el mount es undefined.
  2. Como oldMemo no existe, se cae al else: se ejecuta factory() y se crea el objeto { tag: 'memo', value, deps }.
  3. commitHook empuja el hook a fiber.hooks y hace R.hookIndex++.
  4. Se devuelve hook.value.

Renders siguientes (update)

En el re-render, slot() recupera el MemoHook anterior. La decisión de reusar o recomputar depende de depsEqual, que compara elemento a elemento con Object.is:

src/react/hooks.ts
function depsEqual(a: unknown[] | undefined, b: unknown[] | undefined): boolean {
  if (!a || !b) return false;
  if (a.length !== b.length) return false;
  for (let i = 0; i < a.length; i++) {
    if (!Object.is(a[i], b[i])) return false;
  }
  return true;
}

Alguna dep cambió (Object.is falso)Se crea un hook nuevo con value: factory(): recomputo.
deps es undefineddepsEqual devuelve false: recomputa en cada render (sin memoización efectiva).
Object.is, no igualdad profunda

La comparación es superficial y referencial. Si una dep es un objeto o arreglo recreado en cada render ({}, [], una función literal), Object.is siempre da falso y useMemo recomputa siempre. Para que la memoización sirva, las deps tienen que ser valores estables (primitivos, o referencias que a su vez estén memoizadas).

La actualización / disparador

useMemo no tiene disparador propio: no expone ningún dispatch, no encola nada en una queue y nunca llama a scheduleUpdate. No puede provocar un re-render por sí mismo. Lo que lo hace recalcular es, exclusivamente, que otro hook (típicamente un useState) dispare un re-render y que, en ese nuevo render, alguna de sus deps haya cambiado de identidad. En otras palabras: useMemo es un pasajero del ciclo de render, nunca su conductor.

Orden de ejecución

El chequeo de deps y la eventual ejecución de factory() ocurren en línea, durante el render, en el momento exacto en que el cuerpo del componente invoca useMemo. Esto lo distingue de useEffect, que difiere su trabajo hasta después del commit del DOM. Por eso la fábrica de useMemo debe ser pura y sin efectos secundarios: corre en plena fase de render, antes de que reconcileChildren procese la salida.

Reglas de los hooks

Como todos los hooks, useMemo se identifica por su posición: slot() lee fiber.alternate.hooks[R.hookIndex] y commitHook incrementa R.hookIndex. Si llamaras a useMemo condicionalmente, el índice se desalinearía y en el próximo render recuperarías el hook de otra posición: por ejemplo el MemoHook podría chocar contra un StateHook. De ahí la regla de oro: los hooks se llaman siempre, en el mismo orden, en el nivel superior del componente.

Test que lo fija

El test "useMemo recalcula solo cuando cambian deps" monta un componente con useMemo(() => { computes++; return a * 2; }, [a]). Tras el mount computes vale 1; un setB(1) (que no toca a) lo deja en 1; recién un setA(1) lo lleva a 2. Demuestra que la fábrica solo corre cuando la dep [a] cambia.

Diagrama de secuencia

MOUNT (a=0, b=0) useMemo(factory, [0]) old = undefined ──► factory() corre ──► value = 0, deps = [0] computes = 1 UPDATE por setB(1) (a=0, b=1) ── re-render del componente useMemo(factory, [0]) old.deps = [0], deps = [0] ──► depsEqual = true hook = oldMemo ──► factory() NO corre computes = 1 (sin cambios) UPDATE por setA(1) (a=1, b=1) ── re-render del componente useMemo(factory, [1]) old.deps = [0], deps = [1] ──► Object.is(0,1) = false ──► depsEqual = false value = factory() = 2 ──► computes = 2 (En ningún caso useMemo agenda el re-render: lo dispara setState.)
