Calcular sin tocar el DOM

Render phase

La render phase es la etapa donde el motor procesa cada fiber, ejecuta los componentes de función y calcula qué debe cambiar, pero sin mutar el DOM real. Todo el trabajo ocurre en memoria y puede interrumpirse.

El corazón: performUnitOfWork

El workLoop procesa el árbol una unidad a la vez. Cada unidad es un fiber, y la función que lo procesa es performUnitOfWork. Su primera responsabilidad es decidir de qué clase de componente se trata para elegir la rama de procesamiento; su segunda responsabilidad es devolver el siguiente fiber a procesar (child, luego sibling, luego subiendo por los parents).

src/React.ts
function performUnitOfWork(fiber) {
  const isFunctionComponent = fiber.type instanceof Function
  if (isFunctionComponent) {
    updateFunctionComponent(fiber)
  } else {
    updateHostComponent(fiber)
  }
  // Return  next unit of work
  //Finally we search for the next unit of work. We first try with the child, then with the sibling, then with the uncle, and so on.
  if (fiber.child) {
    return fiber.child
  }
  let nextFiber = fiber
  while (nextFiber) {
    if (nextFiber.sibling) {
      return nextFiber.sibling
    }
    nextFiber = nextFiber.parent
  }
}

La bifurcación: función vs. host

La discriminación es notablemente simple: const isFunctionComponent = fiber.type instanceof Function. Cuando escribimos <Counter />, el pragma createElement guarda en fiber.type la referencia a la función Counter, no una cadena. En cambio, cuando escribimos <div>, fiber.type es el string "div". Así, fiber.type instanceof Function distingue con exactitud entre un componente de función (a ejecutar) y un componente anfitrión (host, un nodo del DOM). Según el resultado se toma una de dos ramas mutuamente excluyentes: updateFunctionComponent o updateHostComponent.

Nota

Un componente de función no tiene DOM propio. Su cometido es producir hijos ejecutándose; el DOM lo aportan los host components que aparezcan en lo que retorna.

Rama 1: updateFunctionComponent

Antes de ejecutar el componente hay que preparar el terreno para los hooks. Se fija el work in progress fiber global (wipFiber = fiber), se resetea el contador de hooks a cero (hookIndex = 0) y se inicializa una lista vacía de hooks en el propio fiber (wipFiber.hooks = []). Esta preparación es la que permite que useState y useEffect sepan a qué fiber pertenecen y en qué posición van.

src/React.ts
function updateFunctionComponent(fiber) {
  wipFiber = fiber
  hookIndex = 0
  wipFiber.hooks = []
  const children = [fiber.type(fiber.props)]
  reconcileChildren(fiber, children)
  // ...recolección de efectos...
}

La línea decisiva es const children = [fiber.type(fiber.props)]: aquí se ejecuta la función del componente. Como fiber.type es Counter, esto equivale a llamar Counter(fiber.props), lo que corre su cuerpo (incluidos los useState/useEffect) y devuelve el elemento JSX que retorna. Ese resultado único se envuelve en un array y se pasa a reconcileChildren, que lo compara contra el árbol anterior. A diferencia de un host component, aquí los hijos provienen de correr la función, no de leerlos directamente de las props.

Rama 2: updateHostComponent

Para un nodo del DOM el trabajo es distinto. Si el fiber todavía no tiene un nodo asociado (!fiber.dom), se crea con createDom(fiber); luego se reconcilia contra fiber.props.children, es decir, los hijos que ya venían en las props.

src/React.ts
function updateHostComponent(fiber) {
  if (!fiber.dom) {
    fiber.dom = createDom(fiber)
  }
  reconcileChildren(fiber, fiber.props.children)
}

Punto crucial: crear no es montar

Aquí se concentra la idea central de la fase. createDom efectivamente llama a document.createElement(fiber.type) (o createTextNode), pero el nodo resultante queda flotando en memoria: nunca se hace appendChild sobre el documento. La creación del nodo y la aplicación de sus props inicial ocurren, pero la inserción en el árbol visible se pospone. Esa mutación real del DOM sucede más tarde, toda de golpe, en la commit phase.

Idea clave

La render phase es interrumpible precisamente porque no toca el DOM visible. Si el navegador corta el trabajo a la mitad, el usuario nunca ve una UI incompleta: solo se descartan nodos que aún vivían en memoria. La coherencia se garantiza porque commitRoot solo corre cuando nextUnitOfWork se agotó, es decir, cuando todo el árbol ya fue calculado.

En resumen, la render phase recorre fiber por fiber, ejecuta los componentes de función para obtener sus hijos, crea (sin insertar) los nodos que faltan y delega en la reconciliación el cálculo de las diferencias que después aplicará el commit.
