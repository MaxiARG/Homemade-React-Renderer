Estado local

useState — ciclo de vida completo

useState es la pieza que le da memoria a un componente función. Su valor no vive en la función — que se re-ejecuta entera en cada render — sino en un objeto Hook que persiste reutilizándose desde el fiber del render anterior.

Firma y para qué sirve

La firma es useState<S>(initial: S): [S, Dispatch<SetStateAction<S>>]: recibe un valor inicial (o una función que lo produce) y devuelve el par [estado, dispatch]. El dispatch (el clásico setState) acepta tanto un valor directo como una función actualizadora prev → next.

src/react/hooks.ts
export function useState<S>(initial: S): [S, Dispatch<SetStateAction<S>>] {
  const { fiber, old } = slot();
  const hook =
    (old as StateHook<S> | undefined) ??
    ({
      tag: 'state',
      state: typeof initial === 'function' ? (initial as () => S)() : initial,
      queue: [],
      dispatch: null as unknown as Dispatch<SetStateAction<S>>,
    } as StateHook<S>);
  // ...
}

Un uso mínimo: const [n, setN] = useState(0), y luego setN(n + 1) o setN(c => c + 1) para programar un re-render.

Dónde vive el estado

El estado vive en un StateHook, una de las ramas de la unión discriminada Hook definida en types.ts. Su etiqueta es tag: 'state' y guarda tres cosas: el state actual, una queue de acciones pendientes y el dispatch estable.

src/react/types.ts
export interface StateHook<S = unknown> {
  tag: 'state';
  state: S;
  queue: Array<(prev: S) => S>;
  dispatch: Dispatch<SetStateAction<S>>;
}

La persistencia es el corazón del modelo: en cada render, slot() lee el hook del render anterior desde fiber.alternate?.hooks?.[R.hookIndex]. Si existe, se reutiliza el mismo objeto (con su state, su queue y su dispatch ya creados); solo si no existe se crea uno nuevo. Ese reuso es lo que hace que el valor sobreviva a la re-ejecución de la función.

src/react/hooks.ts
function slot(): { fiber: Fiber; old: Hook | undefined } {
  const fiber = R.wipFiber;
  if (!fiber || !fiber.hooks) {
    throw new Error('Los hooks solo pueden llamarse dentro del render de un componente.');
  }
  const old = fiber.alternate?.hooks?.[R.hookIndex];
  return { fiber, old };
}

Primer render (mount)

  1. El reconciliador, en updateFunctionComponent, fija R.wipFiber = fiber, pone R.hookIndex = 0 y crea fiber.hooks = [] antes de ejecutar el cuerpo del componente.
  2. Al invocar useState(0), slot() devuelve old = undefined (no hay alternate todavía), así que se construye un StateHook fresco.
  3. El valor inicial se evalúa: si initial es una función se llama ((initial as () => S)(), la variante lazy initializer); si no, se usa tal cual.
  4. La queue arranca vacía y aún no hay acciones que aplicar.
  5. commitHook(fiber, hook) empuja el hook a fiber.hooks e incrementa R.hookIndex++, dejando el índice listo para el próximo hook.
src/react/hooks.ts
function commitHook(fiber: Fiber, hook: Hook): void {
  fiber.hooks!.push(hook);
  R.hookIndex++;
}

Renders siguientes (update)

En un re-render, slot() encuentra el hook viejo en fiber.alternate.hooks[hookIndex] y el operador ?? lo reutiliza en vez de crear otro. Antes de devolver el estado, se aplica la cola: cada acción es una función prev → next que se ejecuta en secuencia sobre hook.state, y luego la cola se vacía.

src/react/hooks.ts
// Aplicar acciones encoladas (valor directo o funcional).
if (hook.queue.length > 0) {
  for (const action of hook.queue) {
    hook.state = action(hook.state);
  }
  hook.queue.length = 0;
}

Nótese que la cola guarda siempre funciones actualizadoras. Cuando se llama setN(5) con un valor directo, el dispatch lo envuelve en () => 5 antes de encolarlo, de modo que el bucle de aplicación es uniforme. Aplicar varias acciones en orden da el batching correcto: cada una recibe el resultado de la anterior, sin depender de una variable n ya obsoleta del render viejo.

La actualización / disparador

El dispatch se crea una sola vez (guardado con if (!hook.dispatch)) para que su identidad sea estable entre renders — igual que React garantiza que el setter no cambia. Como el objeto hook se reutiliza, ese dispatch también sobrevive. Al llamarlo: normaliza la acción a función, la empuja a la cola, marca el fiber como sucio y programa el trabajo.

src/react/hooks.ts
if (!hook.dispatch) {
  const capturedHook = hook;
  hook.dispatch = (action: SetStateAction<S>) => {
    const updater =
      typeof action === 'function' ? (action as (p: S) => S) : () => action;
    capturedHook.queue.push(updater);
    const target = fiberRef.current;
    if (target) {
      target.hasPendingUpdate = true;
      scheduleUpdate(target);
    }
  };
}
const fiberRef = (hook as unknown as { _fiberRef?: { current: Fiber } })._fiberRef ?? {
  current: fiber,
};
fiberRef.current = fiber;

El detalle sutil es fiberRef.current. El dispatch es estable pero el fiber cambia en cada render (doble buffer). Si el closure capturara el fiber del mount, apuntaría a un fiber viejo. En su lugar captura un fiberRef mutable que en cada render se actualiza a fiber: así el dispatch siempre apunta al fiber vigente. Marca hasPendingUpdate = true (lo que desactiva el bailout) y llama a scheduleUpdate.

src/react/scheduler.ts
export function scheduleUpdate(fiber: Fiber): void {
  const root = fiber.root;
  if (!root) return;
  root.dirtyFibers.add(fiber);
  pendingRoots.add(root);
  ensureFlush();
}

ensureFlush agenda un queueMicrotask — salvo que estemos dentro de flushSync (batchDepth > 0), en cuyo caso el trabajo se drena al salir. Ese es el batching: varios dispatch seguidos apilan acciones en la cola y comparten un único render.

Orden de ejecución

El useState corre durante la render phase, dentro de updateFunctionComponent, en el mismo momento en que se ejecuta el cuerpo del componente. La aplicación de la cola sucede antes de devolver el estado, así que el componente ya renderiza con el valor nuevo. El re-render que dispara el dispatch, en cambio, es asíncrono (microtask) a menos que se fuerce con flushSync. Los efectos corren después, tras el commit.

Reglas de los hooks

Todo el mecanismo se apoya en que R.hookIndex sea estable: el hook nº0 de este render debe corresponder al hook nº0 del anterior, porque slot() recupera el estado viejo justamente por alternate.hooks[hookIndex]. El índice avanza únicamente en commitHook, en el orden en que se llaman los hooks.

Cuidado

Por eso useState no puede llamarse dentro de un if o un bucle: si en un render se saltara una llamada, todos los índices siguientes se desalinearían y cada hook recuperaría el estado equivocado. El orden de las llamadas es la identidad de cada hook.

Diagrama de secuencia

MOUNT updateFunctionComponent: R.wipFiber = fiber, R.hookIndex = 0, fiber.hooks = [] useState(0) slot() -> old = undefined (no hay alternate) crea StateHook { tag:'state', state:0, queue:[], dispatch:null } crea dispatch estable + fiberRef.current = fiber commitHook -> hooks.push, hookIndex++ return [0, dispatch] | v DISPATCH setN(c => c + 1) queue.push(c => c + 1) target = fiberRef.current (fiber vigente) target.hasPendingUpdate = true scheduleUpdate(target) -> dirtyFibers.add, ensureFlush (microtask / flushSync) | v UPDATE (nuevo render del mismo fiber) slot() -> old = alternate.hooks[hookIndex] (MISMO StateHook) aplica queue: state = (c => c + 1)(0) = 1 ; queue.length = 0 dispatch ya existe (identidad estable) ; fiberRef.current = fiber nuevo commitHook -> hookIndex++ return [1, dispatch] -> commit escribe el DOM
