Ciclo de vida completo

Hemos diseccionado cada pieza por separado: createElement, el fiber, el workLoop, la reconciliación, el commit y los hooks. Ahora seguiremos a un único componente —Counter— de punta a punta, desde que React.render lo arranca hasta que un clic dispara una actualización y sus efectos se limpian y vuelven a correr.

El componente que trazamos

Counter concentra los tres mecanismos interesantes: estado con useState, un efecto con cleanup dependiente de [counter], y un manejador onClick que provoca la actualización. Sus console.log nos servirán de sondas para observar cada fase.

src/Examples/components/Counter.tsx
const Counter = () => {
  const [counter, setCounter] = window.React.useState(2)
  console.log('Counter Render started ejecutado exitosamente')

  window.React.useEffect(() => {
    console.log('Counter useEffect ejecutado exitosamente')
    return (() => {
       console.log('xx Counter useEffect cleanup ejecutado exitosamente xx')
    })
  }, [counter])

  console.log('Counter Render finished ejecutado exitosamente')
  return (
    <div>
      <button onClick={() => { console.log('onClick'); setCounter(c => c + 1); }}>
        Click me
      </button>
      Counter: {counter}
    </div>
  );
};

Fase 1 — Montaje inicial

Cuando la página carga, React.render(element, container) siembra el árbol de trabajo y arranca la maquinaria perezosa del scheduler.

  1. React.render construye el wipRoot —el fiber raíz— con dom: container, alternate: currentRoot (aún null) y lo apunta con nextUnitOfWork = wipRoot. Vacía deletions.
  2. requestIdleCallback invoca workLoop cuando el navegador está ocioso. El bucle procesa unidades de trabajo mientras nextUnitOfWork exista y quede tiempo (shouldYield).
  3. performUnitOfWork corre por cada fiber. Al llegar al fiber de Counter, detecta fiber.type instanceof Function y delega en updateFunctionComponent.
  4. updateFunctionComponent fija wipFiber, reinicia hookIndex = 0 y wipFiber.hooks = [], y ejecuta la función: fiber.type(fiber.props). Aquí se imprime Counter Render started, se registran los hooks (useState, useEffect) y se imprime Counter Render finished. El efecto, con hasChanged = true en el montaje, se encola en wipEffects.
  5. reconcileChildren compara los elementos devueltos contra el árbol anterior (vacío), por lo que cada hijo se marca con effectTag PLACEMENT.
  6. commitRoot corre cuando nextUnitOfWork se agota y aún hay wipRoot. Recorre las deletions, luego commitWork(wipRoot.child), que crea e inserta los nodos reales en el DOM.
  7. useEffect ejecutado: ya con el DOM montado, commitRoot drena wipEffects y se imprime Counter useEffect ejecutado. Este es el momento de los passive effects.

Secuencia de consola esperada en el montaje:

Counter Render started ejecutado exitosamente Counter Render finished ejecutado exitosamente Counter useEffect ejecutado exitosamente

Fase 2 — Actualización por clic

El usuario pulsa el botón. Aquí es donde el alternate y la cola de acciones de useState cobran vida.

  1. onClick se dispara (imprime onClick) y llama setCounter(c => c + 1).
  2. setState encola la acción en hook.queue y reprograma el render: crea un nuevo wipRoot con dom: currentRoot.dom, props: currentRoot.props y alternate: currentRoot, fija nextUnitOfWork = wipRoot y vacía deletions. No renderiza aún: solo agenda trabajo para el próximo workLoop.
  3. Nuevo render: el workLoop vuelve a invocar performUnitOfWork → updateFunctionComponent, que ejecuta Counter de nuevo (imprime otra vez Counter Render started).
  4. useState aplica la acción: recupera el oldHook desde wipFiber.alternate.hooks[hookIndex], copia su state, y reproduce las acciones acumuladas (actions.forEach(action => hook.state = action(hook.state))). Así counter pasa de 2 a 3.
  5. reconcileChildren marca UPDATE: como los tipos coinciden con el árbol previo (sameType), los fibers se reutilizan conservando su dom y se etiquetan UPDATE.
  6. commit actualiza el nodo de texto: commitWork detecta el effectTag UPDATE y llama updateDom sobre el nodo existente; el texto Counter: 2 se reescribe a Counter: 3. No se recrea el DOM, solo se parchea.
  7. cleanup + nuevo efecto: counter cambió, así que hasChanged es true. Al drenar wipEffects en commitRoot, la función encolada ejecuta primero el cleanup del render anterior y luego el efecto nuevo.

Secuencia de consola esperada al hacer clic:

onClick Counter Render started ejecutado exitosamente Counter Render finished ejecutado exitosamente xx Counter useEffect cleanup ejecutado exitosamente xx Counter useEffect ejecutado exitosamente
Idea clave

El render se ejecuta durante la fase de render (los console.log del cuerpo del componente), pero los efectos —y el cleanup— se ejecutan en la fase de passive effects, después del commit. Por eso los mensajes de Render preceden siempre a los de cleanup y useEffect.

Línea de tiempo de las tres fases

El siguiente diagrama sitúa cada console.log de Counter en la fase donde ocurre. La fase de render puede interrumpirse (shouldYield); la de commit y la de effects, no.

RENDER PHASE | COMMIT PHASE | PASSIVE EFFECTS (interrumpible) | (síncrona) | (post-commit) ----------------------- | ---------------------- | ------------------------ performUnitOfWork | commitRoot | wipEffects.forEach(e=>e()) updateFunctionComponent | deletions.forEach | fiber.type(props) | commitWork(child) | hook.cleanup() (previo) useState / useEffect | updateDom / append | hook.effect() (nuevo) reconcileChildren | muta el DOM real | ----------------------- | ---------------------- | ------------------------ "Render started" | | "cleanup ejecutado" "Render finished" | | "useEffect ejecutado"
Nota

Que las tres fases estén tan separadas es lo que permite, en React real, pausar y reanudar el render sin dejar el DOM a medias: nada se muta hasta el commit. Este motor hereda esa disciplina aunque, por simplicidad, recorra todo el árbol en cada pasada. Repasa los detalles de cada etapa en Render phase y Commit phase.
