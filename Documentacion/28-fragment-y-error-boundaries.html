Agrupar sin nodo + capturar errores

Fragment y Error Boundaries

Dos mecanismos que comparten una idea: intervenir en la estructura del árbol de fibers sin ensuciar el DOM. El Fragment agrupa hijos sin crear un nodo contenedor; los Error Boundaries capturan excepciones de un subárbol y las convierten en una UI de reemplazo, todo sin abandonar el render en curso.

Parte 1 · Fragment

Un símbolo, no un componente

Un Fragment no es una función ni una clase: es un símbolo único, comparable por identidad. Vive en types.ts junto al resto de las etiquetas del motor.

src/react/types.ts
/** Símbolo único que identifica un Fragment (`<>...</>`). */
export const Fragment = Symbol('React.Fragment');
export type FragmentType = typeof Fragment;

Al ser un Symbol, dos referencias a Fragment siempre son la misma (Fragment === Fragment), lo que permite reconocerlo con una comparación estricta durante la reconciliación. element.ts lo reexporta para que el usuario lo escriba como type de un elemento (y el pragma React.Fragment del JSX apunte aquí):

src/react/element.ts
import {
  Fragment,
  TEXT_ELEMENT,
  // ...
} from './types.js';

export { Fragment };

Un fiber que no crea DOM

La clave está en beginWork. El Fragment se agrupa con los fibers estructurales (el ROOT y los host), pero mientras un host component crea su nodo con createDom, el Fragment no toca el DOM en absoluto: solo reconcilia sus hijos.

src/react/reconciler.ts
export function beginWork(fiber: Fiber): Fiber | null {
  const type = fiber.type;

  if (isProvider(type)) return updateContextProvider(fiber);

  if (type === 'ROOT' || type === Fragment || isHostType(fiber)) {
    // Fibers "estructurales": intentan bailout por props referenciales.
    if (canBailout(fiber)) {
      cloneChildFibers(fiber.alternate as Fiber, fiber);
      return fiber.child;
    }
    return type === 'ROOT' || type === Fragment
      ? updateFragmentLike(fiber)
      : updateHostComponent(fiber);
  }
  // ...
}

El trabajo real es mínimo. updateFragmentLike reconcilia props.children directamente contra el árbol anterior y devuelve el primer hijo, sin instanciar ningún nodo:

src/react/reconciler.ts
function updateFragmentLike(fiber: Fiber): Fiber | null {
  reconcileChildren(fiber, fiber.props.children ?? []);
  return fiber.child;
}

Compará con updateHostComponent, que hace lo mismo pero antes crea (o reutiliza) el nodo del DOM. Que el Fragment comparta código con el ROOT no es casualidad: ambos son contenedores lógicos sin nodo propio. Por eso el mismo updateFragmentLike sirve para los dos.

Cómo llegan sus hijos al DOM

Si el Fragment no tiene nodo, ¿dónde se insertan sus hijos? La respuesta está en la commit phase. Cuando hay que colocar un subárbol, placeNode distingue entre fibers host (que aportan un dom) y todo lo demás (Fragment, función, clase), que no aporta nodo y por lo tanto delega en sus hijos:

src/react/reconciler.ts
function placeNode(fiber: Fiber, parentDom: DomNode, before: DomNode | null): void {
  if (isHostType(fiber)) {
    if (fiber.dom) {
      if (before) parentDom.insertBefore(fiber.dom, before);
      else parentDom.appendChild(fiber.dom);
    }
  } else {
    // Fragment / función / clase: colocar sus hijos host.
    let child = fiber.child;
    while (child) {
      placeNode(child, parentDom, before);
      child = child.sibling;
    }
  }
}

El efecto es que los hijos de un Fragment se cuelgan del mismo padre DOM que colgaría al Fragment, uno tras otro, como si el Fragment no existiera. Lo mismo hace getHostParentDom al subir buscando el primer ancestro con dom: los fibers Fragment se saltan porque su dom es null.

Árbol de fibers DOM resultante <ul> (host, dom=UL) <ul> └ Fragment (dom=null) <li>a</li> ├ <li>a</li> ───────▶ <li>b</li> └ <li>b</li> </ul> El Fragment desaparece: sus <li> se insertan directamente bajo el <ul> (primer ancestro con dom).
Igual que React

En React, <>...</> tampoco produce un nodo del DOM. La diferencia práctica es que este motor no soporta key en la sintaxis corta; para keyed lists se usan elementos host o componentes con key, como muestra el capítulo de reconciliación por key.

Parte 2 · Error Boundaries

La clase base Component

Los Error Boundaries requieren un componente de clase, porque necesitan un lugar donde vivir el manejador de error y el estado de fallback. El motor ofrece una clase base mínima al estilo React en component.ts:

src/react/component.ts
export abstract class Component<
  P extends object = Props,
  S extends Record<string, unknown> = Record<string, unknown>,
> implements ComponentInstance<P> {
  static isReactComponent = true;

  props: P;
  state: S;
  _fiber?: Fiber;

  constructor(props: P) {
    this.props = props;
    this.state = {} as S;
  }

  setState(partial: Partial<S>): void {
    this.state = { ...this.state, ...(partial as S) };
    if (this._fiber) {
      this._fiber.hasPendingUpdate = true;
      scheduleUpdate(this._fiber);
    }
  }

  abstract render(): ReactElement | null;
}

La marca estática isReactComponent = true es lo que permite reconocer una clase durante la reconciliación (JavaScript no distingue funciones de constructores por sí solo). Sobre ella se apoyan dos predicados:

src/react/component.ts
export function isClassComponent(type: unknown): /* ... */ {
  return (
    typeof type === 'function' &&
    (type as { isReactComponent?: boolean }).isReactComponent === true
  );
}

export function isErrorBoundary(fiber: Fiber): boolean {
  const type = fiber.type as {
    getDerivedStateFromError?: unknown;
    prototype?: { componentDidCatch?: unknown };
  };
  return (
    isClassComponent(fiber.type) &&
    (typeof type.getDerivedStateFromError === 'function' ||
      typeof type.prototype?.componentDidCatch === 'function')
  );
}

Un componente de clase es un Error Boundary solo si define getDerivedStateFromError (método estático que deriva el nuevo estado a partir del error) o componentDidCatch (método de instancia para efectos secundarios, como loggear). La misma lógica se replica en reconciler.ts para no crear una dependencia circular con el scheduler.

Instanciación y memoria de la pila

Cuando beginWork detecta una clase, delega en updateClassComponent, que crea la instancia una sola vez (la guarda en fiber.stateNode), la enlaza al fiber con _fiber y llama a render(). El detalle crucial para los boundaries: antes de renderizar, si el fiber es un boundary, guarda la profundidad de la pila de contextos y de forceRenderDepth.

src/react/reconciler.ts
function updateClassComponent(fiber: Fiber): Fiber | null {
  let instance = fiber.stateNode;
  if (instance === null) {
    instance = new Ctor(fiber.props);
    fiber.stateNode = instance;
  }
  instance.props = fiber.props;
  instance._fiber = fiber;
  fiber.hasPendingUpdate = false;

  // Los error boundaries recuerdan el estado de la pila de contextos para
  // poder restaurarlo si un descendiente lanza una excepción.
  if (isErrorBoundary(fiber)) {
    fiber.ctxDepth = contextStack.length;
    fiber.forceDepth = R.forceRenderDepth;
  }

  const child = instance.render();
  reconcileChildren(fiber, child ? [child] : []);
  return fiber.child;
}

¿Por qué guardar ctxDepth y forceDepth? Porque cuando un descendiente lanza, el render se abandona a mitad de camino: puede haber Providers que empujaron valores a contextStack e incrementaron forceRenderDepth sin que su completeWork (que hace el pop) llegara a ejecutarse. Sin restaurar esas pilas, el estado global quedaría corrupto para el resto del árbol.

Captura: workLoopSync + handleRenderError

El bucle de trabajo síncrono envuelve cada performUnitOfWork en un try/catch. Cualquier excepción durante el render de un fiber se desvía a handleRenderError en lugar de romper el flush.

src/react/scheduler.ts
function workLoopSync(root: RootStateContainer): void {
  while (root.nextUnitOfWork) {
    try {
      root.nextUnitOfWork = performUnitOfWork(root.nextUnitOfWork);
    } catch (error) {
      handleRenderError(root, error);
    }
  }
}

handleRenderError sube desde el fiber que falló buscando el boundary más cercano, restaura las pilas, aplica el estado derivado y reprograma el subárbol del boundary para renderizarlo de nuevo, ahora con el fallback:

src/react/scheduler.ts
function handleRenderError(root: RootStateContainer, error: unknown): void {
  const failing = root.nextUnitOfWork;
  let boundary: Fiber | null = failing ? failing.parent : null;
  while (boundary && !(boundary.stateNode && isErrorBoundary(boundary))) {
    boundary = boundary.parent;
  }
  if (!boundary) {
    R.activeRoot = null;
    throw error; // sin boundary: el error se propaga
  }

  // Restaurar la pila de contextos al punto en que empezó el boundary.
  contextStack.length = boundary.ctxDepth;
  R.forceRenderDepth = boundary.forceDepth;

  const instance = boundary.stateNode as ComponentInstance;
  const Ctor = boundary.type as {
    getDerivedStateFromError?: (e: unknown) => object;
  };
  const derived =
    typeof Ctor.getDerivedStateFromError === 'function'
      ? Ctor.getDerivedStateFromError(error)
      : {};
  instance.state = { ...instance.state, ...(derived as Record<string, unknown>) };
  if (typeof instance.componentDidCatch === 'function') {
    instance.componentDidCatch(error, { componentStack: '' });
  }

  // Re-renderizar SOLO el subárbol del boundary con el estado de fallback.
  boundary.child = null;
  boundary.hasPendingUpdate = true;
  root.nextUnitOfWork = boundary;
}
  1. Búsqueda hacia arriba. Desde failing.parent se recorren los parent hasta hallar un fiber con stateNode instanciado que además sea Error Boundary.
  2. Sin boundary. Si no hay ninguno, el error se relanza y sale del motor: como en React, un error no capturado desmonta la app.
  3. Restauración de pilas. contextStack.length = boundary.ctxDepth descarta las entradas empujadas por Providers a medio completar, y forceRenderDepth vuelve a su valor guardado.
  4. Estado derivado. Se fusiona el resultado de getDerivedStateFromError(error) en instance.state, y si existe se llama a componentDidCatch para efectos.
  5. Re-render acotado. Se limpia boundary.child, se marca hasPendingUpdate y se apunta nextUnitOfWork al propio boundary: el while del workLoopSync continúa y vuelve a entrar al boundary, que ahora renderiza el fallback.

El test que lo verifica

La batería de tests incluye un caso directo: un Boundary con getDerivedStateFromError que envuelve un componente Boom cuyo render lanza. El resultado esperado es que el DOM muestre el fallback, no la excepción.

src/react/__tests__/engine.test.ts
class Boundary extends Component<{ children?: ReactElement[] }, { failed: boolean }> {
  constructor(props: { children?: ReactElement[] }) {
    super(props);
    this.state = { failed: false };
  }
  static getDerivedStateFromError() {
    return { failed: true };
  }
  render() {
    if (this.state.failed) return h('p', null, 'fallback');
    return h('div', null, this.props.children as unknown as ReactElement);
  }
}
function Boom(): ReactElement {
  throw new Error('boom');
}
const { container } = mount(h(Boundary, null, h(Boom)));
expect(container.textContent).toBe('fallback'); // ✓

Flujo de propagación del error

performUnitOfWork(Boom) ──lanza──▶ Error('boom') │ │ │ workLoopSync try/catch ▼ │ handleRenderError(root, error) ◀─────┘ │ │ 1. boundary = Boom.parent ... sube por .parent │ hasta isErrorBoundary === true ▼ [Boundary encontrado] │ │ 2. contextStack.length = boundary.ctxDepth │ forceRenderDepth = boundary.forceDepth │ 3. state = { ...state, ...getDerivedStateFromError(error) } │ componentDidCatch(error, info)? │ 4. boundary.child = null; hasPendingUpdate = true ▼ nextUnitOfWork = boundary │ ▼ workLoopSync sigue ─▶ beginWork(Boundary) render() ve state.failed → <p>fallback</p> commit ▶ el DOM muestra el fallback
Los boundaries son solo de clase

Igual que en React, un componente función no puede ser Error Boundary: no hay equivalente en hooks a getDerivedStateFromError/componentDidCatch. El predicado isErrorBoundary exige isReactComponent === true (una clase que extiende Component) y al menos uno de esos dos manejadores. Un try/catch en el render de una función no captura errores de sus hijos, porque esos hijos se renderizan más tarde, en otra iteración del workLoopSync.

Alcance de la recuperación

La captura re-renderiza solo el subárbol del boundary (nextUnitOfWork = boundary), no todo el root. El resto del árbol ya comprometido no se ve afectado, y las pilas restauradas garantizan que los hermanos que aún faltan procesar vean el contexto correcto.
