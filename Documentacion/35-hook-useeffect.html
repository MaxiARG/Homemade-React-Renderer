Efectos secundarios

useEffect — ciclo de vida completo

useEffect ejecuta efectos secundarios después de que el DOM quedó mutado. El hook se registra durante el render, se recolecta si sus deps cambiaron, y se ejecuta al final del commit, con un cleanup que corre antes de re-ejecutar y también al desmontar el componente.

Firma y para qué sirve

La firma real en src/react/hooks.ts es useEffect(effect: () => void | (() => void), deps?: unknown[]): void. El effect puede devolver una función de limpieza (cleanup). Sirve para sincronizar el componente con el mundo externo: suscripciones, timers, logging, acceso imperativo al DOM ya montado.

uso mínimo
function Reloj() {
  const [t, setT] = useState(0);
  useEffect(() => {
    const id = setInterval(() => setT((x) => x + 1), 1000);
    return () => clearInterval(id); // cleanup: corre al desmontar / re-ejecutar
  }, []); // deps [] -> se suscribe una sola vez
  return h('span', null, String(t));
}

Dónde vive el estado

El estado del hook es un EffectHook, con cuatro campos además del tag:

src/react/types.ts
export interface EffectHook {
  tag: 'effect';
  effect: () => void | (() => void);   // la función a ejecutar
  deps: unknown[] | undefined;         // dependencias de esta ejecución
  cleanup: (() => void) | undefined;   // función devuelta por el effect anterior
  hasChanged: boolean;                 // ¿cambiaron las deps respecto al render previo?
}

El campo crítico que persiste entre renders es cleanup: se lee del hook viejo (wipFiber.alternate.hooks[hookIndex]) y se arrastra al hook nuevo, de modo que la limpieza de la ejecución anterior siga disponible aunque el objeto EffectHook se reconstruya en cada render.

Primer render (mount): registrar el efecto

A diferencia de useMemo, useEffect no ejecuta nada durante el render: solo construye el hook y calcula si debe correr. En el mount, oldEffect es undefined, así que hasChanged es true:

src/react/hooks.ts
export function useEffect(
  effect: () => void | (() => void),
  deps?: unknown[],
): void {
  const { fiber, old } = slot();
  const oldEffect = old as EffectHook | undefined;

  const hasChanged = oldEffect ? !deps || !depsEqual(deps, oldEffect.deps) : true;

  const hook: EffectHook = {
    tag: 'effect',
    effect,
    deps,
    hasChanged,
    cleanup: oldEffect?.cleanup, // se conserva el cleanup previo
  };
  commitHook(fiber, hook);
}
  1. slot() lee R.wipFiber y el hook viejo en R.hookIndex.
  2. Se calcula hasChanged (en el mount, true porque no hay oldEffect).
  3. Se crea el EffectHook, arrastrando oldEffect?.cleanup.
  4. commitHook lo empuja a fiber.hooks e incrementa R.hookIndex. Nótese que useEffect no devuelve valor.

Renders siguientes (update): el cálculo de hasChanged

En re-renders, hasChanged decide si el efecto vuelve a correr. La lógica oldEffect ? !deps || !depsEqual(deps, oldEffect.deps) : true distingue tres semánticas de dependencias:

depsComportamientoPor qué
omitido (undefined)Corre en cada render!deps es true, así que hasChanged siempre es true.
[]Corre una sola vez (mount)En updates hay oldEffect y depsEqual([], []) da true, entonces hasChanged es false.
[x]Corre cuando x cambiadepsEqual compara con Object.is; si difiere, hasChanged es true.

La recolección: collectPassiveEffects

Tras ejecutar el cuerpo del componente, updateFunctionComponent llama a collectPassiveEffects(fiber). Este recorre los hooks del fiber y, por cada EffectHook con hasChanged, empuja una tarea a root.pendingEffects:

src/react/reconciler.ts
function collectPassiveEffects(fiber: Fiber): void {
  if (!fiber.hooks) return;
  const root = fiber.root;
  if (!root) return;
  for (const hook of fiber.hooks) {
    if (hook.tag === 'effect' && hook.hasChanged) {
      const eff = hook as EffectHook;
      root.pendingEffects.push(() => {
        if (typeof eff.cleanup === 'function') eff.cleanup(); // limpia lo anterior
        const cleanup = eff.effect();                          // ejecuta el efecto
        eff.cleanup = typeof cleanup === 'function' ? cleanup : undefined;
      });
    }
  }
}

Importante: la tarea encapsula el patrón cleanup-antes-de-efecto. Cuando finalmente corra, primero limpia la ejecución previa y después ejecuta el efecto nuevo, guardando el cleanup que devuelva para la próxima vez.

La ejecución: commitRoot, después de mutar el DOM

Las tareas recolectadas se disparan al final de commitRoot, una vez que las eliminaciones y mutaciones del DOM ya se aplicaron:

src/react/reconciler.ts
export function commitRoot(root: RootStateContainer): void {
  root.deletions.forEach((f) => commitDeletion(f)); // aquí corren los cleanups de desmontaje
  root.deletions = [];

  if (root.wipRoot) commitWork(root.wipRoot.child); // muta el DOM (PLACEMENT / UPDATE)

  root.currentRoot = root.wipRoot;
  root.wipRoot = null;

  // Efectos pasivos (useEffect) DESPUÉS de mutar el DOM.
  const effects = root.pendingEffects;
  root.pendingEffects = [];
  effects.forEach((run) => run());
}
Por qué después del DOM

Los efectos corren al final a propósito: cuando el effect se ejecuta, el DOM ya refleja el render actual. Así un efecto puede medir un nodo, enfocarlo o suscribirse con la garantía de que la UI que ve el usuario ya está en su lugar. Es la contraparte de useMemo, que corre en el render.

El cleanup: en re-ejecución y en desmontaje

Hay dos caminos por los que corre un cleanup:

  • Re-ejecución (cambio de deps): lo dispara la propia tarea de collectPassiveEffects (if (typeof eff.cleanup === 'function') eff.cleanup()) justo antes de correr el efecto nuevo.
  • Desmontaje: cuando un fiber se elimina, commitDeletion llama a runCleanups antes de sacar los nodos del DOM.
src/react/reconciler.ts
function commitDeletion(fiber: Fiber): void {
  runCleanups(fiber);                 // primero limpiar efectos
  const parentDom = getHostParentDom(fiber);
  removeHostNodes(fiber, parentDom);  // después remover del DOM
}

function runCleanups(fiber: Fiber): void {
  if (fiber.hooks) {
    for (const hook of fiber.hooks) {
      if (hook.tag === 'effect' && typeof hook.cleanup === 'function') {
        hook.cleanup();
      }
    }
  }
  let child = fiber.child;
  while (child) { runCleanups(child); child = child.sibling; } // recursivo en el subárbol
}

Orden de ejecución y reglas de los hooks

El orden global es: render (el componente llama a useEffect, que solo registra y calcula hasChanged; luego collectPassiveEffects encola) → commit (se mutan/eliminan nodos del DOM; los cleanups de desmontaje corren en commitDeletion) → efectos (pendingEffects corre al final de commitRoot: cleanup previo y luego efecto nuevo).

Como los efectos se identifican por posición en fiber.hooks (índice fijado por R.hookIndex vía commitHook), useEffect no puede llamarse condicionalmente: un salto desalinearía la correspondencia entre el hook viejo y el nuevo, y hasChanged se calcularía contra las deps equivocadas.

Test que lo fija

El test "corre tras el commit y limpia en cambio de deps y desmontaje" usa un Child con useEffect(() => { log.push('effect '+n); return () => log.push('cleanup '+n); }, [n]). Tras el mount el log es ['effect 0']; un setN(1) produce ['effect 0', 'cleanup 0', 'effect 1'] (cleanup viejo antes del efecto nuevo); y desmontar el Child agrega 'cleanup 1'. El test hermano "deps [] corre una sola vez" confirma que con [] el efecto corre exactamente 1 vez pese a varios setN.

Diagrama de secuencia

MOUNT (n=0) render: useEffect(fx, [0]) → hasChanged = true (no hay oldEffect) collectPassiveEffects → push a root.pendingEffects commit: DOM montado efectos: fx() → log = ['effect 0'], cleanup = () => log.push('cleanup 0') UPDATE por setN(1) (n=1) render: useEffect(fx, [1]) → Object.is(0,1)=false → hasChanged = true (cleanup 0 se arrastra al hook nuevo) collectPassiveEffects → push a pendingEffects commit: DOM actualizado efectos: cleanup() → 'cleanup 0' luego fx() → 'effect 1' UNMOUNT (setShow(false)) commit: commitDeletion(Child) → runCleanups → cleanup() → 'cleanup 1' removeHostNodes(Child) (recién ahí se saca del DOM) log final: ['effect 0', 'cleanup 0', 'effect 1', 'cleanup 1']
