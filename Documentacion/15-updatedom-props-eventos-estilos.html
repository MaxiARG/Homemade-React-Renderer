Sincronizar props con el DOM

updateDom: props, eventos y estilos

updateDom es la única función del motor que realmente escribe sobre un nodo del DOM: compara las props viejas con las nuevas y aplica exactamente la diferencia.

Dónde encaja

updateDom(dom, prevProps, nextProps) se invoca en dos lugares. Al crear un nodo desde cero, createDom la llama con props previas vacías — updateDom(dom, {}, fiber.props) — de modo que todas las props se traten como «nuevas». Y durante la fase de commit, cuando un fiber tiene effectTag UPDATE, commitWork la invoca con updateDom(fiber.dom, fiber.alternate.props, fiber.props), comparando el estado anterior contra el nuevo.

Los cuatro helpers

Antes de la función hay cuatro pequeñas funciones auxiliares que clasifican y comparan las claves de props. Son la gramática con la que updateDom razona:

src/React.ts
const isEvent = key => key.startsWith("on")
//if the prop name starts with the "on" prefix we'll handle them differently.
const isProperty = key => key !== "children" && !isEvent(key)
const isNew = (prev, next) => key =>
  prev[key] !== next[key]
const isGone = (prev, next) => key => !(key in next)
  • isEvent: la clave empieza por on (ej. onClick) → es un manejador de eventos.
  • isProperty: no es children ni un evento → es una propiedad normal del nodo.
  • isNew(prev, next)(key): el valor cambió entre el render anterior y el nuevo.
  • isGone(prev, next)(key): la clave existía antes pero ya no está en las props nuevas.

Los cuatro pasos de updateDom

El cuerpo de la función es una secuencia de cuatro barridos, cada uno filtrando las claves con los helpers anteriores. El orden importa: primero se retira lo viejo, luego se aplica lo nuevo.

src/React.ts
function updateDom(dom, prevProps, nextProps) {
  //Remove old or changed event listeners
  Object.keys(prevProps)
    .filter(isEvent)
    .filter(key =>
      !(key in nextProps) || isNew(prevProps, nextProps)(key)
    )
    .forEach(name => {
      const eventType = name.toLowerCase().substring(2)
      dom.removeEventListener(eventType, prevProps[name])
    })

  // Remove old properties
  Object.keys(prevProps)
    .filter(isProperty)
    .filter(isGone(prevProps, nextProps))
    .forEach(name => {
      if (name === "style" && typeof prevProps.style === "object") {
        Object.keys(prevProps.style || {}).forEach(styleName => {
          dom.style[styleName] = "";
        });
      } else {
        dom[name] = "";
      }
    });

  // Set new or changed properties
  Object.keys(nextProps)
    .filter(isProperty)
    .filter(isNew(prevProps, nextProps))
    .forEach(name => {
      if (name === "style" && typeof nextProps.style === "object") {
        Object.assign(dom.style, nextProps.style);
      } else {
        dom[name] = nextProps[name];
      }
    });

  // Add event listeners
  Object.keys(nextProps)
    .filter(isEvent)
    .filter(isNew(prevProps, nextProps))
    .forEach(name => {
      const eventType = name.toLowerCase().substring(2)
      dom.addEventListener(eventType, nextProps[name])
    })
}
  1. Quitar listeners viejos o cambiados. Para cada clave on* que desapareció o cuyo valor cambió, se llama a removeEventListener. El nombre del evento se obtiene bajando a minúsculas y quitando el prefijo on con name.toLowerCase().substring(2): onClick → click.
  2. Quitar propiedades que desaparecieron. Para cada propiedad normal presente antes y ausente ahora (isGone), se limpia asignando dom[name] = "" (o el estilo, caso especial que veremos enseguida).
  3. Poner propiedades nuevas o cambiadas. Para cada propiedad que cambió (isNew), se asigna directamente dom[name] = nextProps[name].
  4. Agregar listeners nuevos. Finalmente, para cada evento nuevo o cambiado, se registra con addEventListener.
Nota

Retirar el listener viejo antes de añadir el nuevo evita fugas: si en cada render se limitara a hacer addEventListener, un mismo nodo acumularía manejadores duplicados. Como en useState cada setState genera funciones nuevas, sin este paso 1 la limpieza sería imposible.

Estilos: objeto contra cadena

La propiedad style recibe un trato especial porque puede llegar de dos maneras. Cuando es un objeto, el paso 3 usa Object.assign(dom.style, nextProps.style), que copia cada par propiedad/valor sobre el objeto style del nodo. Este es el patrón idiomático que se ve en el ejemplo StyledComponent:

src/Examples/StyledComponent.tsx
const StyledComponent = () => {
  return <div style={{
    color: "blue",
    backgroundColor: "white",
    padding: "20px"
  }}>Just a big Blue styled box</div>;
};

La doble llave style={{ ... }} no es un capricho de sintaxis: la llave exterior abre una expresión JSX y la interior es el objeto literal de estilos. Tras transpilar, se convierte en createElement("div", { style: { color: "blue", ... } }), y ese objeto es justo lo que Object.assign vuelca sobre dom.style.

Cuando style es una cadena (por ejemplo style="background: salmon"), la condición typeof nextProps.style === "object" es falsa y cae en la rama else, que hace dom[name] = nextProps[name]: asigna la cadena a dom.style, lo que el navegador interpreta como el cssText completo del elemento.

Bug corregido

Bug nº2 — Limpieza de estilo string. Al remover props viejas (paso 2), la rama de style hacía Object.keys(prevProps.style) asumiendo siempre un objeto. Con un estilo en cadena, Object.keys("background: salmon") devuelve los índices de los caracteres ("0", "1", …) y terminaba escribiendo dom.style["0"] = "", etc. La corrección fue guardar esa rama tras typeof prevProps.style === "object"; una cadena cae ahora al else, que limpia el cssText completo con dom[name] = "".

Clasificación de props

En resumen, cada clave de props recibe uno de cuatro tratamientos según qué sea:

Tipo de prop Detectada por Tratamiento en updateDom
children key === "children" Ignorada: no es un atributo del DOM, la manejan los fibers hijos.
Eventos on* isEvent addEventListener / removeEventListener con el nombre en minúsculas sin on.
style name === "style" Objeto → Object.assign(dom.style, obj); cadena → asignada a dom.style (cssText).
Resto (id, className, nodeValue…) isProperty Propiedad directa: dom[name] = value.
Idea clave

Este renderer asigna propiedades del DOM (dom[name]), no atributos (setAttribute). Por eso funciona className pero no class: la primera es una propiedad del elemento, la segunda solo un atributo. Es la misma convención que React expone en JSX.
