Confianza mediante tests

Testing con Jest + jsdom

Un motor de renderizado sin tests es una hipótesis. La batería de engine.test.ts ejerce el ciclo completo —render, hooks, reconciliación por key, error boundaries— contra un DOM real de jsdom, de forma determinista gracias a flushSync. Este capítulo explica el setup y cómo está estructurada la cobertura.

El setup: jsdom + babel-jest

El motor está pensado para el navegador (usa ES modules con extensión .js en los imports y se compila con modules:false para no romper el bundling nativo). Node no puede ejecutar eso tal cual, así que la configuración de tests define un entorno de transpilación propio, deliberadamente independiente del build.

jest.config.js
module.exports = {
  testEnvironment: 'jsdom',
  testMatch: ['**/__tests__/**/*.test.ts?(x)', '**/*.test.ts?(x)'],
  // Los imports usan extensión .js (requerida por ES modules en el navegador);
  // aquí la quitamos para resolver los archivos .ts en los tests.
  moduleNameMapper: {
    '^(\\.{1,2}/.*)\\.js$': '$1',
  },
  transform: {
    '^.+\\.[jt]sx?$': [
      'babel-jest',
      {
        babelrc: false,
        configFile: false,
        presets: [
          ['@babel/preset-env', { targets: { node: 'current' } }],
          '@babel/preset-typescript',
          [
            '@babel/preset-react',
            { pragma: 'React.createElement', pragmaFrag: 'React.Fragment' },
          ],
        ],
      },
    ],
  },
};

Tres decisiones merecen desglose:

  1. testEnvironment: 'jsdom'. Los tests corren en Node, pero jsdom provee un document y window completos. El motor puede llamar document.createElement, appendChild, insertBefore y leer innerHTML/textContent como si estuviera en un navegador, sin mocks del DOM.
  2. Transform con babel-jest a CommonJS. Se fija babelrc: false y configFile: false para ignorar el .babelrc del build (que usa modules:false). En su lugar, @babel/preset-env con targets: { node: 'current' } transpila a CommonJS, que es lo que Jest sabe cargar. @babel/preset-typescript borra los tipos y @babel/preset-react queda disponible por si se usa JSX (con el mismo pragma React.createElement).
  3. moduleNameMapper. Este es el puente entre los dos mundos. En el código fuente, todos los imports terminan en .js (obligatorio para ES modules nativos: import ... from './types.js'). Pero en disco los archivos son .ts. La regex '^(\\.{1,2}/.*)\\.js$' → '$1' quita la extensión .js de los imports relativos, dejando que el resolver de Jest encuentre el .ts correspondiente.
Por qué dos configuraciones de Babel

El build para navegador conserva los ES modules (modules:false) porque el runtime nativo los entiende y se sirven directamente. Jest, en cambio, necesita CommonJS. Aislar la config de tests con babelrc: false evita que una cambie a la otra: cada pipeline transpila para su propio destino.

El helper de montaje

Para no depender del pragma JSX, los tests usan createElement directamente bajo el alias h. Un helper mount() crea un contenedor, lo agrega al document.body, monta un root y renderiza:

src/react/__tests__/engine.test.ts
const h = createElement as (
  type: unknown,
  props?: Record<string, unknown> | null,
  ...children: unknown[]
) => ReactElement;

function mount(el: ReactElement): { container: HTMLElement; root: ReactRoot } {
  const container = document.createElement('div');
  document.body.appendChild(container);
  const root = createRoot(container);
  root.render(el);
  return { container, root };
}

afterEach(() => {
  document.body.innerHTML = '';
});

El afterEach limpia el body entre tests para que los múltiples roots no se pisen. Como cada root es independiente (encapsula su propio RootStateContainer), montar varios en el mismo test también es válido.

Determinismo con flushSync

El scheduler agrupa las actualizaciones en un microtask (batching), lo que en producción es deseable pero en un test introduce asincronía: tras un setN(1), el DOM todavía no cambió. Para verificar sincrónicamente, cada acción que dispara un re-render se envuelve en flushSync, que drena todo el trabajo pendiente antes de devolver el control:

src/react/__tests__/engine.test.ts
const btn = container.firstChild as HTMLButtonElement;
expect(btn.textContent).toBe('n=0');
flushSync(() => btn.click());   // procesa el setState del onClick al instante
expect(btn.textContent).toBe('n=1');

Así los expect se leen justo después del cambio, sin await ni timers falsos. Es el mismo flushSync del capítulo de batching, reutilizado como herramienta de test.

Áreas cubiertas

La batería (agrupada en bloques describe) recorre el motor de punta a punta:

ÁreaQué verifica
Render básicoHost + texto, ignora null/undefined/booleanos, className/atributos, estilo objeto que limpia sub-claves.
useState + batchingActualiza en un click; dos setState en un tick producen un solo re-render.
Bailout / memoUn setState no re-renderiza al hermano sin cambios; memo salta con props iguales.
useEffect + cleanupCorre tras el commit, limpia al cambiar deps y al desmontar; deps [] corre una sola vez.
useRef / useMemo / useCallbackRef conserva identidad; memo recalcula solo con deps distintas; callback estable con deps iguales.
useReducerDespacha acciones y actualiza el estado.
useContextLee el valor del Provider y re-renderiza al cambiarlo.
Reconciliación por keyReordenar preserva nodos del DOM y su estado; elimina items que salen de la lista.
Error boundariesCaptura un error de render y muestra el fallback.
Múltiples rootsDos roots son independientes: un setState en uno no afecta al otro.

Dos tests representativos

Batching: dos setState, un render

Este test cuenta cuántas veces se ejecuta el cuerpo del componente. Tras dos setState en el mismo flushSync, el conteo sube de 1 a 2 (no a 3): las dos actualizaciones se coalescen en un único re-render.

src/react/__tests__/engine.test.ts
test('batching: dos setState en un tick = un solo re-render', () => {
  let renders = 0;
  let setA!: Dispatch<SetStateAction<number>>;
  let setB!: Dispatch<SetStateAction<number>>;
  function App() {
    renders++;
    const [a, sa] = useState(0);
    const [b, sb] = useState(0);
    setA = sa;
    setB = sb;
    return h('div', null, `${a}-${b}`);
  }
  const { container } = mount(h(App));
  expect(renders).toBe(1);
  flushSync(() => {
    setA(1);
    setB(2);
  });
  expect(container.textContent).toBe('1-2');
  expect(renders).toBe(2); // no 3
});

Reordenar por key preserva nodo y estado

El test más exigente de reconciliación: cada Item tiene su propio estado (useState). Se incrementa el contador de "a", se reordena la lista a [c, a, b] y se verifica que el nodo del DOM de "a" sigue siendo el mismo objeto (toBe, identidad referencial) y conserva su estado a:1.

src/react/__tests__/engine.test.ts
test('reordenar preserva nodos del DOM y el estado', () => {
  let setOrder!: Dispatch<SetStateAction<string[]>>;
  function Item({ id }: { id: string }) {
    const [n, setN] = useState(0);
    return h(
      'li',
      { 'data-id': id, onClick: () => setN((c) => c + 1) },
      `${id}:${n}`,
    );
  }
  function List() {
    const [order, s] = useState(['a', 'b', 'c']);
    setOrder = s;
    return h('ul', null, ...order.map((id) => h(Item, { key: id, id })));
  }
  const { container } = mount(h(List));
  const ul = container.firstChild as HTMLElement;
  const nodeA = ul.querySelector('[data-id="a"]') as HTMLElement;

  flushSync(() => nodeA.click());
  expect(nodeA.textContent).toBe('a:1');

  flushSync(() => setOrder(['c', 'a', 'b']));
  const children = Array.from(ul.children) as HTMLElement[];
  expect(children.map((c) => c.getAttribute('data-id'))).toEqual(['c', 'a', 'b']);
  // El nodo "a" es el MISMO objeto y conserva su estado.
  expect(ul.querySelector('[data-id="a"]')).toBe(nodeA);
  expect(nodeA.textContent).toBe('a:1');
});

Que nodeA sobreviva un reordenamiento es la prueba de que el diffing por key reutiliza fibers en vez de recrearlos, tal como se explica en el capítulo de reconciliación por key.

Cómo correr los tests

El package.json expone los scripts habituales. Basta con:

terminal
npm test          # jest, una pasada
npm run test:watch  # jest --watch, re-ejecuta al guardar
