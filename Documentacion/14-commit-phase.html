Aplicar los cambios

Commit phase

La fase de commit es el momento en que todo el trabajo calculado durante la reconciliación se traslada, de una sola vez y sin interrupciones, al DOM real.

De la fase de render a la fase de commit

Durante la render phase el motor construye un árbol de fibers nuevo (el wipRoot) y, mediante reconciliación, etiqueta cada fiber con un effect tag (PLACEMENT, UPDATE o DELETION). Ese trabajo es interrumpible: el workLoop puede cederle el control al navegador tantas veces como haga falta porque, hasta aquí, nada ha tocado el DOM. El disparador de la fase de commit vive en el propio bucle de trabajo:

src/React.ts
function workLoop(deadline) {
  let shouldYield = false
  while (nextUnitOfWork && !shouldYield) {
    nextUnitOfWork = performUnitOfWork(nextUnitOfWork)
    shouldYield = deadline.timeRemaining() < 1
  }

  if (!nextUnitOfWork && wipRoot) {
    commitRoot();
  }

  requestIdleCallback(workLoop)
}

La condición !nextUnitOfWork && wipRoot es la clave: solo cuando ya no queda ninguna unidad de trabajo pendiente y existe un árbol en progreso, se llama a commitRoot(). Es decir, el árbol completo está calculado antes de que se aplique el primer cambio.

commitRoot: el orquestador

La función commitRoot coordina la fase entera en cuatro pasos precisos:

src/React.ts
function commitRoot() {
  //And then, when we are commiting the changes to the DOM, we also use the fibers from that array.
  deletions.forEach(commitWork)
  commitWork(wipRoot.child)
  currentRoot = wipRoot
  wipRoot = null

  // Ejecutar todos los useEffect. Estos siempre se ejecutan luego de haber ejecutado todo lo demas.
  wipEffects.forEach(e => e());
  wipEffects = [];
}
  1. Procesar las eliminaciones. deletions.forEach(commitWork) recorre el array deletions, que la reconciliación fue llenando con los fibers viejos marcados como DELETION. Se procesan primero porque esos fibers no forman parte del nuevo árbol wipRoot y deben tratarse aparte.
  2. Aplicar el resto del árbol. commitWork(wipRoot.child) arranca el recorrido recursivo desde el primer hijo de la raíz en progreso, aplicando las altas (PLACEMENT) y actualizaciones (UPDATE).
  3. Fijar el árbol comprometido. currentRoot = wipRoot guarda la referencia al árbol que acaba de tocar el DOM. Ese currentRoot será el alternate del próximo render, es decir, el «antes» contra el que se diferencia el «después». Luego wipRoot = null cierra la fase.
  4. Ejecutar los efectos. wipEffects.forEach(e => e()) corre todos los useEffect pendientes al final, cuando el DOM ya está actualizado, y luego vacía la lista con wipEffects = []. Este orden es deliberado: los efectos deben ver la interfaz ya pintada (véase 17 · Hook useEffect y cleanup).

commitWork: recorrido y mutación

commitWork es una función recursiva que visita cada fiber y aplica la mutación que su effectTag indica. Su primer trabajo, sin embargo, es encontrar el nodo del DOM padre. No basta con mirar fiber.parent: los componentes de función no tienen nodo propio (dom es null), así que hay que subir por el árbol hasta topar con un fiber que sí tenga dom.

src/React.ts
function commitWork(fiber) {
  if (!fiber) {
    return
  }

  let domParentFiber = fiber.parent
  //to find the parent of a DOM node we'll need to go up the fiber tree until we find a fiber with a DOM node.
  while (!domParentFiber.dom) {
    domParentFiber = domParentFiber.parent
  }
  const domParent = domParentFiber.dom

  if (fiber.effectTag === "PLACEMENT" && fiber.dom != null) {
    domParent.appendChild(fiber.dom)
  } else if (fiber.effectTag === "UPDATE" && fiber.dom != null) {
    updateDom(fiber.dom, fiber.alternate.props, fiber.props)
  } else if (fiber.effectTag === "DELETION") {
    commitDeletion(fiber, domParent)
  }
  // recorrido recursivo por el fiber tree
  commitWork(fiber.child)
  commitWork(fiber.sibling)
}

Una vez localizado domParent, se decide qué hacer según el effect tag:

  • PLACEMENT con dom no nulo → domParent.appendChild(fiber.dom): se inserta el nodo nuevo.
  • UPDATE con dom no nulo → updateDom(fiber.dom, fiber.alternate.props, fiber.props): se sincronizan las props cambiadas sobre el nodo existente (detalle completo en 15 · updateDom).
  • DELETION → commitDeletion(fiber, domParent): se retira el nodo.

Al final, las dos llamadas commitWork(fiber.child) y commitWork(fiber.sibling) continúan el recorrido en profundidad por todo el árbol, exactamente igual que la estructura enlazada child/sibling que se detalló en el recorrido del árbol.

commitDeletion: bajar hasta encontrar DOM

Eliminar un fiber tiene su propia sutileza. Si el fiber eliminado es un componente de función, no tiene dom propio, así que hay que bajar por sus hijos hasta encontrar el primer nodo real que efectivamente quitar del padre. Además, antes de tocar el DOM, hay que ejecutar los cleanups de sus hooks: un componente desmontado debe limpiar sus efectos.

src/React.ts
function commitDeletion(fiber, domParent) {
  // Ejecutar cleanups de hooks antes de eliminar el nodo
  if (fiber.hooks) {
    fiber.hooks.forEach(hook => {
      if (typeof hook.cleanup === 'function') {
        hook.cleanup();
      }
    });
  }

  if (fiber.dom) {
    domParent.removeChild(fiber.dom);
  } else if (fiber.child) {
    commitDeletion(fiber.child, domParent);
  }
}
Idea clave

Fíjate en la simetría con commitWork: para insertar y actualizar hay que subir hasta un padre con DOM; para eliminar hay que bajar hasta un hijo con DOM. En ambos casos la razón es la misma: los componentes de función son «transparentes» al DOM, no producen ningún nodo por sí mismos.

¿Por qué una fase separada e ininterrumpible?

La separación entre render y commit es una de las decisiones de arquitectura más importantes de React, y este clon la respeta. El comentario que precede a render en el fuente lo explica sin rodeos: el problema de mutar el DOM mientras se construye el árbol es que «el navegador podría interrumpir nuestro trabajo antes de terminar de renderizar el árbol completo. En ese caso, el usuario vería una interfaz incompleta».

Por eso el trabajo se divide en dos: una fase de render interrumpible, que solo calcula y etiqueta fibers en memoria y puede pausarse mil veces con requestIdleCallback sin que se note; y una fase de commit atómica, que aplica todos los cambios de golpe. Como commitRoot no consulta deadline.timeRemaining() ni cede el control en ningún punto, el usuario nunca ve un estado intermedio: o ve la interfaz anterior, o ve la nueva, nunca una a medio construir.

Nota

En React real la fase de commit también es síncrona e ininterrumpible por esta misma razón, y se subdivide internamente (before mutation, mutation, layout). Aquí todo eso se condensa en commitRoot más el barrido de wipEffects, que hace las veces de la fase de efectos pasivos.
